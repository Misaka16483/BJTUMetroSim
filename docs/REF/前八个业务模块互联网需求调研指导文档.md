# 前八个业务模块互联网需求调研指导文档

本文档用于指导小组通过互联网搜索、筛选、记录和归纳前八个业务仿真模块的需求。本文不依赖现有项目文档，所有需求均从公开资料、标准、开源工具文档、论文、行业资料和产品资料中提取，并最终转化为可实现、可测试、可追踪的软件需求。

前八个业务模块包括：

1. 线路与轨道仿真模块
2. 车辆动力学仿真模块
3. 牵引供电仿真模块
4. 信号与列控仿真模块
5. 司机台/驾驶行为仿真模块
6. 车站与站台仿真模块
7. 调度与运行图仿真模块
8. 环境与扰动仿真模块

## 1. 调研目标

本次调研不是简单摘抄资料，而是要回答四类问题：

1. 该模块在真实轨道交通系统中负责什么业务？
2. 该模块在仿真系统中至少需要模拟哪些对象、状态、事件和规则？
3. 该模块与其他模块交换哪些数据？
4. 如何判断该模块“仿真得合理”或“功能实现成功”？

最终输出不是资料堆积，而是需求表、数据字典、场景清单、接口候选表和评估指标。

## 2. 调研输出物

每个模块至少交付 5 类材料。

### 2.1 资料来源登记表

用于记录所有搜索到并采用的资料。

| 字段 | 说明 |
|---|---|
| Source ID | 资料编号，例如 `SRC-INF-001` |
| 模块 | 对应模块，例如线路与轨道 |
| 标题 | 网页、论文、标准或文档标题 |
| URL | 原始链接 |
| 来源类型 | 标准/官方文档/开源工具文档/论文/厂商资料/新闻案例 |
| 发布机构 | 例如 ERA、UIC、railML、DLR、FRA |
| 发布时间 | 若能查到则填写 |
| 可信度评分 | 1-5 分 |
| 可用内容摘要 | 该来源能支持哪些需求 |
| 使用限制 | 是否为摘要、是否需要二次验证、是否有版权限制 |

### 2.2 需求候选表

| 字段 | 说明 |
|---|---|
| Req ID | 需求编号，例如 `INF-FR-001` |
| 模块 | 所属模块 |
| 类型 | 功能/数据/接口/算法/场景/性能/约束/评估 |
| 需求描述 | 用“系统应...”描述 |
| 来源依据 | Source ID + 简短说明 |
| 优先级 | Must/Should/Could/Won't |
| 验收方式 | 演示、单元测试、仿真测试、日志检查、人工检查 |
| 风险 | 无设备、资料不足、模型过复杂、接口未知等 |

### 2.3 数据对象清单

每个模块要提取核心对象、字段和单位。例如线路模块应包含区间、站点、坡度、曲线、限速、道岔；车辆模块应包含速度、位置、加速度、牵引力、制动力、质量等。

### 2.4 场景清单

每个模块至少整理 3 个可演示场景。例如：

- 正常进站停车
- 临时限速运行
- 信号禁止通过
- 供电电压下降
- 车门故障导致延误

### 2.5 评估指标清单

每个模块至少提出 2 个可测指标。例如：

- 停车误差
- 速度曲线是否满足限速
- 是否违反信号授权
- 延误传播时间
- 故障注入后系统响应时间

## 3. 资料优先级

互联网资料应按以下优先级采用。

| 优先级 | 来源类型 | 使用建议 |

| B | 开源仿真工具官方文档 | 作为功能拆解和接口设计参考 |
| C | 学术论文、大学资料、公开数据集 | 作为模型、算法和评估指标参考 |
| D | 厂商产品介绍、白皮书、案例 | 作为功能边界和展示方式参考 |

## 4. 搜索执行流程

### 步骤 1：确定模块边界

每个模块先写 5 句话：

1. 该模块模拟什么现实对象？
2. 该模块输入什么？
3. 该模块输出什么？
4. 该模块影响哪些其他模块？
5. 本项目最小实现范围是什么？

示例：

```text
模块：车辆动力学仿真模块
现实对象：列车车辆及其运动过程
输入：牵引/制动指令、线路坡度、限速、供电状态
输出：位置、速度、加速度、能耗、停车状态
影响模块：信号、供电、车站、调度、评估
最小范围：单列车一维运动，支持牵引、制动、阻力、停车误差计算
```

### 步骤 2：建立中英文关键词表

每个模块必须同时使用中文和英文关键词。英文资料通常更容易找到标准、论文和开源工具文档；中文资料更适合理解国内术语。

关键词组合规则：

```text
领域词 + 对象词 + 行为词 + 文档类型
```

例如：

```text
railway infrastructure gradient speed restriction simulation documentation
train dynamics traction braking curve simulation model
轨道交通 信号系统 列控 仿真 需求
railway timetable dispatching simulation requirements
```

### 步骤 3：使用高级搜索语法

推荐搜索语法：

```text
site:railml.org rolling stock timetable infrastructure
site:sumo.dlr.de/docs railway simulation train
site:era.europa.eu ERTMS ATO operational principles pdf
filetype:pdf railway traction power supply simulation requirements
filetype:pdf train dispatching simulation disruption
"railway dwell time" "platform" "door"
"train performance simulation" "tractive effort" "braking"
```

每个模块至少执行：

- 3 次官方/标准类搜索
- 3 次开源工具/产品类搜索
- 3 次论文/案例类搜索

### 步骤 4：筛选资料

每条资料按 5 项评分，满分 25 分。

| 维度 | 评分标准 |
|---|---|
| 权威性 | 官方标准 5，开源官方文档 4，论文 3，厂商 2，博客 1 |
| 相关性 | 直接覆盖模块 5，部分覆盖 3，背景资料 1 |
| 可提取性 | 有对象/字段/流程/接口 5，只有概念 2 |
| 时效性 | 5 年内 5，10 年内 3，更早资料需说明原因 |
| 可验证性 | 可交叉验证 5，孤立资料 2 |

采用原则：

- 20 分以上可作为主要依据。
- 15-19 分可作为辅助依据。
- 15 分以下只作为线索。

### 步骤 5：提取需求

阅读资料时按以下标签摘录，不要整段复制。

| 标签 | 提取内容 |
|---|---|
| OBJ | 业务对象，例如区间、信号机、列车、站台 |
| ATTR | 对象属性，例如长度、坡度、速度、状态 |
| ACT | 行为，例如发车、制动、授权、开门 |
| RULE | 规则，例如红灯禁止通过、到站后满足条件才能开门 |
| IF | 接口，例如车辆状态发送给信号模块 |
| METRIC | 指标，例如延误、停车误差、能耗 |
| SCENE | 场景，例如临时限速、供电故障 |

### 步骤 6：转化为规范需求

将摘录内容转化为统一句式：

```text
系统应 + 功能动作 + 对象 + 约束/条件 + 输出结果
```

示例：

```text
系统应根据列车当前位置、目标停车点和当前速度，实时计算距离目标点的剩余距离，并以不低于 10 Hz 的频率提供给自动停车算法。
```

不要写成：

```text
系统需要很智能地停车。
```

### 步骤 7：交叉验证

每条 Must 级需求至少要有 2 个来源支持，或者 1 个强权威来源支持。

交叉验证方式：

- railML 用于验证数据对象是否合理。
- SUMO/OpenTrack 用于验证仿真功能和输出指标是否合理。
- ERA/ERTMS 用于验证信号、列控、ATO 相关概念是否合理。
- 论文用于验证算法模型或评估指标是否合理。

## 5. 各模块调研指南

## 5.1 线路与轨道仿真模块

### 调研目标

明确线路网络如何表达，列车位置如何定位，轨道条件如何影响车辆运行。

### 推荐搜索关键词

```text
railway infrastructure simulation track topology gradient curve speed restriction
RailTopoModel railway topology track section switch signal
railML infrastructure schema track station switch gradient speed
轨道交通 线路 拓扑 坡度 曲线 限速 仿真
铁路线路 数据模型 轨道 区间 道岔 信号机
```

### 重点资料入口

- RailTopoModel: https://www.railtopomodel.org/
- railML infrastructure 相关资料入口: https://www.railml.org/en/subschemas
- SUMO railways: https://sumo.dlr.de/docs/Simulation/Railways.html

### 需要提取的对象

```text
线路 Line
区间 Section
轨道 Track
站点 Station
站台 Platform
停车点 StopPoint
道岔 Switch
信号机 Signal
坡度 Gradient
曲线 Curve
限速区 SpeedRestriction
里程标 Mileage
闭塞分区 Block
```

### 需求问题清单

1. 线路是单线、双线还是多线？
2. 位置采用坐标、里程还是拓扑节点表示？
3. 是否需要坡度、曲线和限速？
4. 道岔是否需要动态改变方向？
5. 信号机、站台、停车点如何挂接在线路上？
6. 是否需要支持多站、多区间？
7. 线路数据是否需要从文件导入？

### 可形成的需求示例

```text
INF-FR-001 系统应支持以“线路-区间-轨道-里程点”的结构描述单条仿真线路。
INF-FR-002 系统应支持配置站点、站台和目标停车点，并向车辆模块提供当前位置到目标停车点的距离。
INF-FR-003 系统应支持配置区间限速，并向信号与控制模块提供当前位置对应的最高允许速度。
INF-DR-001 线路数据应至少包含区间长度、站点位置、停车点位置、限速值和坡度。
```

### 评估指标

- 位置计算误差是否在允许范围内。
- 站点、停车点、限速区是否能被正确识别。
- 车辆跨区间运行时线路状态是否连续。

## 5.2 车辆动力学仿真模块

### 调研目标

明确列车运动模型、牵引制动响应、阻力模型和状态输出。

### 推荐搜索关键词

```text
train dynamics simulation traction braking curve resistance model
rolling stock simulation tractive effort braking effort speed distance
railML rolling stock train braking traction parameters
OpenTrack rolling stock train performance simulation
列车动力学 仿真 牵引 制动 阻力 模型
列车 速度 距离 制动曲线 停车误差
```

### 重点资料入口

- railML rolling stock 入口: https://www.railml.org/en/subschemas
- OpenTrack overview: https://www.opentrack.ch/mobile/opentrack_e/opentrack_e.html
- SUMO railway simulation: https://sumo.dlr.de/docs/Simulation/Railways.html

### 需要提取的对象

```text
列车 Train
车辆编组 Consist
质量 Mass
速度 Speed
加速度 Acceleration
位置 Position
牵引级位 TractionNotch
制动级位 BrakeNotch
牵引力 TractionForce
制动力 BrakingForce
运行阻力 Resistance
制动延迟 BrakeDelay
最大速度 MaxSpeed
停车状态 StopState
```

### 需求问题清单

1. 使用一维运动模型还是更复杂模型？
2. 牵引和制动是连续值还是档位值？
3. 是否考虑坡度阻力、基本阻力、曲线阻力？
4. 是否模拟制动响应延迟？
5. 车辆状态更新频率是多少？
6. 是否需要支持人工控制和自动控制共同输入？
7. 是否需要输出速度-时间、位置-时间曲线？

### 可形成的需求示例

```text
VEH-FR-001 系统应根据牵引/制动指令、列车质量、线路坡度和运行阻力更新列车速度、加速度和位置。
VEH-FR-002 系统应支持牵引级位和制动级位输入，并将其转换为牵引力或制动力。
VEH-FR-003 系统应在列车速度降至阈值以下且距离目标停车点满足条件时判定列车停车完成。
VEH-DR-001 车辆状态应包含时间戳、位置、速度、加速度、牵引级位、制动级位和运行阶段。
```

### 评估指标

- 速度和位置更新是否连续。
- 相同输入下运行结果是否可重复。
- 自动停车误差是否可计算。
- 是否能输出完整运行曲线。

### 调研目标

明确供电系统如何影响牵引能力、能耗和故障场景。

### 重点资料入口

- OpenTrack/SUMO 类工具可用于理解供电对运行输出的影响。
- 交通运输机构、铁路运营商、厂商白皮书可作为系统结构参考。

### 需要提取的对象

```text
供电区段 PowerSection

```

### 需求问题清单

1. 项目是否需要真实电网计算，还是简化为供电约束？
2. 供电状态如何影响车辆牵引力？
3. 是否统计能耗？
4. 是否模拟再生制动？
5. 是否需要供电故障场景？
6. 供电数据如何可视化？

### 可形成的需求示例

```text
PWR-FR-001 系统应维护每个供电区段的可用状态、电压和功率限制。
PWR-FR-002 当供电区段处于欠压或故障状态时，系统应限制车辆模块可用牵引力。
PWR-FR-003 系统应根据车辆牵引功率和制动状态估算单次运行能耗。
PWR-SC-001 系统应支持注入供电故障场景，并记录列车控制策略的响应。
```

### 评估指标

- 能耗统计是否随牵引/制动变化。
- 供电故障是否能触发牵引限制。
- 供电状态是否能在可视化界面实时显示。

```text
信号状态 SignalAspect
```

### 需求问题清单

1. 项目采用信号机显示、闭塞分区，还是移动授权抽象？
2. 信号状态有哪些？红、黄、绿是否足够？

6. 信号状态如何与线路、道岔和列车位置关联？

### 可形成的需求示例

```text
SIG-FR-001 系统应根据列车位置、闭塞分区占用和进路状态生成当前信号状态。
SIG-FR-002 系统应向车辆控制模块提供当前位置的最高允许速度。
SIG-FR-003 当列车超过允许速度阈值或越过禁止通过信号时，系统应触发紧急制动事件。
SIG-FR-004 ATO 自动驾驶指令应受到信号与 ATP 约束，不得主动生成违反授权的牵引指令。
```

### 评估指标

- 是否出现闯红灯、越权运行。
- 超速检测是否及时。
- 自动控制是否始终满足信号约束。

### 需求问题清单

1. 司机台输入来自真实硬件、键盘，还是界面按钮？
2. 控制输入是离散档位还是连续值？
3. 系统支持哪些模式：手动、自动、手自一体？
4. 自动模式下司机是否可以接管？
5. 哪些操作需要记录？
6. 操作异常如何提示？

### 可形成的需求示例

```text
DRV-FR-001 系统应支持手动驾驶模式，允许用户输入牵引、制动、方向和紧急制动指令。
DRV-FR-002 系统应记录所有司机操作事件，包括时间戳、操作类型、操作值和列车状态。
DRV-FR-003 系统应支持从自动模式切换到手动模式，并在切换时清除自动控制输出。
DRV-IF-001 司机台模块应向控制模块输出标准化 DriverInput 数据对象。
```

### 评估指标

- 操作到车辆响应的延迟。
- 操作记录完整性。
- 模式切换是否安全、可追踪。

## 5.6 车站与站台仿真模块

### 调研目标

明确到站、停车窗、开关门、停站时间和发车条件。

### 推荐搜索关键词

```text
railway station platform dwell time simulation door operation
train dwell time model boarding alighting platform train interface
platform screen door train stopping accuracy simulation
地铁 车站 站台 停站时间 开关门 屏蔽门 仿真
列车 停车精度 站台 发车条件 乘降
```

### 重点资料入口

- 站台停站时间论文可用于理解 dwell time。
- 运营商或厂商资料可用于理解屏蔽门、开关门条件。
- railML/SUMO/OpenTrack 可用于理解站点和停站在仿真中的表达。

### 需要提取的对象

```text
车站 Station
站台 Platform
停车窗 StopWindow
目标停车点 StopTarget
车门 Door
屏蔽门 PlatformScreenDoor
停站时间 DwellTime
乘降 PassengerExchange
发车条件 DepartureCondition
越站 SkipStop
```

### 需求问题清单

1. 停车合格范围是多少？
2. 停车后是否需要开门？
3. 开门条件是否包括速度为 0、停车误差合格、站台侧正确？
4. 停站时间是固定值还是与乘客流相关？
5. 发车条件是什么？
6. 是否模拟屏蔽门联动？

### 可形成的需求示例

```text
STN-FR-001 系统应支持配置每个车站的目标停车点和允许停车误差范围。
STN-FR-002 当列车速度为 0 且停车误差在允许范围内时，系统应允许执行开门操作。
STN-FR-003 系统应记录到站时间、开门时间、关门时间和发车时间。
STN-SC-001 系统应支持站台门故障或停站延长场景。
```

### 评估指标

- 停车误差。
- 停站时间偏差。
- 开关门条件是否被正确约束。

## 9. 质量检查清单

提交前逐项检查：

- 每个模块至少有 5 条功能需求。
- 每个模块至少有 3 个数据对象。
- 每个模块至少有 2 个场景。
- 每个模块至少有 2 个评估指标。
- 每条 Must 需求有来源依据。
- 所有需求都能被测试或演示。
- 没有“智能化”“高效”“友好”等无法验证的空泛表述。
- 需求之间没有明显冲突。
- 数据对象命名统一。
- 接口输入输出能连接后四个支撑模块。
