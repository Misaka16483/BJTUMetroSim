# 轨道交通综合仿真系统需求说明书

版本：v0.1 审核稿

日期：2026-07-07

状态：待用户审核

## 1. 文档目的

本文档用于整合现有需求调研、老师提供的线路数据表、补充互联网资料和前期 AI 优化问题讨论，形成后续软件设计文档的需求基线。

本文档重点覆盖：

1. 基本列控系统与计算机联锁系统。
2. 基于客流与列车负载量的车辆运行调度。
3. 司机台智能控制与 ATO/ATP 监督。
4. 车辆动力学与牵引供电模块的全部备选优化问题。
5. 线路、车站、扰动、通信、数据记录、可视化等必要支撑需求。
6. 多阶段逐步实现与深化路线。

## 2. 项目定位

本项目拟开发一个基于真实线路基础数据的轨道交通综合仿真系统。系统应能够加载老师提供的线路数据，构建电子地图与信号基础设施模型，模拟列车在线路上的运行、列控/联锁约束、司机台操作、自动驾驶控制、乘客流与车辆负载、运行图调度、供电约束和扰动事件。

系统不是单一“停车算法”程序，而是一个可逐步深化的综合仿真平台：

```
真实线路数据
  ↓
电子地图与信号基础设施
  ↓
列车运行与司机台控制
  ↓
列控/联锁安全约束
  ↓
客流与车辆负载驱动调度
  ↓
供电、能耗与扰动仿真
  ↓
AI 优化与评估回放
```

## 6. 系统范围

## 6.1 范围内

系统应覆盖：

1. 线路基础数据解析与电子地图构建。
2. 基本列控系统：固定闭塞/逻辑区段占用、限速、移动授权、ATP 监督。
3. 基本计算机联锁系统：进路办理、道岔锁闭、区段占用检查、信号开放。
4. 车辆动力学：位置、速度、加速度、牵引、制动、停车判定。
5. 牵引供电：供电状态、功率限制、能耗、再生制动估算、欠压/断电。
6. 司机台智能控制：手动、ATO、ATP 监督、警惕、紧急制动、司机辅助建议。
7. 客流与列车负载：进站客流、候车人数、上下车、满载率、滞留乘客。
8. 运行调度：发车间隔、停站时间、跳停、延误传播、扰动重排。
9. 环境扰动：大客流、车门/屏蔽门故障、临时限速、信号故障、供电故障。
10. 数据记录、指标评估、历史回放和可视化。
11. AI 优化实验：车辆/供电控制参数优化、客流调度优化、扰动重排等。

## 6.2 第一阶段暂不强制

以下内容可作为后续深化，不作为第一阶段验收硬要求：

1. 完整 CBTC 移动闭塞工程级实现。
2. 完整列控安全认证级逻辑。
3. 完整牵引供电潮流计算。
4. 真实乘客个体级仿真。
5. 多线路换乘网络级调度。
6. 深度强化学习稳定训练。
7. 真实司机台硬件全部按钮和显示协议还原。

## 7. 总体功能架构

建议系统划分为 12 个模块：

| 编号 | 模块 | 核心职责 |
| --- | --- | --- |
| M1 | 线路与轨道仿真 | 解析线路数据、构建拓扑、查询限速/坡度/站台/设备 |
| M2 | 车辆动力学仿真 | 根据牵引/制动/坡度/阻力更新列车状态 |
| M3 | 牵引供电仿真 | 模拟供电状态、功率限制、能耗与再生制动 |
| M4 | 信号与列控仿真 | 计算允许速度、MA、ATP 监督、信号状态 |
| M5 | 司机台/驾驶行为仿真 | 手动驾驶、ATO、ATP 覆盖、司机辅助 |
| M6 | 车站与站台仿真 | 站台、停车、屏蔽门、上下客、停站时间 |
| M7 | 调度与运行图仿真 | 运行图、发车间隔、客流负载驱动调度 |
| M8 | 环境与扰动仿真 | 注入故障、客流、临时限速等扰动 |
| M9 | 通信接口与数据交换 | 状态总线、命令总线、设备/仿真接口 |
| M10 | 仿真时钟与场景管理 | 场景加载、统一仿真时间、生命周期 |
| M11 | 数据记录、评估与回放 | 记录状态/事件/指标，支持回放与对比 |
| M12 | 可视化与人机交互 | 线路图、司机台、调度面板、曲线和报告 |

## 8. 核心业务流程

## 8.1 单列车自动运行流程

```
加载线路数据与场景
  ↓
生成列车任务：起点、目标站、计划到达时间
  ↓
信号/联锁检查进路与区段状态
  ↓
司机台 ATO 生成牵引/制动指令
  ↓
车辆模块更新位置、速度和加速度
  ↓
供电模块计算功率、能耗和牵引限制
  ↓
列控模块更新允许速度、MA、ATP 监督
  ↓
到站停车、开门、上下客、关门、发车
  ↓
记录指标并生成报告
```

## 8.2 客流驱动运行调度流程

```
生成或导入时变客流
  ↓
车站模块计算站台候车人数和上下车需求
  ↓
车辆模块更新列车负载率
  ↓
调度模块判断拥挤、满载、滞留与延误
  ↓
调整发车间隔、停站时间、跳停或加开策略
  ↓
信号模块校验最小追踪间隔和进路安全
  ↓
记录乘客等待、列车负载和运行图偏差
```

## 8.3 扰动响应流程

```
扰动模块按时间/位置/人工操作注入事件
  ↓
影响车辆、信号、供电、车站或调度模块
  ↓
调度模块生成恢复策略
  ↓
司机台/ATO 根据限制调整控制
  ↓
系统记录扰动影响、恢复时间和指标变化
```

## 9.1 线路与轨道基础数据需求

| ID | 优先级 | 需求 |
| --- | --- | --- |
| REQ-LINE-001 | Must | 系统应能解析 `线路数据(1).xls` 中第 1-4 行元信息，并按声明行数/列数读取真实数据。 |
| REQ-LINE-002 | Must | 系统应将 `65535` 识别为空引用，并统一将 cm、cm/s 转换为 m、m/s 或 km/h。 |
| REQ-LINE-003 | Must | 系统应根据 `点表`、`Seg表` 和 `道岔表` 构建线路拓扑图。 |
| REQ-LINE-004 | Must | 系统应支持按列车当前位置查询所在 Seg、Seg 偏移量、相邻 Seg、道岔关系。 |
| REQ-LINE-005 | Must | 系统应支持按位置查询静态限速、坡度、站台、信号机和应答器。 |
| REQ-LINE-006 | Must | 系统应在加载线路数据时执行引用一致性检查，并输出缺失引用、越界偏移、孤立对象报告。 |
| REQ-LINE-007 | Should | 系统应可视化展示线路拓扑、站台、信号机、道岔和列车位置。 |
| REQ-LINE-008 | Should | 系统应支持线路数据导出为 JSON 或 SQLite 以供其他模块快速读取。 |

## 9.2 基本列控系统需求

列控系统负责监督列车是否在授权范围内、限速范围内、安全距离内运行。第一阶段建议采用固定闭塞/逻辑区段 + 简化 MA；第二阶段支持准移动闭塞或虚拟块。

| ID | 优先级 | 需求 |
| --- | --- | --- |
| REQ-TC-001 | Must | 系统应维护闭塞区段、逻辑区段、计轴区段的占用状态。 |
| REQ-TC-002 | Must | 系统应根据列车位置更新当前区段占用。 |
| REQ-TC-003 | Must | 系统应根据静态限速、临时限速和信号状态输出 `permittedSpeedMps`。 |
| REQ-TC-004 | Must | 系统应生成 `SignalState`，包含当前信号、允许速度、MA 终点、ATP 紧急制动请求。 |
| REQ-TC-005 | Must | 当列车超过允许速度、越过禁止信号或越过 MA 终点时，系统应触发 ATP 紧急制动请求。 |
| REQ-TC-006 | Must | 系统应支持红/黄/绿或允许/警告/禁止的简化信号状态。 |
| REQ-TC-007 | Should | 系统应支持点式模式和 CBTC 模式下接近区段、触发区段的差异化处理。 |
| REQ-TC-008 | Should | 系统应支持准移动闭塞或虚拟块，MA 终点可随前车尾部动态更新。 |
| REQ-TC-009 | Should | 系统应输出闯红灯次数、超速干预次数、临时限速触发次数、模式切换次数。 |

## 9.3 计算机联锁系统需求

计算机联锁系统负责进路办理、道岔锁闭、信号开放和冲突防护。老师线路表中的 `进路表`、`道岔表`、`计轴区段表`、`保护区段表` 是本模块核心数据。

| ID | 优先级 | 需求 |
| --- | --- | --- |
| REQ-CBI-001 | Must | 系统应从 `进路表` 读取进路名称、始端信号机、终端信号机、包含计轴区段、保护区段、接近区段和触发区段。 |
| REQ-CBI-002 | Must | 系统应支持进路办理请求，输入始端信号机、终端信号机或进路编号。 |
| REQ-CBI-003 | Must | 办理进路前，系统应检查进路包含计轴区段是否空闲。 |
| REQ-CBI-004 | Must | 办理进路前，系统应检查保护区段是否满足安全条件。 |
| REQ-CBI-005 | Must | 办理进路时，系统应根据进路要求锁闭相关道岔和区段。 |
| REQ-CBI-006 | Must | 进路锁闭成功后，系统才允许开放始端信号机。 |
| REQ-CBI-007 | Must | 存在冲突进路、区段占用或道岔不可用时，系统应拒绝开放信号并给出原因。 |
| REQ-CBI-008 | Must | 列车通过进路后，系统应按区段释放规则释放已通过区段。 |
| REQ-CBI-009 | Should | 系统应支持引导/降级进路模式，限制速度并记录事件。 |
| REQ-CBI-010 | Should | 系统应支持防淹门、车库门、SPKS 开关等设备状态对进路办理的影响。 |
| REQ-CBI-011 | Could | 系统应支持自动生成或校验联锁表，检测冲突进路和遗漏防护。 |

## 9.6 司机台智能控制需求

司机台模块既要支持手动驾驶，又要支持 ATO 自动驾驶和 ATP 监督覆盖。智能控制的第一阶段应采用可解释规则控制 + 参数优化，而不是完全黑箱控制。

| ID | 优先级 | 需求 |
| --- | --- | --- |
| REQ-CAB-001 | Must | 系统应维护司机台手柄和按钮状态，包括牵引、制动、方向、警惕、紧急制动。 |
| REQ-CAB-002 | Must | 系统应支持 MANUAL、ATO、ATP_SUPERVISED、SH/调车等模式。 |
