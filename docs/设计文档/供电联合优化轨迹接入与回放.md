# 供电联合优化轨迹接入与回放

## 1. 目的与边界

本设计将“多车牵引/制动时序与超级电容控制联合优化”的列车输入从实验内部解析代理中解耦，使同一套直流牵引供电求解器和储能控制评估器可以接收：

- V2 解析代理轨迹，用于快速、确定性的算法回归；
- 内存轨迹，用于单元测试和外部仿真器接入；
- `POWER-TRAJECTORY-V1` JSONL 记录，用于可复现实验；
- 主仿真引擎公开 `TickSnapshot`，用于联锁和运行图稳定后的正式联调。

本次实现不修改联锁、运行图、调度运行时或主引擎 tick 逻辑。所谓“动力学不闭合”是原联合优化实验的解析代理/输入证据问题，不等同于主仿真系统动力学不闭合。新接口的作用是用统一契约分别验证两者，避免再混为一谈。

## 2. 统一轨迹契约

每个 `TrajectoryFrame` 表示一个仿真时刻，包含零到多列车的 `TrainTrajectorySample`。正式供电评估所需的核心字段如下：

| 类别 | 字段 |
| --- | --- |
| 标识与时空 | `sim_time_ms`、`train_id`、`direction`、`mileage_m` |
| 车辆动力学 | `speed_mps`、`acceleration_mps2`、`mass_kg`、牵引力、电制动力、空气制动力、坡度、可选运行阻力 |
| 供电负荷 | 辅助功率、可选牵引功率请求、可选再生功率；功率缺失时由力和速度换算 |
| 运行约束 | 允许速度、阶段、当前/下一站、发车授权、联锁保持原因、活动进路、折返计数 |

校验器检查时间单调性、帧内列车唯一性、方向与数值合法性、牵引制动重叠、限速、相邻速度/加速度一致性、里程连续性和可计算时的动力学力平衡。主引擎快照中的加速度由相邻快照差分得到；第一帧没有历史数据时记为零，因此正式采集应从停车或仿真初始化阶段开始。

## 3. 提供器与候选变量语义

| 提供器 | 时序变量可变 | 储能变量可变 | 用途 |
| --- | --- | --- | --- |
| `ProxyTrajectoryProvider` | 是 | 是 | 保持 V2 快速优化和回归 |
| `InMemoryTrajectoryProvider` | 否 | 是 | 测试、一次仿真多组储能参数复用 |
| `JsonlTrajectoryProvider` | 否 | 是 | 固定轨迹回放、复现实验 |
| `EngineRunTrajectoryProvider` | 由 runner 声明 | 是 | 每组时序参数驱动主引擎重跑 |

固定回放不能声称验证了新的发车展宽、牵引切换或制动切换参数。评估器会显式拒绝这种候选，而不是静默地用旧轨迹计算。仅修改储能限值或触发阈值时，主引擎 runner 会复用相同轨迹，不重复运行车辆仿真。

主引擎使用绝对仿真时刻，评估配置通过 `start_time_ms` 指定轨迹窗口起点；解析代理仍默认从 0 开始。

## 4. 记录和回放

采集端只依赖公开快照：

```python
adapter = EngineSnapshotTrajectoryAdapter()
with JsonlTrajectoryRecorder("run.jsonl", metadata={"scenarioId": "line9"}) as recorder:
    for snapshot in snapshots:
        recorder.write(adapter.frame_from_snapshot(snapshot))
    recorder.write_metadata({"trackingMetrics": final_tracking_metrics})
```

运行图 KPI 通常只能在仿真结束后计算，因此记录器允许在文件尾追加元数据。回放器会合并全部元数据记录。

轨迹可用以下命令独立校验：

```powershell
python tools/validate_power_trajectory.py outputs/run.jsonl --fixed-roster
```

回放参与供电评估时，应以轨迹首帧时刻配置 `start_time_ms`，以回放时长配置 `horizon_sec`，再把 `JsonlTrajectoryProvider` 注入 `JointPowerEvaluator`。

## 5. 正式实验门禁

一条轨迹通过结构校验并不代表联合优化结果有效。正式结果还必须同时满足：

1. JSONL 元数据包含速度跟踪 RMSE、平均/最大发车偏差、运行时间偏差和停车误差；缺失时 `operationalMetrics` 约束明确失败。
2. 运行图和联锁已稳定，轨迹没有非预期长时间保持、发车饥饿或进路死锁。
3. 动力学残差阈值与采样周期、快照舍入精度一致；解析代理仍应达到数值精确闭合，主引擎回放默认允许 `5 kN` 离散残差。
4. 同一时序轨迹分别执行无储能基线和多组储能控制，避免把列车运行变化误计为储能收益。
5. 最优候选必须由可改变时序变量的主引擎 runner 重新运行验证，不能只依赖固定回放。

## 6. 与队友工作的衔接

当前即可继续进行解析代理回归、储能控制搜索、轨迹契约测试和固定回放分析，不必等待联锁修复。以下工作需等待联锁故障和运行图接入完成：

- 采集无异常保持的全程多车主引擎轨迹；
- 计算真实发车、运行时间和停车位置 KPI；
- 启用 `ENGINE_TRAJECTORY_JSONL` 条件集成测试；
- 用 `EngineRunTrajectoryProvider` 对时序变量做正式联合优化和独立复跑。

因此后续安排应是“并行推进基础设施与储能侧优化，正式时序结论等待主仿真运行链稳定”，而不是整体停工，也不应在等待期间修改联锁或运行图核心来绕过故障。
