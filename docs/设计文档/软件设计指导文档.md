# 轨道交通仿真系统软件设计指导文档

## 1. 文档目的

本文档用于指导小组把需求转化为可实现的软件设计。设计目标是：在有限实训时间内，完成一个结构清晰、可联调、可演示、可解释的软件系统，而不是堆叠临时代码。

推荐系统名称：

> 轨道交通仿真司机台控制与智能停站系统

推荐核心闭环：

```text
读取仿真状态 -> 解析列车状态 -> 判断控制阶段 -> 生成控制指令 -> 下发到仿真系统 -> 记录和评估结果
```

## 2. 设计原则

### 2.1 先做最小闭环

优先让系统跑通：

```text
连接 -> 读速度/位置 -> 发牵引/制动 -> 自动停车 -> 记录误差
```

供电、完整信号、调度、多车等内容可以设计接口，但不要一开始全部实现。

### 2.2 仿真设备和算法解耦

自动停车算法不要直接操作 TCP/UDP/串口。应通过统一接口读取状态、发送指令，这样可以同时支持真实仿真台和离线模拟器。

```text
控制算法 -> ControlPort 接口 -> 真实设备适配器/离线模拟器
```

### 2.3 数据先规范，界面后美化

实时状态字段、控制指令字段和日志字段必须统一，否则后期调参和答辩展示会很痛苦。

### 2.4 每个模块都能单独测试

通信模块可用假数据测试，算法模块可用离线车辆模型测试，界面模块可用模拟状态流测试。

## 3. 总体架构

建议采用分层架构：

```text
┌──────────────────────────────────────────────┐
│ 可视化与人机交互层                            │
│ 状态监控、手动控制、模式切换、曲线、日志        │
├──────────────────────────────────────────────┤
│ 应用服务层                                    │
│ 场景管理、自动停站服务、数据记录、结果评估      │
├──────────────────────────────────────────────┤
│ 领域模型与控制算法层                          │
│ TrainState、ControlCommand、线路/站点、控制器  │
├──────────────────────────────────────────────┤
│ 接口适配层                                    │
│ 真实仿真台适配器、司机台适配器、离线模拟器      │
├──────────────────────────────────────────────┤
│ 基础设施层                                    │
│ TCP/UDP/串口、配置、日志、CSV/SQLite、时钟       │
└──────────────────────────────────────────────┘
```

## 4. 推荐模块拆解

### 4.1 通信接口模块

职责：

- 建立和关闭连接。
- 按接口协议收发数据。
- 完成字节流/报文到结构化对象的转换。
- 处理心跳、超时、重连、异常。

输入：

- 仿真系统报文。
- 司机台输入报文。

输出：

- `TrainState`
- `DeviceState`
- 通信事件。
- 控制指令发送结果。

设计建议：

```text
ITransport：底层传输接口，屏蔽 TCP/UDP/串口差异。
ProtocolCodec：协议编解码，负责报文和对象转换。
SimulationClient：仿真系统客户端，负责连接和状态收发。
DriverDeskClient：司机台客户端，负责实体司机台输入。
```

### 4.2 状态模型模块

职责：

- 定义系统内部统一数据模型。
- 屏蔽外部协议字段命名差异。


### 4.3 手动控制模块

职责：

- 接收实体司机台或界面按钮输入。
- 转换为标准控制指令。
- 记录人工操作过程。

关键设计点：

- 手动模式下，算法不接管牵引/制动。
- 手动输入也要经过安全检查。
- 手动和自动模式切换要有明确状态机。

### 4.4 自动停站控制模块

职责：

- 根据当前位置、速度、目标停车点、限速和运行阶段生成牵引/制动指令。
- 支持参数配置和调试。

建议先实现分段规则控制，再升级为 PID 或目标速度曲线控制。

#### 4.4.1 分段规则控制

控制阶段：

| 阶段 | 条件 | 行为 |
|---|---|---|
| 启动 | 速度低且距离远 | 小/中级牵引 |
| 加速 | 距离较远且未达限速 | 继续牵引 |
| 巡航 | 接近目标速度 | 惰行或小牵引 |
| 制动 | 进入制动距离 | 按距离和速度制动 |
| 精调 | 距离很近且速度低 | 小制动/释放制动 |
| 停车 | 速度接近 0 且误差可接受 | 保持制动，结束 |

伪代码：

```text
distance = target_position - current_position

if communication_lost:
    return safe_command()

if distance <= 0:
    return brake_or_emergency()

target_speed = speed_profile(distance)

if current_speed > target_speed + tolerance:
    return brake_command()
elif current_speed < target_speed - tolerance:
    return traction_command()
else:
    return coast_command()
```

#### 4.4.2 目标速度曲线

自动停车可以基于制动距离估算：

```text
v_target = sqrt(2 * a_brake * max(distance - margin, 0))
```

其中：

- `v_target`：当前距离下允许的目标速度。
- `a_brake`：预估舒适制动减速度。
- `distance`：距离目标停车点的剩余距离。
- `margin`：安全余量。

如果当前速度高于目标速度，则制动；低于目标速度则牵引或惰行。

### 4.5 安全约束模块

职责：

- 对控制指令做最后检查。
- 防止异常指令直接下发设备。

规则示例：

| 规则 | 处理 |
|---|---|
| 通信中断 | 停止自动控制，提示人工接管 |
| 速度超过限速 | 增大制动或触发保护 |
| 已越过停车点 | 制动并记录过停 |
| 指令超出协议范围 | 限幅或拒绝发送 |
| 模式未知 | 不执行自动控制 |

### 4.6 场景管理模块

职责：

- 管理一次实验的生命周期。
- 组织开始、暂停、结束、重置和回放。

状态机建议：

```text
Idle -> Connected -> Ready -> Running -> Stopping -> Completed
                               ↓
                             Fault
```

状态说明：

| 状态 | 含义 |
|---|---|
| Idle | 未连接 |
| Connected | 已连接设备 |
| Ready | 已选择线路、目标站和模式 |
| Running | 列车运行中 |
| Stopping | 进入停车控制阶段 |
| Completed | 实验完成并生成结果 |
| Fault | 通信或控制异常 |

### 4.7 数据记录与评估模块

职责：

- 记录运行全过程。
- 计算评价指标。
- 支持答辩展示和问题复盘。

建议记录 CSV 或 SQLite。

运行日志字段：

```text
timestamp
position_m
speed_mps
distance_to_target_m
traction_level
brake_level
driving_mode
control_phase
signal_state
event
```

评价指标：

| 指标 | 含义 |
|---|---|
| 停车误差 | 实际停车位置与目标停车点距离 |
| 运行时间 | 从开始到停车完成的时间 |
| 最大速度 | 全程最高速度 |
| 平均速度 | 总距离/运行时间 |
| 制动次数 | 制动指令切换次数 |
| 越站标记 | 是否超过目标停车点 |
| 成功率 | 多次运行中成功停车的比例 |
| 通信异常次数 | 超时、断连、解析失败次数 |

### 4.8 可视化与交互模块

职责：

- 给用户展示系统当前状态。
- 支持模式切换、参数配置和实验启动。

建议主界面区域：

| 区域 | 内容 |
|---|---|
| 连接区 | 连接状态、端口、重连按钮 |
| 运行状态区 | 速度、位置、目标距离、模式 |
| 控制区 | 手动/自动模式、牵引、制动、急停 |
| 算法区 | 控制阶段、目标速度、控制参数 |
| 曲线区 | 速度-时间、位置-时间、距离-时间 |
| 日志区 | 通信事件、控制事件、告警 |
| 结果区 | 停车误差、运行时间、成功/失败 |

## 5. 供电、车辆、轨道、信号等子系统设计边界

实训项目不建议完整实现所有专业仿真模型，但要在设计上给出边界。

| 子系统 | 本项目建议实现 | 本项目暂不实现 |
|---|---|---|
| 车辆仿真 | 读取速度/位置；离线模式下实现简化动力学 | 完整多质点车辆动力学 |
| 轨道仿真 | 线路、站点、目标停车点、限速 | 完整轨道几何和道岔联锁 |
| 信号系统 | 信号状态、限速、运行许可的简化约束 | 完整 CBTC/联锁/ATP 安全认证 |
| 供电系统 | 供电可用/不可用、可选能耗统计 | 牵引供电潮流和再生制动精确计算 |
| 调度系统 | 单车任务、目标站、计划时间 | 多车运行图和冲突调整 |
| 司机台 | 输入读取、控制映射、状态显示 | 司机培训评分体系 |

## 6. 接口设计

### 6.1 内部接口

#### 状态读取接口

```text
get_current_state() -> TrainState
```

#### 指令发送接口

```text
send_command(command: ControlCommand) -> SendResult
```

#### 控制算法接口

```text
compute_command(state: TrainState, context: ScenarioContext) -> ControlCommand
```

#### 数据记录接口

```text
record_sample(state: TrainState, command: ControlCommand, event: Event) -> None
```

### 6.2 外部接口

外部接口要根据老师提供的协议补全。接口表模板：

| 接口编号 | 方向 | 报文/字段 | 类型 | 单位 | 刷新周期 | 说明 |
|---|---|---|---|---|---|---|
| IF-SIM-001 | 输入 | position | float | m | 待确认 | 列车当前位置 |
| IF-SIM-002 | 输入 | speed | float | m/s | 待确认 | 当前速度 |
| IF-SIM-003 | 输入 | signal_state | enum | - | 待确认 | 信号状态 |
| IF-SIM-004 | 输出 | traction_level | int | 级位 | 待确认 | 牵引指令 |
| IF-SIM-005 | 输出 | brake_level | int | 级位 | 待确认 | 制动指令 |
| IF-SIM-006 | 输出 | emergency_brake | bool | - | 待确认 | 紧急制动 |

### 6.3 通信异常处理

| 异常 | 检测方式 | 处理 |
|---|---|---|
| 连接失败 | 建链失败或握手失败 | 提示用户检查端口和设备 |
| 接收超时 | 超过 N 个周期未收到状态 | 停止自动控制，记录告警 |
| 报文解析失败 | 校验和/长度/字段错误 | 丢弃报文，记录错误 |
| 发送失败 | socket/串口发送异常 | 重试或进入 Fault 状态 |
| 数据跳变 | 速度/位置突变超过阈值 | 标记异常，不直接用于控制 |

## 7. 数据库/文件设计

短周期建议优先使用 CSV；如果界面需要查询历史记录，可以用 SQLite。

### 7.1 配置文件

建议使用 JSON/YAML：

```text
connection:
  type: tcp
  host: 127.0.0.1
  port: 9000

control:
  cycle_ms: 100
  max_traction_level: 5
  max_brake_level: 7
  stop_tolerance_m: 1.0
  comfortable_deceleration: 0.8

  target_station: StationA
  target_stop_position_m: 1200.0
```

### 7.2 实验结果文件

```text
runs/
```


## 8. 技术选型建议

### 8.1 快速实现路线

推荐：

```text
Python + PyQt/PySide + CSV/SQLite + matplotlib/pyqtgraph
```

优点：

- 开发快。
- 适合通信、算法、图表和数据记录。
- 离线模拟器容易实现。

适用情况：

- 接口协议不要求特定语言。
- 小组成员 Python 基础较好。

### 8.2 工程化路线

可选：

```text
C# + WPF/WinForms + SQLite
Java + JavaFX + SQLite
C++ + Qt
```

适用情况：

- 需要更强桌面端体验。
- 已有设备 SDK 是 Windows/C++/C#。

### 8.3 技术选型决策表

| 维度 | Python | C# | C++ |
|---|---|---|---|
| 开发速度 | 高 | 中 | 低 |
| 界面开发 | 中 | 高 | 中 |
| 通信开发 | 高 | 高 | 高 |
| 算法调试 | 高 | 中 | 中 |
| 性能 | 中 | 高 | 高 |
| 学习成本 | 低 | 中 | 高 |

## 9. 测试设计

### 9.1 测试层次

| 层次 | 测试内容 |
|---|---|
| 单元测试 | 协议解析、速度曲线计算、停车误差计算 |
| 集成测试 | 通信模块 + 状态模型 + 控制算法 |
| 离线仿真测试 | 不连接设备，验证算法能停住 |
| 现场联调测试 | 连接 704 仿真台，验证真实数据闭环 |
| 演示测试 | 按答辩脚本完整运行一遍 |

### 9.2 核心测试用例

| 编号 | 名称 | 输入 | 预期结果 |
|---|---|---|---|
| TC-CON-001 | 连接仿真系统 | 正确 IP/端口 | 连接成功，开始接收状态 |
| TC-CON-002 | 通信中断处理 | 断开连接 | 系统提示异常，停止自动控制 |
| TC-STA-001 | 状态解析 | 一帧合法状态报文 | 正确解析速度、位置、模式 |
| TC-CTL-001 | 指令限幅 | 超范围制动级位 | 系统限幅或拒绝发送 |
| TC-ATO-001 | 离线自动停车 | 初始位置、速度、目标点 | 最终停车误差小于阈值 |
| TC-ATO-002 | 过站保护 | 距离为负 | 系统制动并记录过停 |
| TC-LOG-001 | 数据记录 | 一次完整运行 | 生成 CSV 和 summary |

## 12. 风险设计

| 风险 | 影响 | 设计应对 |
|---|---|---|
| 仿真台排队使用 | 联调时间不足 | 先实现离线模拟器 |
| 接口协议不清楚 | 通信模块延期 | 设计协议适配层，先用假数据 |
| 自动停车调参困难 | 停车误差大 | 先规则控制，再参数优化 |
| 现场演示失败 | 答辩风险 | 准备录屏、离线 demo 和历史数据 |
| 小组分工不清 | 进度失控 | 每个模块绑定负责人和验收标准 |

## 13. 答辩展示设计

最终演示建议按以下脚本：

1. 打开系统，连接仿真台或离线模拟器。
2. 展示实时速度、位置、目标距离。
3. 进入手动模式，展示牵引/制动指令变化。
4. 切换自动停站模式，点击开始。
5. 系统自动控制列车运行并停车。
6. 展示停车误差、运行时间、速度曲线和日志。
7. 展示需求追踪矩阵和测试结果。

答辩故事线：

```text
我们从轨道交通仿真业务出发，提炼出单列车停站控制场景；
根据接口协议完成仿真设备通信；
设计了状态模型、控制算法、安全约束和数据评估模块；
最终实现手动/自动控制，并用停车误差、运行时间和稳定性验证效果。
```
