# 轨道交通仿真系统数据库设计文档

版本：v0.1  
日期：2026-07-08  
适用范围：Phase 0、Phase 1、Phase 2 详细设计；Phase 3 实验数据库概要设计  
依据文档：`轨道交通仿真系统软件设计文档.md`、`前端接入与双层界面设计.md`、`五人分工与执行规范.md`

## 1. 设计目标

数据库设计服务四类核心需求：

1. 线路数据快速加载：Phase 0 把老师提供的 Excel 线路数据转成稳定、可校验的 `data/cache/line_map.json`；Phase 1/2 再迁移为可查询的 SQLite 静态库。
2. 仿真过程记录：记录列车状态、控制命令、事件、指标、联锁、客流、调度、供电、扰动和平台接口健康。
3. 回放与前端展示：支持按运行批次、仿真时间、列车、车站、区段、供电分区查询历史状态。
4. 优化实验复现：支持 Phase 3 多场景、多参数、多随机种子的批量实验与指标对比。

数据库不直接承担高频实时控制职责。实时闭环优先由内存状态和消息总线完成，数据库负责采样记录、事件留痕、回放和分析。

## 2. 数据库文件划分

建议拆成一类 JSON 缓存和三类 SQLite 文件，避免静态数据、单次运行记录和批量实验结果互相污染。

| 数据库 | 建议路径 | 作用 | 生命周期 |
|---|---|---|---|
| Phase 0 线路 JSON 缓存 | `data/cache/line_map.json` | 当前权威静态数据源，供 `TrackMapService` 和 Phase 0 API 直接读取 | Excel 线路数据更新时重建 |
| 线路静态库 | `data/cache/line9_static.sqlite` | Phase 1/2 迁移目标，保存线路、站台、Seg、信号机、道岔、限速、坡度、进路、区段、供电分区配置 | 线路数据更新时重建 |
| 单次运行库 | `outputs/runs/<run_id>/run.sqlite` | 某一次仿真的状态、命令、事件、指标、接口健康 | 每次运行生成一个 |
| 实验汇总库 | `outputs/experiments/experiments.sqlite` | 多次运行的参数、目标函数、Pareto 结果、对比结论 | Phase 3 启用 |

Phase 0 静态线路数据以 `line_map.json` 为唯一权威源，SQLite 静态库暂不作为运行依赖。Phase 0/1 可以先只实现单次运行库中的 `runs`、`events`、`metrics` 三张基础表；Phase 1 再补齐 JSON 到 SQLite 的导入器和静态查询切换能力；Phase 2 逐步补充 typed records。Phase 3 再启用实验汇总库。

## 2.2 核心关系速查

| 关系 | 关联方式 | 说明 |
|---|---|---|
| 线路到车站 | `stations.line_id -> lines.line_id` | 一条线路包含多个车站 |
| 车站到站台 | `platforms.station_id -> stations.station_id` | 一个车站可有多个方向/站台 |
| 站台到轨道 | `platforms.segment_id -> segments.segment_id` | 停站和前端定位依赖该关系 |
| Seg 到相邻 Seg | `segment_links.from_segment_id/to_segment_id -> segments.segment_id` | 车辆沿拓扑推进 |
| 进路到信号 | `routes.start_signal_id/end_signal_id -> signals.signal_id` | 进路由始端/终端信号定义 |
| 进路到区段 | `route_sections.route_id -> routes.route_id` | 联锁检查占用和保护区段 |
| 进路到道岔 | `route_switches.route_id -> routes.route_id` | 办理进路时锁闭道岔 |
| 供电分区到 Seg | `power_section_segments.power_section_id -> power_sections.power_section_id` | 计算同一供电分区内功率需求 |
| 运行记录到批次 | 所有运行表 `run_id -> runs.id` | 每条运行记录都必须能追溯到运行批次 |
| 实验 trial 到运行库 | `experiment_trials.run_db_path -> outputs/runs/.../run.sqlite` | 实验汇总库不复制全部运行明细 |

## 3. 通用规范

## 3.1 命名规范

| 类型 | 规范 | 示例 |
|---|---|---|
| 表名 | 小写蛇形命名，复数或记录类后缀保持一致 | `train_telemetry`, `power_records` |
| 主键 | 单表自增主键统一使用 `id` | `id INTEGER PRIMARY KEY AUTOINCREMENT` |
| 原始导入 ID | 使用 `<entity>_id`，SQLite 类型为 `INTEGER` | `segment_id`, `platform_id`, `signal_id`, `station_id` |
| 业务 ID | 字段名必须带业务语义，SQLite 类型为 `TEXT` | `station_business_id`, `route_id`, `train_id`, `power_section_id` |
| 外键 | 使用被引用实体字段名 | `run_id`, `segment_id`, `station_id` |
| 时间 | 仿真时间用 `sim_time_ms`，墙钟时间用 ISO 字符串 | `started_at`, `created_at` |
| 单位 | 数值字段名必须带单位 | `speed_mps`, `power_kw`, `energy_kwh` |
| JSON | JSON 字段后缀 `_json` | `metadata_json`, `payload_json` |
| 状态 | 状态字段用大写枚举字符串 | `ONLINE`, `LOCKED`, `UNDERVOLTAGE` |

ID 语义约束：

1. `segment_id`、`platform_id`、`signal_id`、`station_id` 保存 `line_map.json` 或老师表格中的原始整数 ID，不得写入 `S-GGZ` 这类业务字符串。
2. 客流、调度、停站等业务记录使用 `station_business_id` 表示稳定车站业务 ID，例如 `S-GGZ`；需要展示时可冗余写入 `station_code`。
3. API 中的 `stationId` 仅兼容 Phase 0 静态站点映射；Phase 1+ 业务接口应使用 `stationBusinessId`。
4. `route_id`、`train_id`、`power_section_id` 不是老师表格原始整数时，按业务 ID 处理，使用 `TEXT`。

## 3.2 数据类型

| 逻辑类型 | SQLite 类型 | 说明 |
|---|---|---|
| ID | `TEXT` 或 `INTEGER` | 业务 ID 用 `TEXT`，内部主键用 `INTEGER` |
| 时间戳 | `INTEGER` / `TEXT` | 仿真时间 `INTEGER ms`，墙钟时间 ISO `TEXT` |
| 布尔 | `INTEGER` | `0=false`, `1=true` |
| 枚举 | `TEXT` | 写入大写枚举值 |
| 小数 | `REAL` | 速度、功率、能耗、比例 |
| JSON | `TEXT` | 使用 UTF-8 JSON 字符串 |

## 3.4 写入策略

| 数据类型 | 写入频率 | 策略 |
|---|---:|---|
| 高频列车状态 | 100 ms 或 1 s | 默认 1 s 落库；安全事件即时落库；调试模式可 100 ms |
| 控制命令 | 命令变化时 | 状态变化或来源变化时落库 |
| 安全/扰动事件 | 即时 | 必须完整记录 |
| 客流/调度/供电 | 1 s | 与调度周期保持一致 |
| 平台接口原始帧 | 按接口周期 | 只保存摘要、hash、长度和解析结果；必要时另存二进制文件 |
| 指标 | 运行结束或阶段性 | 可重复写入，但同名指标需带 `scope` |

## 3.5 索引原则

所有运行记录表至少建立：

```sql
CREATE INDEX idx_<table>_run_time ON <table>(run_id, sim_time_ms);
```

按实体查询频繁的表增加：

```sql
CREATE INDEX idx_train_telemetry_train_time ON train_telemetry(run_id, train_id, sim_time_ms);
CREATE INDEX idx_section_occupation_section_time ON section_occupation_records(run_id, section_id, sim_time_ms);
CREATE INDEX idx_station_passenger_station_time ON station_passenger_records(run_id, station_business_id, direction, sim_time_ms);
```

## 4. 线路静态库设计

线路静态库是 Phase 1/2 迁移目标。Phase 0 的 `TrackMapService`、前端轨道级视图和静态 API 仍以 `data/cache/line_map.json` 为权威数据源；只有当导入器验收通过后，才允许通过配置切换到 SQLite 静态库。

SQLite 连接必须显式开启外键：

```sql
PRAGMA foreign_keys = ON;
```

JSON 到 SQLite 的迁移验收项：

1. 导入器读取 `line_map.json`，生成 `line9_static.sqlite` 和一份校验报告。
2. `segments`、`signals`、`platforms`、`speed_restrictions`、`gradients` 等核心对象行数必须与 JSON 缓存一致。
3. `stations.station_business_id` 必须由 `station_code` 稳定派生，且 `UNIQUE(line_id, station_code)`、`UNIQUE(station_business_id)` 校验通过。
4. 所有 `platforms.segment_id`、`signals.segment_id`、`speed_restrictions.segment_id`、`gradients.segment_id` 外键必须能解析到 `segments.segment_id`。
5. API 可通过配置 `LINE_STATIC_SOURCE=json|sqlite` 切换数据源；切换前后 `/api/lines/9/track-map` 的关键计数和字段映射保持一致。

## 4.2 基础线路表

```sql
CREATE TABLE lines (
    line_id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    direction_a_name TEXT,
    direction_b_name TEXT,
    metadata_json TEXT NOT NULL DEFAULT '{}'
);

CREATE TABLE stations (
    station_id INTEGER PRIMARY KEY,
    line_id TEXT NOT NULL,
    station_code TEXT NOT NULL,
    station_business_id TEXT NOT NULL,
    station_name TEXT NOT NULL,
    sequence_no INTEGER,
    mileage_m REAL,
    lat REAL,
    lng REAL,
    metadata_json TEXT NOT NULL DEFAULT '{}',
    UNIQUE(line_id, station_code),
    UNIQUE(station_business_id),
    FOREIGN KEY(line_id) REFERENCES lines(line_id)
);

CREATE TABLE platforms (
    platform_id INTEGER PRIMARY KEY,
```

```sql
SELECT sim_time_ms, direction, arrivals, boarding, alighting, waiting, platform_density_pax_per_m2
FROM station_passenger_records
WHERE run_id = ? AND station_business_id = ?
ORDER BY sim_time_ms;
```

## 8.4 查询供电限牵时段

```sql
SELECT sim_time_ms, power_section_id, requested_power_kw, available_power_kw,
       traction_limit_ratio, voltage_level
FROM power_records
WHERE run_id = ? AND traction_limit_ratio < 1.0
ORDER BY sim_time_ms;
```

## 8.5 查询接口异常

```sql
SELECT sim_time_ms, adapter_id, status, drop_count, crc_error_count,
       parse_error_count, missed_frame_count, last_error
FROM adapter_health_records
WHERE run_id = ? AND status IN ('DEGRADED', 'OFFLINE')
ORDER BY sim_time_ms;
```
