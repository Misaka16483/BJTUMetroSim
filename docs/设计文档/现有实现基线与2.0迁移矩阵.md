> 文档状态：概要设计配套附件

## 1. 文档目的

本文档回答三个问题：

1. 当前仓库中已经有什么代码，实际接到了系统哪一层。
2. 当前实现距离需求说明书2.0和概要设计2.0还有什么差距。
3. 后续开发应以什么顺序复用、改造或替换现有代码。

本文档是现有实现的审计基线，不替代需求说明书和概要设计。原“七子系统”划分保留为历史代码来源说明；2.0开发统一按“四个业务模块 + 公共仿真内核 + 表现与适配层”组织。

## 2. 基线使用规则

### 2.1 基线边界

当前工作区包含未提交修改，因此引用本文件判断完成度时必须同时记录：

```text
branch + commit SHA + workingTreeClean + auditDate
```

提交、合并或回退后，应重新执行本文件第11章的验证命令。不能把某个成员本地尚未提交的能力直接计入远端`main`完成度。

### 2.2 事实修正

| 原总结中的表述 | 当前代码事实 | 修正结论 |
|---|---|---|
| 9号线采用DC 1500V | `line9_power_topology.json`配置额定750V、牵引所空载825V | 统一写为DC 750V第三轨教学模型 |
| 供电前端为静态/mock | 引擎生成`powerNetwork`快照，前端读取并支持故障、复位和联络开关操作 | 已形成L4可视闭环雏形，但参数尚未校核 |
| 潮流未接入`engine._tick()` | `_tick()`调用`_update_power()`并记录列车电压、牵引所功率和再生能量 | 已接入主tick，但对动力学的限牵反馈滞后一tick |
| 客流指标为占位值 | `StationService`已处理到达、上下客、滞留、停站时间和载客量 | 已形成L3运行闭环，尚未影响列车质量 |
| DCDP代码存在但关闭 | 引擎配置`use_dynamic_programming_profile=True` | 当前已启用并按列车缓存 |
| WebSocket前后端已接通 | 后端具备广播器，React没有建立WebSocket连接，仍以200 ms REST轮询 | 后端L2，前后端闭环未完成 |
| 当前已经支持多车调度 | 状态与服务多处使用列车列表，但默认场景只有`T0901`，前端仍有`trains[0]`和`T0901`硬编码 | 属于多车结构预留，未通过多车场景验证 |
| 核心闭环已完整打通 | 联锁未进入主tick，客流质量未进入动力学，实际供电限制存在一tick延迟 | 只能称为单车运营闭环雏形 |

## 3. 2.0模块口径

### 3.1 目标划分

| 2.0区域 | 包含职责 | 旧七系统代码去向 |
|---|---|---|
| 公共仿真内核 | 会话、时钟、命令队列、tick、快照、事件、记录、回放 | 原基础设施层和各系统编排代码 |
| 模块一：供电仿真 | DC网络、潮流、故障、再生、限牵、保护 | 原牵引供电系统 |
| 模块二：驾驶与一维动力学 | 司机级位、ATO/ATP仲裁、牵引传动、制动混合、纵向动力学 | 原牵引系统、制动系统、动力学系统和司机台输入 |
| 模块三：联锁与调度 | 区段占用、进路、道岔、信号、MA、ATS、车库生命周期 | 原信号系统中的CI、列控和调度部分 |
| 模块四：客流与热力图 | 分时需求、站台队列、上下客、停站、载荷、热力图 | 原运营指标中的客流相关部分 |
| 表现与适配层 | React工作台、实物司机台、HMI、实验室平台接口 | 原三维视景/司机台和网络适配代码 |

### 3.2 暂缓范围

以下内容不是2.0首轮完整闭环的前置条件：

1. Unity/Unreal三维视景。
2. 工程级MVB/CAN设备网络和完整CCU/TCU/BCU节点仿真。
3. 工程级CBTC无线协议栈。
4. 云数据库和多租户部署。

通信时延、丢包和设备断连仍需实现，但作为公共适配与控制场景能力，而不是先建设完整列车通信网络。

## 4. 成熟度定义

不再使用“功能条目等权计数”计算百分比。每项能力使用以下成熟度：

| 等级 | 名称 | 判定条件 |
|---|---|---|
| L0 | 未实现 | 无可调用代码或只有文档描述 |
| L1 | 骨架 | 有类型、接口、静态页面或占位数据，不能独立证明算法 |
| L2 | 独立可运行 | 算法或适配器可独立运行并有测试，但未进入主tick |
| L3 | 主循环集成 | 已进入统一仿真时钟并能改变权威运行状态 |
| L4 | 端到端可观察 | 后端、API和前端/记录器闭环，操作与结果可观察 |
| L5 | 已验证 | 使用受控数据和场景完成定量验收、误差分析及回归固化 |

成熟度描述的是证据链，不表示算法复杂度。L4页面也可能仍使用工程估算参数，只有完成校核后才能达到L5。

### 6.1 当前tick的关键时序问题

当前顺序是：

```text
推进列车 -> 累积客流 -> 计算供电 -> 调度 -> 记录 -> 快照
```

列车推进读取的是上一个tick的`tractionLimitRatio`，然后才根据本tick已经执行过的牵引/制动百分比计算供电请求。这种一拍延迟可以作为当前基线，但2.0应改为：

```text
生成全部列车需求 -> 供电求解 -> 返回能力限制 -> 计算实际力 -> 动力学积分
```

## 14. 2.0完成定义

某项功能只有同时满足以下条件，才能在进度报告中标为“完成”：

1. 有明确输入、输出、单位、数据来源和异常语义。
2. 已进入统一仿真tick或被明确标注为离线工具。
3. 不通过前端mock伪造后端领域结果。
4. 有单元或领域测试。
5. 有至少一个端到端场景证明它影响了预期状态。
6. 关键指标被记录，可回放或导出复核。
7. 与需求编号和设计组件建立追溯关系。

因此，后续中期/最终报告应同时给出“当前成熟度、证据和剩余限制”，不再只给单一百分比。
