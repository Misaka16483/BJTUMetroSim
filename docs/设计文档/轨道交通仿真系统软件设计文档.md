# 轨道交通综合仿真系统软件设计文档

版本：v0.2  
日期：2026-07-08  
依据文档：`docs/需求与调研文档/轨道交通仿真系统需求说明书.md`、`docs/轨交多系统平台接口协议汇总20260630.docx`、`docs/设计文档/数据库设计文档.md`、`docs/设计文档/API设计文档.md`

## 1. 文档目的

本文档用于将已审核的需求说明书转化为可实施的软件设计方案。当前版本重点解决：

1. 如何解析老师提供的真实线路数据。
2. 如何建立统一数据模型、仿真时钟、消息总线和记录体系。
3. 如何完成 Phase 0 的数据基线与工程框架。
4. 如何完成 Phase 1 的单车运行、ATO 控制、基本列控和 ATP 防护。
5. 如何完成 Phase 2 的联锁、车站客流、供电约束、扰动和调度闭环设计。
6. 如何为 Phase 3 的 AI 优化与调度深化预留实验框架。
7. 如何利用学校实验室多系统平台接口，在自研模块能力不足或成本过高时接入信号、司机台、视景、车辆/RT-LAB 等外部系统。
8. 如何把“自研仿真 + 平台接口调用”组织成可验证、可对比、可做算法优化研究的闭环平台。

本文档不是最终编码规范，但应作为后续详细开发、接口定义、任务拆分和测试验收的共同基线。

## 2. 设计原则

1. 真实数据优先：第一版应从 `线路数据(1).xls` 构建电子地图，避免手写玩具线路。
2. 分阶段可运行：每个阶段都应形成可演示闭环，而不是等全部模块完成才运行。
3. 安全规则兜底：AI 或 ATO 输出不得绕过 ATP、限速、MA、紧急制动等安全约束。
4. 模块低耦合：业务模块通过统一消息和状态快照交互，减少直接互相调用。
5. 可回放可复现：所有仿真输入、状态、指令、事件和指标应可记录和复盘。
6. 可替换实现：UI、算法、通信协议可替换，核心数据模型和接口保持稳定。
7. 自研优先、接口兜底：线路、调度、ATO、供电、评价和优化算法优先自研；信号、车辆、司机台、视景等已有协议的复杂系统允许通过平台接口接入。
8. 平台不是黑箱终点：调用学校平台接口不是降低工作量，而是把已有实验室系统作为高保真对照、约束源或执行器；我们的含金量体现在统一建模、接口适配、闭环实验设计、算法优化和结果评估。
9. 同一模块多保真：关键模块应同时支持 Mock/简化模型、自研模型、平台接口模型三种实现，便于逐步开发、对照验证和优化实验。

## 2.1 平台接口利用原则

老师提供的 `轨交多系统平台接口协议汇总20260630.docx` 说明学校实验室已有多系统联调平台，覆盖信号、ATP-DMI、司机台、网络显示屏/信号屏、视景、车辆/RT-LAB 等接口。协议中未给出独立的牵引供电仿真接口，也未给出牵引变电所、接触网/第三轨电压、电流、潮流、再生能量回馈等供电网络字段。因此本文档将学校平台定位为“已有外部子系统集合”，而不是假设所有复杂模块都已有可调用接口。

设计时采用以下策略：

| 模块 | 第一选择 | 平台接口定位 | 我们的核心工作 |
|---|---|---|---|
| 线路/轨道数据 | 自研 | 无需调用平台 | 从 `线路数据(1).xls` 建立电子地图、站台/Seg/信号机/限速/坡度/进路数据基线 |
| 调度/运行图 | 自研 | 可向外部系统发送列车增删改与运行状态 | 构建客流负载驱动调度、运行图重排、指标评价 |
| ATO/司机辅助 | 自研 | 可接司机台 PLC/DMI 接口做硬件或半实物交互 | 控制算法、舒适度/能耗/准点多目标优化 |
| 基本列控/ATP | Phase 1 自研简化版 | 高保真信号/ATP 接口可作为约束源与对照源 | 先实现规则可解释模型，再与平台信号输出对比校验 |
| 联锁/信号 | 简化自研 + 平台可选 | 信号系统 UDP 接口可提供道岔、信号机、列车方向等状态 | 做进路办理逻辑、状态映射、故障/延迟鲁棒性实验 |
| 车辆动力学 | 自研一维模型 + 平台可选 | 车辆模型 UDP 20ms、RT-LAB API 500ms，可提供速度/加速度/累计里程 | 用自研模型快速迭代，用 RT-LAB/车辆接口做高保真验证 |
| 供电仿真 | 自研 | 当前协议无独立供电仿真接口；RT-LAB 路径名含 `PowerSystemAndTrainsV1`，但已列变量主要是列车编号、操作指令、Seg、偏移、方向、激活端等车辆控制变量 | 自研供电分区、牵引功率、欠压约束、再生能量估算；用车辆响应间接标定，不把 RT-LAB 误写成供电接口 |
| 视景系统 | 平台优先 | UDP 100ms 同步列车位置、信号机、道岔、速度等 | 我们做仿真状态源和宏微观可视化，视景系统用于沉浸展示与演示 |

因此，本系统不是“从零造所有子系统”，而是“用统一数据模型和优化实验框架组织自研模块与实验室平台模块”，形成一个可做算法研究的综合仿真工作台。

## 2.2 协议能力边界核查

为避免把字段名或显示状态误判为完整仿真接口，所有外部系统接入前必须做“协议能力边界核查”。本次根据老师协议重新核对后的结论如下：

| 模块 | 协议中是否有明确接口 | 可直接使用的数据 | 不能直接假设的能力 | 设计处理 |
|---|---|---|---|---|
| 信号/列控 | 有 | 信号机、道岔、ATP/ATO 安全/非安全输入输出、列车运行方向、加车删车等 | 不等同于完整自研联锁规则库 | 自研简化联锁，平台信号用于对照或在环 |
| ATP-DMI | 有 | 速度、允许速度、目标速度、目标距离、模式、DMI 回复、驾驶台输入状态 | 不等同于完整列控核心源码 | 用于 DMI 显示和安全约束验证 |
| 司机台 PLC | 有 | 司控器、按钮、门控、指示灯、牵引/制动相关硬线状态 | 不提供车辆动力学或供电网络计算 | 作为人机交互和硬件在环输入 |
| 网络显示屏/信号屏 | 有 | 速度、时间、站点、信号、牵引切换等显示/交互字段 | 不提供信号系统内部逻辑 | 作为显示同步接口 |
| 视景系统 | 有 | 列车速度、EdgeID、区段距离、方向、他车、信号机、道岔、发车倒计时 | 不提供运行决策、联锁计算或供电计算 | 作为三维展示输出，不作为安全闭环唯一依据 |
| 车辆/RT-LAB | 有 | 车辆加速度、速度、累计里程；列车控制命令；RT-LAB 项目加载、执行、变量读写 | 不能仅凭 `PowerSystemAndTrainsV1` 路径名推断存在供电潮流/电压/能耗接口 | 作为车辆动力学高保真对照；供电只可用车辆响应间接标定 |
| 供电仿真 | 无独立接口 | 司机台/车辆协议中存在少量“电源供电状态、牵引力、牵引命令”等车辆侧字段 | 牵引变电所、供电分区电压电流、潮流、再生能量、欠压传播 | Phase 2 自研 `PowerService`，后续若老师补充接口再新增 `PowerAdapter` |

后续新增任何平台接口，都必须在本表中登记“能读什么、能写什么、不能证明什么、进入系统后映射到哪个内部模型”。

M13 是 v0.2 新增的关键层。它负责把学校平台接口中的二进制报文、端口、周期、大小端、CRC、字段单位转换成系统内部的 `TrainState`、`SignalState`、`ControlCommand`、`CabDisplayState`、`PlatformEvent` 等标准消息。`PowerState` 当前由自研 `PowerService` 产生；只有在后续获得明确供电协议后，才新增供电平台适配器。业务层不直接依赖 UDP/TCP/RT-LAB API，从而保证同一算法既能在自研模型上跑，也能在外部平台对照数据上复核。

## 4.2 运行时数据流

Phase 1 单车运行时，每个 tick 的推荐数据流如下：

```text
SimulationClock 产生 tick
  ↓
M8 EnvironmentService 注入到期扰动
  ↓
M1 TrackMapService 查询列车当前位置线路条件
  ↓
M4 TrainControlService 计算允许速度、MA、ATP 请求
  ↓
M5 CabControlService 生成 ControlCommand
  ↓
M13 PlatformAdapter 判断当前模块采用自研实现还是平台实现
  ├─ 自研车辆/供电：M3 PowerService 生成 PowerState 或默认牵引限制
  ├─ 平台车辆：VehicleRtLabAdapter/VehicleUdpAdapter 写入控制命令并读取速度/加速度/里程
  ├─ 平台信号：SignalUdpAdapter 读取道岔/信号机/ATP 输出
  └─ 平台司机台/视景：CabTcpAdapter/ViewUdpAdapter 同步显示和硬件输入
  ↓
M2 VehicleDynamicsService 更新 TrainState
  ↓
M6 StationService 判断停车/到站事件
  ↓
M11 Recorder 记录状态、指令、事件、指标
  ↓
M12 UI 刷新显示
```

注意：Phase 1 的供电模块可先使用默认 `tractionLimitRatio=1.0`，但接口必须保留。

当外部平台接入时，系统的 tick 不应强行要求所有外部接口同频。建议用“仿真主时钟 + 接口采样缓存”方式处理：

| 外部接口 | 协议周期 | 系统处理 |
|---|---:|---|
| 信号系统 UDP | 信号侧 250ms；总控转发 100ms | 以 100ms 为输入采样，缺包时保留上一有效状态并记录通信质量 |
| ATP-DMI | 160ms | 按序列号、CRC48、1.6s 时效、连续 10 周期断链规则进行有效性判断 |
| 司机台 PLC | PLC 上行 100ms；上位机下行单次触发 | 输入事件化，输出命令需幂等，避免重复触发 |
| 视景系统 UDP | 100ms | 作为显示同步接口，不作为安全闭环唯一依据 |
| 车辆模型 UDP | 20ms | 可高频采样，聚合到仿真主 tick；用于动力学高保真验证 |
| 车辆/RT-LAB API | 500ms | 适合加载/启动/参数写入/低频监视，不承担高频控制闭环 |

## 4.4 学校平台接口适配设计

平台接口适配层的目标是将老师协议中的外部系统统一抽象为内部标准端口，避免业务算法直接处理二进制报文。

### 4.4.1 接口能力摘要

| 外部系统 | 协议/周期 | 可获得能力 | 可写入能力 | 对本项目价值 |
|---|---|---|---|---|
| 信号系统 | UDP/IP；信号侧约 250ms，总控侧约 100ms；小端报文 | 道岔状态、信号机状态、ATP/ATO/车辆相关输出、列车运行方向 | 列车速度、累计走行距离、方向、载重、紧急制动、司机台开关量、加车/删车/清空车 | 用作联锁/列控高保真状态源，校验自研信号模型 |
| ATP-DMI | 冗余以太网；160ms；大端；CRC48；1.6s 时效 | 当前速度、允许速度、目标速度、目标距离、驾驶模式、文本提示等 | DMI 回复、驾驶台输入状态 | 用作 ATP/DMI 显示与安全约束验证 |
| 司机台 PLC | TCP；PLC 为 Server；192.168.100.123:8001/8002/8003；上行 100ms；WORD 小端 | 司控器、按钮、门控、指示灯等硬件状态 | 26 byte 控制报文，驱动司机台输出 | 用作真实司机台人机交互与硬件在环输入 |
| 网络显示屏/信号屏 | TCP；网络屏 192.168.100.122:8888，线上实包 570 byte；信号屏 192.168.100.121:9999，线上实包 68 byte（包头 `TotalLen=62`） | 屏端牵引切换请求等 | 上位机向屏幕发送速度、时间、站点、信号等显示内容 | 用于司机台显示联动与演示 |
| 视景系统 | UDP；ATS 与视景控制机 100ms；端口 8302/8303 | 作为显示端接收状态 | 列车速度、EdgeID、区段距离、方向、他车、信号机、道岔、发车倒计时 | 用于三维场景同步展示，提高演示效果 |
| 车辆/RT-LAB 模型 | UDP 20ms；API 500ms；模型侧 192.168.200.110:23001，平台侧 192.168.200.102:23002 | 1-20 车加速度、速度、累计里程；API 输出变量 | 加速/减速/惰行命令、加减速百分比；API 写入模型参数 | 用作车辆动力学高保真对照，校准自研车辆模型；不作为供电仿真接口 |
| 供电仿真 | 当前协议未提供 | 无牵引变电所、电压、电流、潮流、再生能量等供电网络字段 | 无 | Phase 2 自研供电约束模型；未来如补充协议再扩展 `PowerAdapter` |

### 4.4.2 内部标准端口

外部协议不进入业务模块，统一进入以下内部端口：

```python
class PlatformAdapter:
    def connect(self) -> None: ...
    def disconnect(self) -> None: ...
    def poll(self, now_ms: int) -> list[Envelope]: ...
    def publish_command(self, command: Envelope) -> None: ...
    def health(self) -> AdapterHealth: ...
```

每个适配器输出统一消息：

| 外部字段 | 内部模型 | 转换规则 |
|---|---|---|
| 速度 cm/s、mm/s | `TrainState.speed_mps` | `cm/s / 100` 或 `mm/s / 1000` |
| 累计走行距离 cm | `TrainState.distance_m` | `cm / 100` |
| 区段偏移 mm | `TrackPosition.offset_m` | `mm / 1000` |
| 方向 0x55/0xaa | `Direction.UP/DOWN` | 统一由线路方向字典映射 |
| 信号灯状态字节 | `SignalState.aspect` | 红/黄/绿/白/蓝/灭灯/断丝枚举 |
| 道岔 0x01/0x02/0x04 | `SwitchState.position` | 定位/反位/四开 |
| ATP/DMI 速度与距离 | `MovementAuthority`、`CabDisplayState` | 保留原始值并转换单位 |
| 加减速命令 0/1/2 + 百分比 | `ControlCommand` | 惰行/牵引/制动 + ratio |

### 4.4.3 通信健壮性要求

平台接口接入后必须记录通信质量，否则优化实验会把通信异常误当成算法效果：

1. 每个适配器记录 `last_rx_time`、`last_tx_time`、`sequence`、`drop_count`、`crc_error_count`、`stale_count`。
2. ATP-DMI 报文按协议执行序列号、CRC48、1.6s 时效、连续 10 周期断链规则。
3. UDP 接口采用“最后有效值 + 过期标记”策略；超过时效后状态进入 `STALE`，不得继续作为安全决策唯一依据。
4. 所有外部报文需保存原始 bytes 摘要和解析后字段，便于复盘。
5. 平台接口不可直接覆盖安全规则；外部输出进入系统后仍需经过统一安全约束检查。

### 4.4.4 三种运行模式

| 模式 | 外部平台依赖 | 用途 |
|---|---|---|
| Pure-Sim | 无 | 日常开发、CI 测试、算法快速迭代 |
| Hybrid-Sim | 部分接口接入，如车辆/司机台/视景 | 半实物演示、关键模块高保真验证 |
| Platform-in-the-loop | 多接口同时接入 | 接近实验室平台联调，验证调度/ATO/优化策略对真实接口的适配性 |

所有算法实验必须标注运行模式，报告中不得混淆简化模型结果与平台闭环结果。

## 8. Phase 1 详细设计

## 8.1 车辆动力学模块

### 8.1.1 职责

`VehicleDynamicsService` 根据控制命令、线路条件和供电限制更新列车状态。

v0.2 起，车辆动力学不再只有单一自研模型，而是支持三种实现：

| 实现 | 说明 | 适用场景 |
|---|---|---|
| `SimpleVehicleModel` | 本文档 8.1.4 的一维纵向模型 | 快速开发、CI、算法早期迭代 |
| `CalibratedVehicleModel` | 用平台/RT-LAB 输出标定牵引、制动、阻力参数 | 论文/报告中的可解释自研模型 |
| `PlatformVehicleModel` | 通过车辆 UDP 或 RT-LAB API 读写外部模型 | 车辆动力学高保真验证；供电模型仍由自研 `PowerService` 计算 |

三种实现必须输出同一 `TrainState`，算法层不得依赖某个具体车辆模型。

### 8.1.2 输入

```text
上一 tick TrainState
ControlCommand
SignalState
PowerState
TrackCondition
VehicleConfig
dt
```

### 8.1.3 输出

```text
TrainState
VehicleEvent
EnergyDemand
```

### 8.1.4 第一版动力学模型

先采用一维纵向运动模型：

```text
net_force = traction_force - brake_force - resistance_force - gradient_force
acceleration = net_force / mass
speed_next = max(0, speed + acceleration * dt)
distance = speed * dt + 0.5 * acceleration * dt^2
position_next = TrackNavigator.advance(distance)
```

第一版参数：

```json
{
  "trainId": "T001",
  "massKg": 180000,
  "maxSpeedMps": 22.22,
  "maxTractionLevel": 5,
  "maxBrakeLevel": 5,
  "tractionForcePerLevelN": 20000,
  "brakeForcePerLevelN": 25000,
  "basicResistanceN": 3000,
  "stopSpeedThresholdMps": 0.05
}
```

### 8.1.5 牵引/制动力计算

```text
traction_force = tractionLevel * tractionForcePerLevelN * tractionLimitRatio
brake_force = brakeLevel * brakeForcePerLevelN
if emergencyBrake:
    brake_force = emergencyBrakeForceN
```

约束：

```text
牵引和制动不应同时生效
若 emergencyBrake=true，则 tractionLevel 强制为 0
若 speed_next < 0，则 speed_next = 0
```

### 8.1.6 坡度处理

Phase 1 可提供两种模式：

```text
ignoreGradient=true：不计坡度，只记录坡度字段
ignoreGradient=false：使用已确认单位计算 gradient_force
```

在坡度单位未确认前，默认 `ignoreGradient=true`。

### 8.1.7 平台车辆/RT-LAB 接入

老师协议提供两类车辆接口：

| 接口 | 方向 | 周期 | 数据 |
|---|---|---:|---|
| UDP 模型→平台 | 车辆模型输出 | 20ms | 1-20 车加速度、速度、累计里程 |
| UDP 平台→模型 | 控制输入 | 20ms | 1-20 车加速/减速/惰行命令、加减速百分比 |
| RT-LAB API | 加载/启动/停止/读写变量 | 500ms | `OpalOpenProject`、`OpalLoad`、`OpalExecute`、`OpalGetAcqGroupSyncSignals`、`OpalSetParametersByName` |

接入后，`PlatformVehicleModel` 的职责是：

1. 将 `ControlCommand` 转换为外部接口的 `0=惰行、1=加速、2=减速` 和百分比。
2. 将外部输出的速度、加速度、累计里程转换为 `TrainState`。
3. 维护外部模型 20ms 高频数据与系统 100ms 主 tick 的聚合关系。
4. 保存外部模型输出，用于标定 `CalibratedVehicleModel`。
5. 对 RT-LAB API 路径如 `PowerSystemAndTrainsV1/.../Train_Control/...` 建立变量字典，不在业务代码中硬编码路径。注意：该路径名不能被解释为已有供电仿真接口，只有协议表中明确列出的车辆控制/定位变量才可接入。

## 8.2 司机台与 ATO 控制模块

### 8.2.1 职责

`CabControlService` 负责将人工手柄、ATO 算法和 ATP 覆盖合成为最终 `ControlCommand`。

当接入学校司机台时，`CabTcpAdapter` 负责连接 PLC、网络屏和信号屏：

| 子接口 | 通信 | 数据意义 |
|---|---|---|
| PLC | TCP，PLC 为 Server，端口 8001/8002/8003，上行 100ms | 司控器、按钮、门控、指示灯等硬件状态 |
| 网络显示屏 | TCP，192.168.100.122:8888，线上实包 570 byte | 上位机向屏幕发送时间、速度、站点等信息 |
| 信号显示屏 | TCP，192.168.100.121:9999，线上实包 68 byte（包头 `TotalLen=62`） | 上位机向信号屏发送信号/模式相关显示 |

司机台接入后，系统仍由 `CabControlService` 做最终命令合成，硬件输入只是一类 `DriverInput` 来源。

### 8.2.2 模式

| 模式 | 说明 |
|---|---|
| MANUAL | 直接采用司机输入 |
| ATO | 根据目标速度/目标距离自动生成指令 |
| ATP_SUPERVISED | 人工驾驶，但 ATP 可覆盖 |
| SH | 调车/降级模式，Phase 1 可只保留状态 |

### 8.2.3 ATO 第一版控制策略

第一版采用规则控制器：

```text
distance = distanceToTargetM
v = speedMps
brake_distance = v^2 / (2 * expectedDeceleration)

if emergencyBrakeRequired:
    emergencyBrake = true
elif distance <= stopTolerance and v <= stopSpeedThreshold:
    traction = 0, brake = holdBrakeLevel
elif distance <= brake_distance + brakeMargin:
    traction = 0, brake = computeBrakeLevel(v, distance)
elif v < targetCruiseSpeed and v < permittedSpeed:
    traction = computeTractionLevel(v, targetCruiseSpeed)
else:
    traction = 0, brake = 0
```

推荐参数：

```json
{
  "targetCruiseSpeedMps": 12.0,
  "expectedDecelerationMps2": 0.8,
  "brakeMarginM": 20.0,
  "stopToleranceM": 1.0,
  "holdBrakeLevel": 1,
  "maxTractionLevel": 4,
  "maxBrakeLevel": 4
}
```

### 8.2.4 ATP 覆盖规则

最终命令合成顺序：

```text
driver/ATO command
  ↓
apply power limit
  ↓
apply speed limit guard
  ↓
apply ATP emergency override
  ↓
publish final ControlCommand
```

若 `SignalState.emergencyBrakeRequired=true`：

```json
{
  "tractionLevel": 0,
  "brakeLevel": 0,
  "emergencyBrake": true,
  "source": "ATP_OVERRIDE"
}
```

平台司机台实验可形成一个有价值的对照：同一运行场景下比较“人工驾驶输入”“规则 ATO”“AI 调参 ATO”在停车误差、舒适度、能耗和准点性上的差异。

## 8.3 基本列控模块

### 8.3.1 职责

`TrainControlService` 在 Phase 1 提供简化 ATP/信号监督。

## 8.5 PowerService 第一版

Phase 1 的 `PowerService` 保持轻量：

```text
默认 available=true
tractionLimitRatio=1.0
根据 tractionForce * speed 估算 powerKw
累计 energyKwh
```

供电故障、欠压、再生制动放到 Phase 2/3。

设计策略：

| 层级 | 实现 | 目的 |
|---|---|---|
| P1 简化估算 | `tractionForce * speed` 估算能耗 | 为 ATO/调度优化提供早期目标函数 |
| P2 自研供电模型 | 供电分区、最大功率、启动峰值、欠压阈值、再生能量吸收上限 | 研究多车同时牵引、峰值功率约束和节能调度 |
| 车辆平台间接标定 | 使用车辆/RT-LAB 的速度、加速度、累计里程对自研车辆阻力/牵引参数做校准 | 提高功率估算输入的可信度，但不替代供电仿真 |

因此，供电模块不能再假设由学校平台直接提供。Phase 2 必须自研 `PowerService`，先实现工程上可解释的分区功率约束、能耗估算、再生制动吸收估算和欠压触发逻辑；后续若老师补充供电接口，再通过新增 `PowerAdapter` 做高保真替换或复核。

## 9. Phase 2 详细设计

Phase 2 的目标是把 Phase 1 的单车安全运行闭环扩展为多车、多站、联锁、客流、供电、扰动和调度联动闭环。该阶段不追求安全认证级 CBTC/CI 细节，而是实现可解释、可复现实验、可与学校平台接口对照验证的工程化仿真内核。

Phase 2 的关键增量：

| 增量方向 | 具体能力 |
|---|---|
| 计算机联锁 | 进路办理、冲突检查、道岔锁闭、保护区段、进路释放 |
| 区段占用 | 基于列车头尾位置推导逻辑区段/计轴区段占用 |
| 车站客流 | 进站客流、站台候车、上下车、列车负载、停站时间修正 |
| 运行调度 | 基于间隔、延误、客流、供电约束的发车/扣车/停站调整 |
| 供电约束 | 自研供电分区功率估算、牵引限额、再生能量估计、车辆响应间接标定 |
| 环境扰动 | 限速、车门/屏蔽门、信号、道岔、供电、通信等扰动注入 |
| 平台接入 | 对接信号、车辆/RT-LAB、司机台、视景等学校平台接口 |
| 实验记录 | 保存进路、占用、客流、调度、供电、扰动、接口健康数据 |

## 9.1 Phase 2 范围与运行模式

Phase 2 必须实现以下最小可用能力：

1. 同一线路上运行至少 3 列车，能够计算追踪间隔、区段占用和冲突风险。
2. 支持车站到车站或信号到信号的进路办理，进路锁闭后才能开放对应信号或生成移动授权。
3. 支持按时段变化的车站客流输入，列车负载反向影响停站时间和调度决策。
4. 支持供电分区最大牵引功率约束，并把 `tractionLimitRatio` 传递给车辆/ATO 控制。
5. 支持扰动事件注入和恢复，扰动能影响列控、联锁、车站、供电或接口健康状态。
6. 支持纯自研仿真、混合仿真和平台在环三种运行模式。

三种运行模式如下：

| 模式 | 英文名 | 使用场景 | 外部平台依赖 |
|---|---|---|---|
| 纯自研仿真 | Pure-Sim | 日常开发、批量优化、课堂展示 | 无 |
| 混合仿真 | Hybrid-Sim | 自研调度/客流 + 平台信号或车辆校验 | 可选 |
| 平台在环 | Platform-in-the-loop | 关键场景复核、接口联调、实验室展示 | 强依赖 |

Phase 2 默认仿真节拍：

| 子系统 | 推荐周期 | 说明 |
|---|---:|---|
| 主仿真循环 | 100 ms | 与信号/总控平台常见周期对齐 |
| 车辆积分 | 100 ms 或 20 ms | 自研模型可 100 ms，RT-LAB 车辆接口可 20 ms |
| 联锁和区段占用 | 100 ms | 确保进路、信号、MA 能及时更新 |
| 调度决策 | 1 s | 不需要每个主循环都重新优化 |
| 车站客流 | 1 s | 上下车、停站倒计时、站台人数更新 |
| 客流到达率刷新 | 30 s 或 60 s | 按客流曲线片段更新 |
| 记录落库 | 1 s，关键事件即时 | 状态快照低频，安全事件高频 |

## 9.2 Phase 2 主运行流程

Phase 2 的一次主循环按以下顺序执行：

```text
1. PlatformAdapter.poll()
   读取学校平台接口帧，更新信号、道岔、车辆、司机台、视景、RT-LAB 健康状态。

2. EnvironmentService.update()
   激活或恢复扰动事件，生成对子系统的影响因子。

3. SectionOccupationService.update(trainStates)
   根据列车头尾位置推导区段占用。

4. RouteService.update()
   处理进路请求、冲突检查、道岔锁闭、进路释放。

5. TrainControlService.update()
   基于进路、区段占用、限速、扰动、信号状态计算 MA/允许速度/目标点。

6. PowerService.update()
   聚合牵引功率请求，计算供电分区状态和牵引限额。

7. StationService.update()
   更新站台候车、上下车、列车负载、停站时间和发车条件。

8. DispatchService.update()
   基于时刻表、间隔、客流、负载、供电和扰动给出调度决策。

9. VehicleService.update()
   根据控制命令、供电限额和车辆动力学更新列车状态。

10. Recorder/UI/EventBus.flush()
    推送前端状态，记录事件和指标。
```

当外部平台接口不可用时，`PlatformAdapter` 不阻塞主循环，而是把对应适配器置为 `DEGRADED` 或 `OFFLINE`，系统自动回退到自研模型。

### 9.3.2 进路办理流程

```text
RouteRequest
  -> 校验进路存在
  -> 校验方向和起止信号
  -> 校验列车接近起始信号且未越过办理边界
  -> 校验逻辑区段、计轴区段、保护区段空闲
  -> 校验敌对进路未锁闭
  -> 校验所需道岔可转换且无故障
  -> 校验车站门、屏蔽门、站台发车条件
  -> 可选校验供电分区可用功率
  -> 锁闭道岔
  -> 锁闭进路区段和保护区段
  -> 开放起始信号或生成 MA
  -> 发布 RouteLocked 事件
```

失败时必须写入 `failureReason`，例如：

| 失败原因 | 说明 |
|---|---|
| `ROUTE_NOT_FOUND` | 进路表不存在 |
| `SECTION_OCCUPIED` | 区段被列车占用 |
| `CONFLICT_ROUTE_LOCKED` | 敌对进路已锁闭 |
| `SWITCH_UNAVAILABLE` | 道岔故障、失表或被其他进路锁闭 |
| `PLATFORM_NOT_READY` | 车门/屏蔽门/站台发车条件不满足 |
| `POWER_UNAVAILABLE` | 供电故障或严重欠压，不允许办理发车进路 |

### 9.3.3 进路释放流程

进路释放分为自动释放、取消释放和故障释放：

| 类型 | 条件 | 处理 |
|---|---|---|
| 自动释放 | 列车尾部清出进路最后一个释放区段 | 解锁进路区段、保护区段和相关道岔 |
| 取消释放 | 进路锁闭但列车未接近 | 可直接取消，信号关闭 |
| 接近锁闭释放 | 列车已接近起始信号 | 信号关闭后等待 `approachReleaseDelaySec` 再释放 |
| 故障释放 | 设备恢复或人工强制释放 | 记录人工/故障事件，要求 UI 二次确认 |

Phase 2 不实现真实安全系统的人工解锁流程认证，但必须保留事件记录和状态追踪，保证优化实验可追溯。

### 9.3.4 联锁与列控 MA 的关系

Phase 2 后，`TrainControlService` 的 MA 由以下边界共同决定：

```text
maEnd =
  min(
    routeEndSignalOrProtectionBoundary,
    firstOccupiedSectionBoundary,
    temporarySpeedRestrictionBoundary,
    platformStopPoint,
    disturbanceForcedStopPoint
  )
```

若进路未锁闭，起始信号必须保持禁止显示，列车 MA 不得越过该信号。若平台信号接口处于 `ONLINE`，系统可以采用两种策略：

1. `PLATFORM_AUTHORITATIVE`：以平台信号/道岔状态为准，自研联锁仅做一致性检查。
2. `SELF_AUTHORITATIVE`：以自研联锁为准，把列车状态和办理结果写给平台或前端展示。

默认开发模式采用 `SELF_AUTHORITATIVE`，接口联调模式采用 `PLATFORM_AUTHORITATIVE` 或 `COMPARE_ONLY`。

## 9.4 区段占用与多车跟踪设计

Phase 2 使用列车头尾位置推导区段占用。列车长度来自车辆参数，头部累计里程来自车辆状态，尾部位置按方向反算：

```text
headMileage = train.position.mileage
tailMileage = headMileage - directionSign * train.lengthM
occupiedSections = sections whose mileage interval overlaps [tailMileage, headMileage]
```

区段类型分为三层：

| 类型 | 作用 |
|---|---|
| `PHYSICAL_SEGMENT` | 前端轨道级展示、车辆定位 |
| `LOGICAL_SECTION` | 列控 MA、运行间隔、调度计算 |
| `AXLE_SECTION` | 联锁进路办理和释放 |

若初始线路数据缺少真实计轴区段，可按信号机、道岔、站台边界自动生成近似区段，并在配置中标记 `derived=true`。后续拿到老师或实验室的更完整联锁数据后，只需要替换 `route_catalog` 与 `section_catalog`，主逻辑不变。

多车跟踪输出：

```json
{
  "simTimeMs": 180000,
  "sections": [
    {
      "sectionId": "A-0901",
      "occupied": true,
      "trainIds": ["T001"],
      "source": "DERIVED_FROM_POSITION"
    }
  ],
  "headways": [
    {
      "frontTrainId": "T001",
      "rearTrainId": "T002",
      "distanceM": 850.0,
      "timeHeadwaySec": 96.5,
      "riskLevel": "NORMAL"
    }
  ]
}
```

当平台信号接口提供区段占用或信号设备状态时，`SectionOccupationService` 需要同时保存 `platformOccupied` 和 `selfOccupied`。两者不一致时发布 `event.adapter.inconsistency`，但不直接中断仿真。

## 9.5 车站客流与列车负载详细设计

车站模块负责把“客流需求”转化为“站台候车、列车载客量、停站时间和调度压力”。它是 Phase 2 调度优化的主要需求来源。

### 9.5.1 核心数据模型

客流到达率：

```json
{
  "stationId": "S-MLJ",
  "direction": "UP",
  "timeSliceSec": [25200, 28800],
  "arrivalRatePaxPerMin": 38.5,
  "odDistribution": [
    {"destStationId": "S-BJXI", "ratio": 0.42},
    {"destStationId": "S-GJJ", "ratio": 0.31}
  ]
}
```

站台候车状态：

```json
{
  "stationId": "S-MLJ",
  "direction": "UP",
  "waitingPax": 286,
  "platformDensityPaxPerM2": 2.4,
  "leftBehindPax": 36,
  "crowdingLevel": "HIGH"
}
```

列车负载状态：

```json
{
  "trainId": "T001",
  "onboardPax": 842,
  "capacityPax": 1460,
  "loadFactor": 0.58,
  "vehicleLoadKg": 54730
}
```

停站计划：

```json
{
  "trainId": "T001",
  "stationId": "S-MLJ",
  "plannedDwellSec": 35,
  "estimatedDwellSec": 49,
  "dispatchHoldSec": 12,
  "doorFaultExtraSec": 0,
  "canDepart": false,
  "blockingReason": "HEADWAY_TOO_SHORT"
}
```

### 9.5.2 客流生成

客流生成优先使用外部输入文件。没有真实客流数据时，使用场景文件中配置的分时到达率：

```text
newPassengers =
  poisson(arrivalRatePaxPerMin * dtSec / 60)
```

支持三类客流场景：

| 场景 | 特征 |
|---|---|
| 平峰 | 到达率低，站台基本无滞留 |
| 早晚高峰 | 到达率高，容易产生满载和站台滞留 |
| 突发客流 | 某站某方向在一段时间内到达率倍增 |

### 9.5.3 上下车与负载计算

到站后先下后上：

```text
alightingPax = onboardPax * alightingRatio(station, direction)
remainingCapacity = capacityPax - (onboardPax - alightingPax)
doorThroughputLimit = doorCapacityPaxPerSec * effectiveDwellSec
boardingPax = min(waitingPax, remainingCapacity, doorThroughputLimit)
leftBehindPax = waitingPax - boardingPax
onboardPaxNext = onboardPax - alightingPax + boardingPax
vehicleLoadKg = onboardPaxNext * averagePassengerWeightKg
```

列车负载量要传递给：

1. `VehicleService`：影响质量、加速度和能耗估算。
2. `PowerService`：影响牵引功率请求。
3. `DispatchService`：作为是否加开、扣车、跳停或延长停站的依据。
4. `PlatformAdapter`：可写入老师平台信号接口中列车负载字段。

### 9.5.4 停站时间模型

Phase 2 使用可解释的线性停站模型：

```text
dwellTimeSec =
  baseDwellSec
  + alpha * boardingPax
  + beta * alightingPax
  + gamma * platformDensityPaxPerM2
  + doorFaultExtraSec
  + dispatchHoldSec
```

最终停站时间需限制在配置区间：

```text
estimatedDwellSec = clamp(dwellTimeSec, minDwellSec, maxDwellSec)
```

发车条件：

| 条件 | 说明 |
|---|---|
| 门关闭 | 车门/屏蔽门状态正常 |
| 进路可用 | 发车进路已锁闭或 MA 已生成 |
| 调度放行 | 未被扣车或延长停站 |
| 供电允许 | 供电分区未处于严重欠压或断电 |
| 安全间隔 | 与前车间隔满足阈值 |

## 9.6 调度闭环详细设计

Phase 2 需要形成“客流-负载-停站-间隔-调度-运行”的闭环。调度模块不替代真实 ATS，而是实现可解释规则，为 Phase 3 的 AI 优化提供基线。

### 9.6.1 调度输入

| 输入 | 来源 |
|---|---|
| 计划时刻表 | 场景文件或数据库 |
| 列车位置、速度、延误 | `VehicleService`、`TrainState` |
| 区段占用、进路状态 | `SectionOccupationService`、`RouteService` |
| 站台候车、列车负载 | `StationService` |
| 供电状态 | `PowerService` |
| 扰动状态 | `EnvironmentService` |
| 平台接口健康 | `PlatformAdapter` |

### 9.6.2 调度决策模型

```json
{
  "decisionId": "DD-0001",
  "simTimeMs": 240000,
  "trainId": "T002",
  "stationId": "S-MLJ",
  "action": "HOLD",
  "durationSec": 18,
  "reason": "HEADWAY_TOO_SHORT",
  "expectedImpact": {
    "headwayBalanceImprovementSec": 22,
    "additionalPassengerWaitSec": 310
  }
}
```

支持的决策：

| 决策 | 说明 | Phase 2 实现方式 |
|---|---|---|
| `HOLD` | 扣车 | 在站内延长停站 |
| `RELEASE` | 放行 | 取消扣车，允许发车 |
| `DWELL_EXTEND` | 延长停站 | 用于客流上车或均衡间隔 |
| `SPEED_LEVEL_ADJUST` | 调整运行速度等级 | 给 ATO/车辆模块设置目标速度等级 |
| `ADD_TRAIN_REQUEST` | 建议加开备用车 | Phase 2 记录和展示，是否真正加车由场景配置决定 |
| `SKIP_STOP_CANDIDATE` | 跳停候选 | Phase 2 只做候选标记，不默认执行 |
| `STAGGER_DEPARTURE` | 错峰发车 | 供电峰值过高时延后部分列车发车 |

### 9.6.3 基线调度规则

Phase 2 默认采用规则调度：

```text
if powerSection.tractionLimitRatio < 0.8:
    stagger departures in the same power section
elif rearHeadwaySec < minHeadwaySec:
    hold current train
elif frontHeadwaySec > maxHeadwaySec and platformCrowdingLevel is HIGH:
    release train as soon as route is available
elif train.loadFactor > overloadThreshold and platform has high leftBehindPax:
    raise ADD_TRAIN_REQUEST
elif disturbance active:
    apply disturbance-specific recovery rule
else:
    follow timetable
```

调度模块必须输出“为什么这么调”的原因字段，便于后续把规则调度与 AI 调度对比。

## 9.7 供电约束详细设计

重新核对老师协议后，当前没有可直接调用的供电仿真接口。协议中出现的 `_nPowerState`、`_nKMAAndElectPower` 属于司机台/车辆显示或硬线状态，`PowerSystemAndTrainsV1/.../Train_Control/...` 属于 RT-LAB 工程变量路径，已列变量主要是列车编号、操作指令、Seg、偏移、方向和激活端，不能视为牵引供电网络接口。因此 Phase 2 供电模块必须自研，定位为“可解释的降阶牵引供电约束模型”。

Phase 2 供电设计目标：

1. 能计算列车牵引/制动引起的区段功率需求和能耗。
2. 能表达供电分区最大供电能力、启动峰值、欠压和断电扰动。
3. 能把供电约束反馈给车辆、ATO 和调度模块，形成闭环。
4. 能支持批量优化实验，而不是依赖昂贵外部仿真。
5. 能在后续获得真实供电参数或接口时，通过参数标定或 `PowerAdapter` 平滑升级。

供电模块不承担真实牵引供电保护整定、短路计算、继电保护或安全认证功能。

### 9.7.1 供电模型

```json
{
  "powerSectionId": "PWR-0901",
  "name": "郭公庄-六里桥",
  "mileageRangeM": [0.0, 6800.0],
  "maxTractionPowerKw": 8500,
  "warningPowerKw": 7200,
  "undervoltageThresholdV": 650,
  "outage": false,
  "connectedTrainIds": ["T001", "T002"]
}
```

供电状态：

```json
{
  "powerSectionId": "PWR-0901",
  "requestedPowerKw": 9300,
  "availablePowerKw": 8500,
  "tractionLimitRatio": 0.91,
  "voltageLevel": "LIMITED",
  "energyKwh": 126.4,
  "regenEnergyKwh": 18.7,
  "source": "SELF_SIM",
  "quality": "ESTIMATED"
}
```

`voltageLevel` 取值：

| 状态 | 条件 |
|---|---|
| `NORMAL` | 请求功率低于告警阈值 |
| `LIMITED` | 请求功率超过告警阈值但未严重欠压 |
| `UNDERVOLTAGE` | 请求功率超过最大供电能力，或场景扰动强制欠压 |
| `OUTAGE` | 供电中断 |

### 9.7.2 自研供电数据来源

Phase 2 初始没有真实供电图纸时，按以下优先级构造供电参数：

| 参数 | 优先来源 | 无数据时默认策略 |
|---|---|---|
| 供电分区边界 | 老师补充资料、线路供电图、牵引变电所位置 | 按相邻 2-3 个车站合并为一个供电分区 |
| 最大牵引功率 | 车辆额定功率、牵引变电所容量资料 | 按列车额定功率和同区间最大同时启动列车数估算 |
| 告警阈值 | 运维规则或实验参数 | `warningPowerKw = 0.85 * maxTractionPowerKw` |
| 欠压阈值 | 供电制式资料 | 使用场景配置，缺省只触发功率限额不计算真实电压 |
| 再生吸收上限 | 车辆制动能量回收资料 | 按同分区牵引需求优先吸收，剩余按比例损失 |
| 列车质量/阻力 | 车辆参数、RT-LAB 车辆响应标定 | 使用默认车辆参数，后续用速度/加速度校准 |

配置文件示例：

```json
{
  "powerSectionId": "PWR-0901",
  "stationRange": ["郭公庄", "六里桥东"],
  "mileageRangeM": [0.0, 6800.0],
  "maxTractionPowerKw": 8500,
  "warningPowerKw": 7200,
  "regenAbsorbLimitKw": 2500,
  "startupPeakWindowSec": 20,
  "maxSimultaneousStarts": 2,
  "derived": true,
  "assumption": "No official power interface; section generated from station range."
}
```

### 9.7.3 功率与能耗估算

```text
tractionPowerKw = tractionForceN * speedMps / 1000 / tractionEfficiency
brakeRegenPowerKw = brakeForceN * speedMps / 1000 * regenEfficiency
requestedPowerKw = sum(max(tractionPowerKw, 0) for trains in section)
energyKwh += tractionPowerKw * dtSec / 3600
regenEnergyKwh += brakeRegenPowerKw * dtSec / 3600
tractionLimitRatio = min(1.0, availablePowerKw / max(requestedPowerKw, epsilon))
```

当 `tractionLimitRatio < 1.0` 时：

1. `VehicleService` 限制牵引命令。
2. `TrainControlService` 可降低速度等级。
3. `DispatchService` 可错峰发车，避免同一供电分区内多车同时启动。
4. UI 显示供电分区颜色和限额原因。

### 9.7.4 降阶电压与限牵逻辑

Phase 2 不做完整潮流计算，采用分区级降阶模型：

```text
loadRatio = requestedPowerKw / maxTractionPowerKw

if outage:
    voltageLevel = OUTAGE
    tractionLimitRatio = 0
elif loadRatio <= warningRatio:
    voltageLevel = NORMAL
    tractionLimitRatio = 1
elif loadRatio <= 1:
    voltageLevel = LIMITED
    tractionLimitRatio = max(minLimitRatio, 1 - kLimited * (loadRatio - warningRatio))
else:
    voltageLevel = UNDERVOLTAGE
    tractionLimitRatio = max(minLimitRatio, 1 / loadRatio)
```

`tractionLimitRatio` 不是安全系统命令，而是仿真中的运行约束。车辆模块收到限额后应限制牵引级位；调度模块收到连续欠压后应触发 `STAGGER_DEPARTURE` 或延后发车。

### 9.7.5 再生制动吸收模型

再生能量按同一供电分区内“制动车产生、牵引车吸收、剩余损失”的顺序估算：

```text
regenGeneratedKw = sum(brakeRegenPowerKw for braking trains)
tractionDemandKw = sum(tractionPowerKw for traction trains)
absorbedRegenKw = min(regenGeneratedKw, tractionDemandKw, regenAbsorbLimitKw)
wastedRegenKw = regenGeneratedKw - absorbedRegenKw
netPowerKw = requestedTractionPowerKw - absorbedRegenKw
```

这能支持两个有研究价值的问题：

1. ATO 节能停车：通过惰行/制动时机减少总能耗。
2. 多车协同调度：通过错峰或同步牵引/制动提高再生吸收率。

### 9.7.6 与车辆/RT-LAB 的关系

车辆/RT-LAB 接口只作为车辆动力学对照和参数标定来源，不直接提供 `PowerState`。可用方式如下：

| 用途 | 输入 | 输出 | 说明 |
|---|---|---|---|
| 车辆参数标定 | 控制级位、速度、加速度、累计里程 | 牵引/制动能力、阻力系数修正 | 提高 `tractionForceN` 和 `brakeForceN` 估算可信度 |
| ATO 结果复核 | 同一控制策略在自研车辆和平台车辆上运行 | 运行时间、速度曲线、停车误差差异 | 复核控制策略鲁棒性 |
| 供电间接复核 | 平台车辆速度/加速度 + 自研供电模型 | 能耗/峰值功率估计 | 仍然是自研供电结果，不标记为平台供电结果 |

`PowerState.source` 当前只允许：

| source | 含义 |
|---|---|
| `SELF_SIM` | 自研供电估算 |
| `SCENARIO_FORCED` | 场景扰动强制设置，例如断电/欠压 |
| `FUTURE_PLATFORM` | 预留，只有获得明确供电协议后才能启用 |

### 9.7.7 供电模块验收补充

| 用例 | 场景 | 期望结果 |
|---|---|---|
| TC-P2-PWR-001 单车能耗 | 单车完整区间运行 | `energyKwh` 随牵引增加，制动阶段产生 `regenEnergyKwh` |
| TC-P2-PWR-002 多车峰值 | 同一供电分区两车同时启动 | `requestedPowerKw` 上升，可能触发 `LIMITED` |
| TC-P2-PWR-003 欠压限牵 | 请求功率超过最大能力 | `UNDERVOLTAGE`，`tractionLimitRatio < 1` |
| TC-P2-PWR-004 再生吸收 | 一车制动、一车牵引 | `absorbedRegenKw > 0`，净功率下降 |
| TC-P2-PWR-005 断电扰动 | 注入 `POWER_OUTAGE` | `tractionLimitRatio = 0`，列车停车或惰行 |
| TC-P2-PWR-006 平台误用防护 | RT-LAB 仅返回车辆变量 | 系统不得生成 `source=FUTURE_PLATFORM` 的 `PowerState` |

## 9.8 扰动与环境事件详细设计

扰动事件统一由 `EnvironmentService` 管理，所有扰动都必须有生命周期、影响范围和恢复条件。

### 9.8.1 扰动数据模型

```json
{
  "disturbanceId": "D001",
  "type": "TEMP_SPEED_RESTRICTION",
  "trigger": {"type": "TIME", "simTimeMs": 300000},
  "durationMs": 600000,
  "scope": {
    "stationIds": [],
    "fromSegId": 13,
    "toSegId": 24,
    "powerSectionIds": []
  },
  "severity": 0.8,
  "state": "ACTIVE",
  "parameters": {
    "speedLimitMps": 8.33
  }
}
```

扰动状态：

| 状态 | 说明 |
|---|---|
| `SCHEDULED` | 已配置但未触发 |
| `ACTIVE` | 正在影响仿真 |
| `RECOVERING` | 影响逐步消退 |
| `CLEARED` | 已恢复 |

### 9.8.2 支持的扰动类型

| 类型 | 影响对象 | 主要影响 |
|---|---|---|
| `TEMP_SPEED_RESTRICTION` | 列控/车辆 | 降低某区间允许速度 |
| `DOOR_FAULT` | 车站/车辆 | 增加停站时间或禁止发车 |
| `PSD_FAULT` | 车站/联锁 | 屏蔽门异常，发车条件不满足 |
| `PASSENGER_SURGE` | 客流/调度 | 到达率倍增，站台拥挤 |
| `SIGNAL_FAULT` | 联锁/列控 | 信号关闭或显示不一致 |
| `SWITCH_FAULT` | 联锁 | 道岔不可转换或失表 |
| `POWER_UNDERVOLTAGE` | 供电/车辆/调度 | 限制牵引，错峰发车 |
| `POWER_OUTAGE` | 供电/列控 | 区段断电，列车停车 |
| `COMMUNICATION_LOSS` | 平台接口 | 外部接口进入降级或离线 |

### 9.8.3 扰动传播规则

```text
TEMP_SPEED_RESTRICTION -> TrainControlService.speedRestrictions
DOOR_FAULT             -> StationService.doorState + DwellTimeModel.faultExtra
PSD_FAULT              -> StationService.canDepart=false
PASSENGER_SURGE        -> PassengerFlowGenerator.arrivalRateMultiplier
SIGNAL_FAULT           -> SignalAspectResolver.forceRestrictiveAspect
SWITCH_FAULT           -> SwitchLockService.health=FAILED
POWER_UNDERVOLTAGE     -> PowerService.availablePowerKw decreases
POWER_OUTAGE           -> PowerService.voltageLevel=OUTAGE
COMMUNICATION_LOSS     -> AdapterHealth.status=OFFLINE/DEGRADED
```

每个扰动都要输出恢复指标：

| 指标 | 含义 |
|---|---|
| `recoveryTimeSec` | 从扰动触发到系统恢复正常的时间 |
| `affectedTrainCount` | 受影响列车数量 |
| `totalDelaySec` | 总延误 |
| `maxPlatformCrowding` | 最大站台拥挤度 |
| `energyPenaltyKwh` | 能耗增加量 |

## 9.9 Phase 2 平台接口设计

## 10. Phase 3 概要设计

## 10.1 AI 优化实验框架

新增组件：

| 组件 | 职责 |
|---|---|
| ExperimentRunner | 批量运行场景 |
| ParameterSampler | 生成参数组合 |
| ObjectiveEvaluator | 计算目标函数 |
| OptimizerAdapter | GA/PSO/BO 接口 |
| ExperimentReportService | 输出优化报告 |

实验框架必须支持多保真验证：

```text
Level 0：Mock/规则模型快速跑通
Level 1：自研可解释模型批量优化
Level 2：平台接口模型复核候选策略中可由协议支撑的部分，例如车辆动力学、信号/司机台/视景；供电仍使用自研模型
Level 3：半实物/司机台/视景联动演示
```

每个优化实验至少输出：

1. 场景定义、随机种子、运行模式、接口健康摘要。
2. 基线策略、优化策略、消融策略对比。
3. 安全约束违反次数，必须为 0 或给出原因。
4. Pareto 前沿或多目标权衡曲线。
5. 自研模型结果与平台对照结果偏差分析，并注明哪些指标来自平台、哪些指标来自自研模型。

这样可以把平台接口变成“验证与增强研究可信度”的工具，而不是替代我们的算法工作。

## 10.2 自动停车与节能参数寻优

优化参数：

```text
coast_start_distance
brake_margin
expected_deceleration
max_traction_level
power_limit_threshold
```

目标函数：

```text
J =
  w1 * stop_error_norm
  + w2 * energy_norm
  + w3 * peak_power_norm
  + w4 * jerk_norm
  + w5 * time_error_norm
  + safety_penalty
```

第一版优化算法：

```text
GA 或 PSO
```

推荐研究问题：

```text
在满足 ATP 安全约束和停车误差 <= 0.5m 的前提下，
最小化区间能耗、峰值功率、冲击度 jerk，并控制运行时间偏差。
```

实验设计：

| 阶段 | 使用模型 | 价值 |
|---|---|---|
| 批量搜索 | 自研车辆 + 简化供电 | 低成本搜索 ATO 参数 |
| 参数标定 | 平台车辆/RT-LAB 输出 | 校准牵引/制动/阻力 |
| 候选复核 | 平台车辆接口 + 自研供电模型 | 检查速度/加速度/停车鲁棒性，并用自研供电模型复算能耗与功率峰值 |
| 人机演示 | 司机台 + 视景 | 展示优化前后驾驶体验 |

## 10.3 客流负载调度优化

调度变量：

```text
departure_time_adjustment
dwell_time_adjustment
skip_stop_decision
holding_time
```

目标：

```text
minimize total_delay
minimize passenger_waiting_time
minimize overload_penalty
minimize timetable_deviation
```

第一版可先用规则或遗传算法，强化学习作为扩展。

可形成的高价值问题：

```text
在客流时变、列车负载变化、供电峰值受限条件下，
动态调整发车间隔、停站时间和区间运行等级，
使乘客等待、拥挤惩罚、运行图偏差和峰值功率同时降低。
```

平台接口作用：

- 车辆平台提供不同调度策略下的速度、加速度、累计里程响应，用于校准车辆动力学。
- 供电功率、欠压和再生吸收由自研 `PowerService` 计算，不能标记为平台供电结果。
- 信号平台提供进路/信号约束，避免调度策略不可执行。
- 司机台/视景用于展示调度调整对驾驶与运行态势的影响。

## 10.4 扰动下运行图重排

算法候选：

## 18. 结论

本设计文档将项目第一阶段收束为一条可落地的主线：

```text
真实线路数据导入
  → 电子地图查询
  → 单列车沿 Seg 拓扑运行
  → ATO 自动控制
  → 基本限速/MA/ATP 防护
  → 到站停车与指标记录
  → 平台接口 Mock/适配
  → 多保真对照验证
```

Phase 0 和 Phase 1 完成后，系统即可形成一个有真实数据、有运行闭环、有安全约束、有指标输出的可演示原型。v0.2 的核心增强是：系统不再假设所有复杂子系统都必须由学生从零实现，而是在协议明确支持的信号、ATP-DMI、司机台、视景、车辆/RT-LAB 等模块上调用学校实验室平台接口；协议未覆盖的供电仿真则由我们自研。

这并不会降低项目含金量。相反，本项目的价值将集中在：

1. 建立统一线路数据、状态消息、时钟和记录体系。
2. 建立自研模型、Mock 模型、平台模型可切换的多保真架构。
3. 将学校平台接口转换为可复现、可回放、可评估的实验对象。
4. 在安全约束下研究 ATO 节能停车、客流负载调度、扰动重排、功率峰值抑制等优化问题。
5. 用平台已有接口复核优化策略中可观测的车辆、信号、司机台和视景结果，同时用自研供电模型评估能耗和功率约束，形成比单纯前端展示或单纯接口调用更有研究价值的闭环。

因此，后续开发应坚持“自研可解释模型打底、平台接口增强可信度、优化算法体现创新性”的路线。
## 匿名场景 A 牵引供电准静态仿真实现补充

### 闭环流程

```text
SimulationEngine
  -> 根据列车位置、速度、牵引/制动阶段生成 TrainPowerRequest
  -> PowerService 转换为 TrainElectricalLoad
  -> DCTractionPowerFlowSolver 求解列车受电电压、电流、牵引限功率、再生能量
  -> PowerService 聚合为旧 power 状态，并保留 last_network_snapshot
  -> SimulationEngine 将 tractionLimitRatio 反馈给列车推进逻辑
  -> API/前端/SQLite 记录输出供电网络快照
```

### 支持工况

1. 正常小双边供电：相邻牵引所共同向同一区间接触轨供电。
2. 多车同时牵引：牵引所电流和馈电臂电流叠加，受电电压下降。
3. 再生制动：再生功率优先被牵引列车吸收，其次由 EFS 容量吸收，剩余计入浪费。
4. 单座牵引所 N-1：支持通过 `/api/sim/power/faults` 将牵引所置为 OUTAGE，并闭合联络开关形成大双边工况。
5. 远程开关操作：支持通过 `/api/sim/power/switches/{switchId}/operate` 对联络开关执行 `OPEN/CLOSED` 操作，并刷新下一 tick 供电潮流。
6. 拓扑与结果落库：仿真引擎加载时将 V0 供电拓扑写入 SQLite 静态表；运行时逐 tick 写入列车电压、牵引所潮流和再生能量。

### 输出指标

| 指标 | 用途 |
|---|---|
| trainVoltageV/currentA | 判断列车受电质量与牵引限制 |
| tractionLimitRatio | 反馈车辆动力学/ATO 的限功率系数 |
| substation current/power/loadRatio | 判断牵引所过载 |
| feeder current/loadRatio | 判断馈电臂过流 |
| absorbed/feedback/wasted regen | 分析再生制动利用 |
| lossesKw | 分析线路损耗 |
| alerts | 欠压、过压、过载、再生浪费等事件 |

### 验证

当前新增测试覆盖：

- 10 座牵引所 V0 拓扑加载。
- 按列车里程定位相邻牵引所。
- 单车中点双边供电左右分流。
- 单车靠近一端时近端牵引所承担更多电流。
- 多车同时启动导致电压下降和负载告警。
- 再生制动被牵引列车吸收。
- 再生无法吸收时产生浪费和限再生告警。
- 单座牵引所退出产生 OUTAGE 告警。
- 主引擎快照和 SQLite 记录包含供电网络结果。
