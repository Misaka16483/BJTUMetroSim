# 正线范围与车辆段数据隔离设计

## 1. 背景

原始《线路数据》同时包含正线、折返线、出入段线、车辆段和库线数据。旧实现将全部数据导入 `line_map.json`，普通运营仿真再在相邻车站的所有站台组合中选取最短 Seg 路径。该方式没有明确阻止搜索进入车辆段支路。

本设计将“全量电子地图”和“正线运行范围”分层，以满足“通常仅模拟主线路运行”的要求。

## 2. 设计决策

1. `data/cache/line_map.json` 继续保留全量数据，不修改、不删除车辆段对象。
2. 新增 `data/scenarios/line9_mainline_scope.json`，作为普通9号线运营场景允许访问的 Seg 白名单。
3. 场景通过 `lineScopeFile` 显式声明使用哪个运行范围。
4. `PathPlanner` 只能在白名单内扩展相邻 Seg；白名单外的起点、终点或中间 Seg 均被拒绝。
5. 启用范围后，相邻车站找不到范围内路径时直接报错，不允许静默回退为公里标差仿真。
6. 特殊的车辆段、出入库或调车场景可以不配置该范围，或配置独立的车辆段范围文件。

## 3. 数据结构

`line9_mainline_scope.json` 包含：

| 字段 | 含义 |
|---|---|
| `schemaVersion` | 范围文件结构版本 |
| `scopeId` | 可记录、可对比的范围版本ID |
| `lineId` | 所属线路，必须与场景一致 |
| `segmentIds` | 运行时允许访问的 Seg 集合 |
| `stationPairPaths` | 12个相邻运营站的已审核路径 |
| `source` | 数据来源 |
| `selectionRule` | 范围生成和审核规则 |

加载时会校验 `segmentIds` 必须等于 `stationPairPaths` 中所有 Seg 的并集，防止修改路径后遗漏同步白名单。

## 4. 运行时流程

```text
加载场景
  -> 读取 lineScopeFile
  -> 校验 lineId 和 Seg ID
  -> 构建受限 PathPlanner
  -> 按公里标匹配运营车站与站台
  -> 在白名单内搜索相邻站路径
  -> 选取范围内最短可行路径
  -> 加载该路径的限速和坡度
  -> 车辆动力学积分
```

引擎快照 KPI 同时输出 `lineScopeEnforced`、`lineScopeId` 和 `lineScopeSegmentCount`，便于API、记录和验收判断当前运行是否受正线范围保护。

## 5. 当前范围基线

`line9-mainline-v1` 包含77个 Seg，覆盖郭公庄至国家图书馆的12个相邻站区间。当前范围基于旧 `line_map.json` 和13站 `stations.csv` 固化，不等同于尚未完成的新上下行 Link v2 数据迁移。

## 6. 验收条件

1. 单车和五车场景能加载 `line9-mainline-v1`。
2. 12个相邻站区间的上、下行路径全部存在。
3. 所有运行路径的 Seg 都是 `segmentIds` 的子集。
4. 白名单外的 Seg（回归用例使用 Seg 307）不能作为受限路径的起点或中间节点。
5. 全量 `line_map.json` 的319个 Seg、60个道岔和249条进路仍然保留，供特殊场景使用。

## 7. 已知边界

1. 当前77个 Seg 是从旧电子地图的已有站间路径固化而来，仍需结合CAD或新 Link 表完成工程复核。
2. 静态 `track-map` API 仍返回全量对象，但增加了正线范围元数据；仿真引擎的路径搜索已强制执行白名单。
3. 联锁领域包尚未进入主 tick，本次先保证车辆运行路径不越界；联锁接入主循环时应复用同一 `LineScope` 过滤进路、道岔和区段。
