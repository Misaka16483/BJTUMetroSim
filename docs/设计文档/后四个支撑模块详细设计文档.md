# 后四个支撑模块详细设计文档

本文档面向轨道交通综合仿真系统的后四个通用支撑模块，给出可落地的软件详细设计。后四个模块包括：

9. 通信接口与数据交换模块
10. 仿真时钟与场景管理模块
11. 数据记录、评估与回放模块
12. 可视化与人机交互模块

这四个模块不直接表达某一个轨道交通业务对象，但它们决定整个系统能否稳定联动、可复现、可演示、可评估。

## 1. 设计目标

系统应支持以下核心闭环：

```text
场景加载
  ↓
仿真时钟驱动
  ↓
各业务模块更新状态
  ↓
通信接口交换数据
  ↓
控制指令生成并下发
  ↓
数据记录与指标评估
  ↓
界面实时展示与回放
```

后四个模块的共同目标：

1. 统一数据交换格式。
2. 保证各模块在同一仿真时间线上运行。
3. 记录完整实验过程，支持复盘和答辩展示。
4. 提供清晰可操作的人机界面。
5. 支持真实设备接入和纯软件仿真两种模式。

## 2. 总体架构

推荐采用分层架构：

```text
┌────────────────────────────────────┐
│ 12. 可视化与人机交互模块             │
├────────────────────────────────────┤
│ 11. 数据记录、评估与回放模块          │
├────────────────────────────────────┤
│ 10. 仿真时钟与场景管理模块            │
├────────────────────────────────────┤
│ 9. 通信接口与数据交换模块             │
├────────────────────────────────────┤
│ 前八个业务仿真模块/外部仿真设备        │
└────────────────────────────────────┘
```

运行时可以采用“事件总线 + 共享状态快照”的方式：

- 高频状态，例如速度、位置、牵引制动级位，进入状态快照。
- 低频事件，例如到站、开门、故障、模式切换，进入事件流。
- 控制指令，例如牵引、制动、紧急制动，进入命令通道。

## 4. 模块 9：通信接口与数据交换模块

## 4.1 模块职责

通信接口与数据交换模块负责连接所有业务模块、控制程序、实体司机台和外部仿真系统。

主要职责：

1. 适配不同通信方式。
2. 解析外部协议。
3. 将外部数据转换为统一内部数据模型。
4. 将内部控制指令转换为外部设备协议。
5. 维护模块健康状态。
6. 提供重连、超时、丢包检测和日志。

## 4.2 子组件设计

| 子组件 | 职责 |
|---|---|
| ProtocolAdapter | TCP/UDP/串口/文件回放等协议适配 |
| MessageCodec | 编码、解码、校验和字段映射 |
| StateBus | 发布状态消息 |
| CommandBus | 传递控制指令 |
| MessageRouter | 根据 messageType 分发消息 |
| DeviceManager | 管理外部设备连接状态 |
| HeartbeatMonitor | 心跳检测和超时判定 |
| ErrorHandler | 通信异常、解析异常和重连处理 |
| ProtocolLogger | 记录原始收发报文 |

## 4.3 运行流程

### 4.3.1 接收外部状态

```text
外部设备/仿真系统
  → ProtocolAdapter 接收字节流
  → MessageCodec 解码
  → 字段映射为 TrainState/SignalState/PowerState
  → MessageRouter 发布到 StateBus
  → 数据记录模块订阅并写入日志
  → 可视化模块订阅并刷新界面
```

### 4.3.2 下发控制指令

```text
司机台/自动控制模块生成 ControlCommand
  → CommandBus
  → MessageRouter
  → MessageCodec 转换为外部协议
  → ProtocolAdapter 发送
  → 等待 ACK 或状态反馈
```

## 4.4 接口设计

### 4.4.1 内部接口

```text
publish(topic, message)
subscribe(topic, handler)
sendCommand(command)
getConnectionStatus(deviceId)
registerAdapter(adapterConfig)
```

主题建议：

```text
state.train
state.signal
state.power
state.station
state.dispatch
command.train
command.driver
event.simulation
health.device
```

### 4.4.2 外部协议适配

至少支持三种模式：

| 模式 | 用途 |
|---|---|
| TCP/UDP 模式 | 对接真实仿真台或外部仿真程序 |
| Serial 模式 | 对接实体司机台、手柄或控制器 |
| Replay 模式 | 无设备时读取历史数据文件进行演示 |

配置示例：

```json
{
  "adapterId": "simulator-main",
  "type": "TCP_CLIENT",
  "host": "127.0.0.1",
  "port": 5001,
  "timeoutMs": 1000,
  "heartbeatIntervalMs": 500,
  "reconnect": true
}
```

## 4.5 异常处理

| 异常 | 处理策略 |
|---|---|
| 连接失败 | 进入 DISCONNECTED，按重连间隔重试 |
| 报文校验失败 | 丢弃该报文，记录原始数据和错误原因 |
| 数据字段缺失 | 使用默认值或标记为 INVALID |
| 状态超时 | 发布 health.device 告警 |
| 指令无 ACK | 重发或提示人工接管 |
| 外部设备断开 | 自动切换到安全状态，停止下发牵引指令 |

## 4.6 状态机

```text
INIT
  ↓
CONNECTING
  ↓ 成功
CONNECTED
  ↓ 收到有效心跳
READY
  ↓ 通信中断
DEGRADED
  ↓ 重连失败
DISCONNECTED
```

## 4.7 验收标准

- 能在无真实设备情况下通过 Replay 模式驱动系统运行。
- 能接收并解析至少一种外部状态报文。
- 能下发至少一种控制指令。
- 通信中断后界面有明确提示。
- 所有原始报文和解析结果可追踪。

## 5. 模块 10：仿真时钟与场景管理模块

## 5.1 模块职责

该模块负责统一仿真时间、调度各模块更新、加载场景、注入事件和控制仿真生命周期。

主要职责：

1. 管理仿真开始、暂停、继续、停止、重置。
2. 维护统一仿真时间。
3. 按固定步长驱动各模块。
4. 加载线路、车辆、信号、供电、调度和扰动配置。
5. 支持倍速、单步、实时模式。
6. 保证场景可重复运行。

## 5.2 子组件设计

| 子组件 | 职责 |
|---|---|
| SimulationClock | 维护仿真时间和运行速率 |
| TickScheduler | 按步长触发模块更新 |
| ScenarioLoader | 加载场景配置文件 |
| ScenarioValidator | 检查场景完整性 |
| ModuleLifecycleManager | 管理模块初始化、启动、停止 |
| EventInjector | 按时间或位置注入扰动事件 |
| SnapshotManager | 保存和恢复仿真状态 |

## 5.3 仿真模式

| 模式 | 说明 |
|---|---|
| REALTIME | 仿真时间与真实时间同步 |
| FAST | 倍速运行，用于快速测试 |
| STEP | 单步执行，用于调试 |
| PAUSE | 暂停，状态保持不变 |
| REPLAY | 按记录数据回放 |

## 5.4 生命周期状态机

```text
IDLE
  ↓ loadScenario
LOADED
  ↓ start
RUNNING
  ↓ pause
PAUSED
  ↓ resume
RUNNING
  ↓ stop
STOPPED
  ↓ reset
IDLE
```

异常状态：

```text
ERROR
```

进入 ERROR 的条件：

- 场景文件缺失关键字段。
- 模块初始化失败。
- 外部设备连接失败且当前场景要求必须连接设备。
- 仿真过程中出现不可恢复异常。

## 5.6 调度算法

推荐固定步长调度：

```text
while running:
  1. 计算当前 simTime
  2. 读取上一周期状态快照
  3. 注入到期事件
  4. 按顺序更新业务模块
  5. 发布新状态快照
  6. 记录数据
  7. 推送界面刷新
  8. 等待下一个 tick
```

默认更新顺序：

```text
环境扰动
→ 调度运行图
→ 线路轨道
→ 信号列控
→ 供电
→ 司机台/控制输入
→ 车辆动力学
→ 车站站台
→ 评估记录
```

## 5.7 验收标准

- 同一场景多次运行结果一致。
- 支持开始、暂停、继续、停止和重置。
- 能按时间触发至少一种扰动。
- 能按位置触发至少一种扰动。
- 场景配置错误时能给出明确错误信息。

## 6. 模块 11：数据记录、评估与回放模块

## 6.1 模块职责

该模块负责将仿真过程转化为证据。它不仅用于调试，也用于答辩展示和项目评分。

主要职责：

1. 记录状态、指令、事件和原始报文。
2. 计算运行指标。
3. 支持实验回放。
4. 导出 CSV、JSON 或报告。
5. 支持对比不同算法或不同场景。

## 6.2 子组件设计

| 子组件 | 职责 |
|---|---|
| TelemetryCollector | 订阅状态和指令消息 |
| RunRecorder | 按运行批次写入数据库或文件 |
| MetricEngine | 计算停车误差、运行时间、能耗等 |
| EventLog | 记录到站、故障、模式切换等事件 |
| ReplayService | 根据记录数据驱动系统回放 |
| ReportExporter | 导出实验报告和图表数据 |
| ComparisonAnalyzer | 对比多次运行结果 |

## 6.4 指标设计

### 6.4.1 停车误差

```text
stop_error_m = actual_stop_position_m - target_stop_position_m
abs_stop_error_m = abs(stop_error_m)
```

合格条件示例：

```text
abs_stop_error_m <= 1.0
```

### 6.4.2 运行时间

```text
run_time_s = arrival_time_s - departure_time_s
```

### 6.4.3 准点偏差

```text
arrival_deviation_s = actual_arrival_time_s - scheduled_arrival_time_s
```

### 6.4.4 平稳性

可用加速度变化率近似：

```text
jerk_mps3 = Δacceleration / Δtime
```

简化指标：

```text
max_abs_jerk
average_abs_jerk
traction_brake_switch_count
```

### 6.4.5 能耗

简化计算：

```text
energy_kwh = Σ power_kw * Δt_h
```

### 6.4.6 安全约束

```text
overspeed_count
red_signal_violation_count
emergency_brake_count
movement_authority_violation_count
```

## 6.5 回放设计

回放流程：

```text

  ↓

  ↓

  ↓

  ↓

```

回放控制：

```text

暂停
倍速
拖动进度

```

## 6.7 验收标准

- 能完整记录一次运行的状态、指令和事件。
- 能计算停车误差、运行时间、最大速度和超速次数。
- 能导出 CSV 文件。
- 能选择历史运行并回放。
- 回放时界面状态与记录数据一致。

## 7. 模块 12：可视化与人机交互模块

## 7.1 模块职责

该模块负责让用户看见、操作和理解仿真过程。它是最终演示效果最直接的模块。

主要职责：

1. 实时展示列车运行状态。
2. 展示线路、站点、信号、供电和目标停车点。
3. 提供手动驾驶控制。
4. 提供场景加载、运行控制和模式切换。
5. 展示运行曲线、日志和评估结果。
6. 支持历史回放。

```text
开门/关门按钮
模式切换：手动/自动/回放
确认按钮
```

约束：

- 自动模式下手动牵引/制动默认禁用，除非执行人工接管。
- 紧急制动按钮始终可用。
- 列车未停稳时开门按钮禁用。
- 通信未连接时禁止下发真实设备指令。

## 9. 非功能需求

## 9.1 性能

- 状态记录频率：建议不低于 10 Hz。
- 界面刷新频率：建议 5-10 Hz。
- 控制指令延迟：本地仿真模式下建议小于 100 ms。
- 单次运行数据量：至少支持 30 分钟连续记录。

## 9.2 可靠性

- 通信异常不能导致程序崩溃。
- 场景配置错误应明确提示。
- 数据记录失败应提示用户，不应静默丢失。
- 紧急制动指令优先级最高。

## 9.3 可测试性

- 所有模块应支持无设备测试。
- 通信模块应支持 MockAdapter。
- 时钟模块应支持 STEP 模式。
- 数据模块应支持固定输入数据回放。
- 界面模块应能连接模拟数据源。

## 9.4 可扩展性

- 新增业务模块时，只需定义消息类型并订阅/发布主题。
- 新增外部设备时，只需新增 ProtocolAdapter。
- 新增指标时，只需扩展 MetricEngine。
- 新增场景时，只需添加场景配置文件。

## 12. 推荐实现顺序

1. 定义统一数据模型。
2. 实现仿真时钟和场景加载。
3. 实现 Mock 通信适配器。
4. 实现 TrainState 和 ControlCommand 数据流。
5. 实现数据记录表。
