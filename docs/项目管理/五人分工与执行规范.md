# 轨道交通仿真系统五人分工与执行规范

版本：v0.1  
日期：2026-07-08  
适用范围：Phase 0、Phase 1、Phase 2 开发与后续 Phase 3 优化实验准备  

## 1. 分工原则

本项目采用“五人并行开发 + 模块主责 + 交叉评审”的方式推进，不单独设立项目负责人。每名成员都必须承担明确代码工作，项目管理、资料整理、文档撰写、答辩材料等工作作为附加职责分配到各成员，不设置只做管理或只写文档的角色。

基本原则：

1. 每人至少负责一个可运行代码模块，并配套完成测试、接口说明和演示样例。
2. “主责”只表示该成员对某模块的代码完整性、接口稳定性和测试覆盖负责，不代表其管理其他成员。
3. 所有跨模块接口必须写入文档或类型定义，不能只靠口头约定。
4. 文档工作必须服务代码落地，不能脱离实现单独堆材料。
5. 每周每人都应有可检查产出：代码提交、测试结果、接口文档、调研补充或演示截图至少一类。

| 成员 C | 信号列控、计算机联锁、区段占用、扰动安全约束 | TrainControlService、MA/限速/ATP、RouteService、SwitchLockService、SectionOccupationService、EnvironmentService | 整理信号/ATP-DMI/联锁规则说明、协议能力边界核查 |

## 3. 代码模块边界

建议代码目录按以下边界组织。实际目录可根据现有工程微调，但模块职责不能混乱。

```text
app/
  core/                 # 成员 A：时钟、消息、场景、记录
  data_import/           # 成员 A：线路数据导入与校验
  domain/
    track/               # 成员 A：电子地图、Seg、站台、信号机静态数据
    vehicle/             # 成员 B：车辆动力学
    control/             # 成员 B/C：ATO、司机台、列控
    interlocking/        # 成员 C：联锁、进路、道岔、区段占用
    station/             # 成员 D：客流、上下车、停站
    dispatch/            # 成员 D：运行图、扣车、放行、调度规则
    power/               # 成员 D：自研供电约束、能耗、再生吸收
    disturbance/         # 成员 C：扰动事件、安全影响
  adapters/              # 成员 B/C/E：平台协议 Mock 与适配器
  api/                   # 成员 E：REST/WS
  tests/                 # 所有人：各自模块测试
<repository-root>/       # 成员 E：前端工程
docs/                    # 所有人：与代码对应的文档
```

模块之间只通过标准数据模型通信，例如 `TrainState`、`ControlCommand`、`MovementAuthority`、`PowerState`、`RouteState`、`StationState`、`DispatchDecision`、`AdapterHealth`。禁止跨模块直接读取对方内部变量。

## 4. 分阶段任务拆分

## 4.1 Phase 0：数据基线与最小框架

| 成员 | 必做代码任务 | 验收产出 |
|---|---|---|
| A | 线路 Excel 解析、line_map 缓存、拓扑校验、仿真时钟、Recorder 初版 | 可导入 `线路数据(1).xls`，输出对象数量、校验报告、缓存文件 |
| B | 车辆参数模型、TrainState 初版、简单车辆推进接口 | 单车可按牵引/制动命令更新速度和位置 |
| C | 静态限速、基本信号状态、MA 数据结构 | 可查询某位置前方限速和信号约束 |
| D | PowerState 默认版、StationState 默认版、指标结构 | 输出默认供电状态和站台基础状态 |
| E | 后端 `/api/health`、`/api/lines/9/track-map`、前端 9 号线数据接入 | 前端能显示 9 号线静态数据或接口返回结果 |

## 4.2 Phase 1：单车运行与安全闭环

| 成员 | 必做代码任务 | 验收产出 |
|---|---|---|
| A | 场景加载、运行记录、回放数据导出 | 单车仿真可保存速度、位置、命令、事件 |
| B | ATO 自动停车、司机台控制合成、车辆/RT-LAB Mock 骨架 | 输出停车误差、运行时间、速度曲线 |
| C | ATP 超速/越权制动、MA 更新、信号约束事件 | 超速或越过 MA 时触发安全事件 |
| D | 能耗估算、站台到站判定、基础评价指标 | 输出能耗、停车成功/越停/欠停结果 |
| E | 仿真启动/暂停/恢复 API，前端运行面板和曲线展示 | 前端能看到列车运行状态和关键指标 |

## 4.3 Phase 2：多车、联锁、客流、供电与调度闭环

| 成员 | 必做代码任务 | 验收产出 |
|---|---|---|
| A | 多车场景管理、SQLite 表扩展、批量运行记录 | 多车状态和事件可落库，可回放 |
| B | 车辆模型标定接口、负载影响车辆质量、车辆平台对照报告 | 列车负载变化能影响加速度/能耗估算 |
| C | 进路办理、道岔锁闭、区段占用、扰动安全传播 | 敌对进路和占用区段能被阻止 |
| D | 客流生成、上下车、停站修正、自研供电约束、规则调度 | 客流-负载-停站-供电-调度形成闭环 |
| E | 宏观线路图、微观轨道图、进路/供电/客流/接口状态 UI | 前端能解释列车、进路、占用、客流、供电和扰动状态 |

## 4.4 Phase 3：AI 优化准备

Phase 3 暂不要求所有算法一次完成，但 Phase 2 代码必须为优化实验留下接口。

| 成员 | 准备任务 |
|---|---|
| A | `ExperimentRunner`、随机种子、批量结果目录和指标汇总 |
| B | ATO 参数可配置化，支持 GA/PSO/BO 搜索停车策略 |
| C | 安全约束检查器，保证优化策略不能绕过 ATP/联锁约束 |
| D | 调度/供电/客流多目标评价函数 |
| E | 优化实验结果展示、Pareto 曲线、策略对比页面 |

## 5. 协作与执行规范

## 5.1 分支规范

建议分支命名：

```text
backend-a-track-core
backend-b-vehicle-ato
backend-c-signal-interlocking
backend-d-station-dispatch-power
frontend-e-visual-api
fix/<short-description>
doc/<short-description>
```

提交到共享分支前必须：

1. 本地能启动或至少能运行相关单元测试。
2. 不提交临时文件、缓存文件、个人 IDE 配置和大体积无关文件。
3. 若修改了接口、数据模型或场景格式，必须同步更新文档。

## 5.2 Commit 规范

提交信息使用以下格式：

```text
type(scope): summary
```

示例：

```text
feat(track): add line9 segment topology loader
fix(vehicle): clamp brake command to safe range
test(power): add undervoltage traction limit case
docs(api): document route request endpoint
```

常用 `type`：

| type | 用途 |
|---|---|
| `feat` | 新功能 |
| `fix` | 缺陷修复 |
| `test` | 测试 |
| `docs` | 文档 |
| `refactor` | 不改变行为的重构 |
| `chore` | 工程配置、依赖、脚本 |

## 5.3 代码规范

后端 Python 规范：

1. 使用类型标注，核心数据结构优先使用 `dataclass` 或 Pydantic 模型。
2. 单位必须写入字段名，例如 `speed_mps`、`distance_m`、`time_ms`、`power_kw`。
3. 不在业务代码中硬编码协议端口、IP、RT-LAB 路径和线路文件路径，应放入配置。
4. 不用字符串到处传状态，状态值应集中定义枚举或常量。
5. 所有跨模块输入都要做边界检查，例如速度非负、牵引级位范围、区段 ID 存在。
6. 平台接口报文解析必须保留原始帧摘要和解析后字段，便于回放和定位问题。

前端规范：

1. API 类型集中放在 `src/types/`，接口请求集中封装。
2. 宏观图、微观图、状态面板拆成独立组件，不把全部逻辑堆在 `App.tsx`。
3. 所有平台接口状态必须显示来源：`SELF_SIM`、`MOCK`、`PLATFORM`、`DEGRADED`。
4. 供电指标必须标注为自研 `PowerService` 输出，不能显示为 RT-LAB 供电结果。
5. 图层状态使用统一颜色语义：正常、限制、故障、离线、选中。

## 5.4 接口规范

后端 API 必须遵守：

1. 路径使用小写和短横线，例如 `/api/lines/9/track-map`。
2. 返回 JSON 使用稳定字段名，禁止随意改字段。
3. 每个状态响应都包含 `simTimeMs` 或 `timestamp`。
4. 错误响应必须包含 `code`、`message`、`detail`。
5. WebSocket 主题必须与设计文档一致，例如 `state.power`、`state.interlocking`、`state.adapter`。

典型错误响应：

```json
{
  "code": "ROUTE_CONFLICT",
  "message": "Route cannot be locked because a conflicting route is active.",
  "detail": {
    "routeId": "R-001",
    "conflictRouteId": "R-002"
  }
}
```

## 5.5 测试规范

每人对自己模块至少提供以下测试：

| 测试类型 | 要求 |
|---|---|
| 单元测试 | 核心算法、状态转换、边界条件必须覆盖 |
| 集成测试 | 与相邻模块至少跑通一个端到端场景 |
| 场景测试 | 使用固定随机种子，输出可复现指标 |
| 异常测试 | 缺数据、越界、接口断连、扰动触发等情况不能崩溃 |

最低验收线：

1. Phase 0：数据导入和 API 启动测试必须通过。
2. Phase 1：单车自动停车场景必须可重复运行。
3. Phase 2：多车联锁冲突、客流停站、供电限牵、扰动恢复至少各有一个测试。

## 5.6 文档规范

文档不是单独岗位。每个成员必须维护自己代码对应的文档：

| 文档类型 | 维护方式 |
|---|---|
| 接口说明 | 修改 API 或消息模型时同步更新 |
| 模块设计 | 完成模块初版后补充数据结构、流程和边界 |
| 测试说明 | 写明场景、输入、期望输出、实际结果 |
| 调研补充 | 只补充会影响需求、参数、规则或验收的内容 |
| 演示说明 | 写清启动命令、演示步骤、预期画面 |

建议每个模块文档包含：

```text
模块目标
输入/输出
核心数据模型
主要流程
异常处理
测试用例
未完成事项
```

## 5.7 评审规范

每个合并请求至少需要一名其他成员检查。检查重点：

1. 是否能运行。
2. 是否破坏已有接口。
3. 是否有测试或可复现场景。
4. 是否同步更新文档。
5. 是否存在跨模块硬编码或重复造模型。
6. 是否把协议未提供的能力误写为平台接口，尤其供电仿真。

评审不是审批权力，而是质量互保。被评审者负责修改，评审者负责指出风险。

## 6. 周期节奏

建议每周固定两次短同步：

| 时间点 | 内容 | 产出 |
|---|---|---|
| 周初 | 各成员确认本周代码任务、接口依赖、阻塞点 | 本周任务表 |
| 周末 | 演示各自可运行内容，记录测试结果和问题 | 周进展记录 |

每次同步控制在 30 分钟内，重点看运行结果和接口问题，不做长篇口头汇报。

## 7. 验收口径

一个任务只有同时满足以下条件，才算完成：

1. 代码已提交。
2. 能在本地按说明运行。
3. 有至少一个测试或演示场景。
4. 相关接口和数据结构已写入文档。
5. 与相邻模块没有未说明的破坏性变更。
6. 若使用平台接口，已标明 `MOCK`、`PLATFORM`、`DEGRADED` 等来源。

不能只以“写完代码”“能在我电脑上跑”“文档写了”作为完成标准。

## 8. 风险与约束

| 风险 | 约束 |
|---|---|
| 分工变成各写各的 | 所有模块必须围绕统一数据模型和消息总线集成 |
| 前端与后端字段不一致 | API 类型由成员 E 维护，但字段变更必须由对应后端成员确认 |
| 平台接口被误解 | 以老师协议为准；没有明确字段就不能写成已有接口 |
| 供电仿真被低估 | Phase 2 明确由成员 D 自研降阶供电模型，成员 B 提供车辆参数标定输入 |
| 文档滞后 | 合并代码前必须检查对应文档是否更新 |
| 测试缺失 | 没有测试或可复现场景的任务不能进入阶段验收 |

## 9. 最小阶段交付清单

Phase 0 最小交付：

- 线路数据导入成功。
- 后端 API 能返回 9 号线静态数据。
- 前端能显示 9 号线或轨道级基础数据。
- 有导入校验报告和运行说明。

Phase 1 最小交付：

- 单车从起点运行到目标站台。
- ATO 能自动停车。
- ATP 能处理超速或越权。
- Recorder 能输出速度、位置、命令、事件和指标。
- 前端能显示运行状态。

Phase 2 最小交付：

- 至少 3 列车运行。
- 进路办理、敌对进路阻止、区段占用可见。
- 客流影响上下车、负载和停站时间。
- 自研供电模型能触发限牵和错峰发车。
- 扰动事件能影响运行并记录恢复指标。
- 前端宏观图和微观图能解释运行状态。

Phase 3 准备交付：

- 至少一个可批量运行的优化实验入口。
- 至少一个基线策略和一个优化策略对比。
- 输出指标表和图形化结果。
- 明确哪些结果来自平台可观测接口，哪些结果来自自研模型。
