## 4. 客流、停站与调度闭环

### 4.1 客流状态模型

我实现了以下车站与列车状态：

- `PassengerDemandProfile`：按车站、方向和时段定义乘客到达率及下车比例。
- `PlatformCrowdState`：维护候车人数、滞留人数、站台面积、密度和拥挤等级。
- `TrainLoadState`：维护列车人数、定员、负载率和乘客质量。
- `BoardingResult`：记录到达、上车、下车、剩余候车和滞留人数。
- `DwellPlan`：记录计划停站、客流修正、调度扣车、门故障和发车条件。

客流生成器采用确定性小数结转方法。这样既能模拟连续到达率，又能保证同一场景重复运行时结果一致，适合作为自动测试和优化实验基线。

### 4.2 上下客与停站时间

列车到站时，系统按照当前车载人数、下车比例、剩余容量、站台候车人数和车门通过能力计算上下客：

```text
下车人数 = min(车载人数, 车载人数 × 下车比例)
可上车人数 = min(候车人数, 剩余容量, 车门通过能力)
滞留人数 = 候车人数 - 实际上车人数
```

停站时间采用可解释的线性修正模型：

```text
停站时间 = 基础停站时间
         + 上车人数系数 × 上车人数
         + 下车人数系数 × 下车人数
         + 站台密度系数 × 站台密度
         + 调度扣车时间
         + 门故障附加时间
```

结果限制在最小/最大停站时间范围内。该模型能够解释大客流为什么会延长停站并传播延误，同时为后续用回归模型标定参数留下接口。

### 4.3 规则调度基线

我实现了 `RuleBasedDispatchService`，根据供电限制、前后车间隔、站台拥挤、列车满载、滞留乘客、扰动和进路可用性输出调度决策：

| 条件 | 决策 | 作用 |
|---|---|---|
| 牵引限制过低 | `STAGGER_DEPARTURE` | 错峰发车，避免同时牵引加重供电压力 |
| 后车间隔过短 | `HOLD` | 扣车，恢复安全间隔 |
| 前车间隔过长且站台拥挤 | `RELEASE` | 尽快放行，提高运力 |
| 满载且滞留严重 | `ADD_TRAIN_REQUEST` | 生成加开列车请求，当前只记录不直接执行 |
| 扰动激活 | `DWELL_EXTEND` | 延长停站用于恢复或组织客流 |
| 无调整需要 | `FOLLOW_TIMETABLE` | 按计划运行 |

规则调度不是最终智能调度算法，它的价值是提供可解释基线。后续任何 GA、PSO、强化学习或运行图重排算法都必须与该基线比较，证明优化是否真正改善等待、延误、满载率和供电峰值。

### 4.4 主引擎集成

我将上述模块接入 `SimulationEngine`，形成如下运行链：

```text
每 tick 生成站台到达客流
  -> 列车到站时处理上下客
  -> 更新车载人数、负载率和停站时间
  -> 供电模块计算牵引限制
  -> 调度读取客流、负载、间隔和供电状态
  -> 生成调度决策
  -> API 返回状态，前端展示，SQLite 记录
```

### 5.4 DC 准静态潮流

列车根据位置找到左右相邻牵引所。左右供电路径电阻由牵引所内阻、馈电电缆、接触轨和回流轨组成：

```text
R_left  = Rss_left  + Rfeeder_left  + (r_contact + r_return) × distance_left
R_right = Rss_right + Rfeeder_right + (r_contact + r_return) × distance_right
```

列车牵引功率由车辆牵引力、速度、辅助功率和效率换算。求解器根据列车功率和线路电阻求列车受电电压、电流，并把负荷分配到左右牵引所。多列车场景下，列车负荷共同影响牵引所电压和负载。

当前输出包括：

- 每列车受电电压、电流、功率、供电区间和左右牵引所。
- 牵引限功率系数和再生限制系数。
- 每座牵引所电压、电流、功率、累计能量和负载率。
- 每个馈电臂电流、功率、负载率和状态。
- 线路损耗、再生生成、邻车吸收、能馈吸收和再生浪费。
- 欠压、过压、牵引所过载、馈电臂过载和设备退出告警。

供电模块输出的 `tractionLimitRatio` 已反馈车辆模型。供电电压下降时，下一步车辆牵引力会受到限制，这使供电不再只是统计模块，而是运行控制约束。

### 5.5 再生制动处理

再生制动能量按以下顺序分配：

1. 优先供同一连通网络中的牵引列车吸收。
2. 剩余部分由配置了能馈能力的牵引所吸收。
3. 超出吸收能力的部分记为 `wastedRegenKw`。
4. 电压达到再生软/硬限制时降低或切断再生能力。

因此，单车场景中 `REGEN WASTE=0` 并不一定异常：列车可能尚未制动、制动功率被辅助负荷或能馈装置吸收，或者当前模型没有产生剩余再生。后续需要用两列车“一个牵引、一个制动”和无吸收能力场景验证再生分配。

## 14. 可能追问与回答

### 14.1 10 座牵引所的位置真实吗？

公开资料支持 9 号线早期工程设置 10 座地下牵引变电所，郭公庄有较明确的公开信息；其余精确位置和分段缺少施工图，因此当前按站点里程和供电半径近似布点。数量和建模方法有依据，具体位置属于 V0 工程估计，所以系统明确标记 `ENGINEERING_ESTIMATE`。

### 14.2 潮流模型与真正电力系统软件有什么区别？

当前模型计算 DC 牵引侧的准静态电压、电流、负荷分配和损耗，不计算 AC 中压环网、电磁暂态、保护继电器和施工图级开关逻辑。它适合秒级运行仿真和优化，但不能替代专业电力系统设计软件。

### 14.3 为什么列车供电要双边计算？

列车通常位于相邻两个牵引所之间，左右供电路径的电阻随列车位置变化。距离近、路径电阻小的一侧承担更多电流。双边模型才能反映列车移动导致的电压变化、牵引所负载转移和故障越区供电。

### 14.4 为什么有时再生浪费是 0？

可能是列车没有处于有效再生制动，也可能是再生功率被同网牵引列车、辅助负荷或能馈装置吸收。只有产生的再生功率超过当前吸收能力时，浪费项才大于 0。后续会用无吸收能力和多车牵引/制动交叠场景专门验证。

### 14.5 调度模块现在算智能调度吗？

当前是可解释规则基线，不是最终 AI 调度。它先把客流、负载、间隔、供电和扰动统一成调度上下文，并提供安全可复现的比较对象。下一阶段的优化算法必须在相同场景下优于该规则基线。

### 14.6 目前是否已经用了 AI？

当前已经有成员 B 实现的 DCDP 速度曲线优化和 PID 跟踪，我完成了 AI 优化问题、目标函数和算法路线设计，但通用 GA/PSO/贝叶斯优化实验尚未落地。准确说法是优化基线和数据闭环已经具备，AI 批量寻优是下一阶段工作。

### 14.7 你的工作与成员 B 有什么接口？

成员 B 的车辆模型输出速度、牵引力和制动力，我的供电模型将其转换为电气负荷，计算受电电压和牵引限制，再反馈车辆模型。我的客流模块还会输出乘客质量，下一阶段将用于修改车辆总质量和能耗。

### 14.8 为什么现在不使用云数据库？

当前是单机固定步长仿真，本地 SQLite 延迟低、部署简单、便于每次运行独立归档，已经覆盖客流、调度和供电记录。云数据库不会提升仿真精度，等到多人共享实验结果或 Web 部署时再迁移 PostgreSQL 更合适。
