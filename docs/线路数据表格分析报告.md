# 线路数据表格分析报告

分析对象：`docs/线路数据(1).xls`

分析日期：2026-07-07

## 1. 总体判断

这份表不是普通业务统计表，而是一套较完整的轨道交通电子地图/信号基础数据配置表。它描述了线路拓扑、轨道区段、计轴区段、逻辑区段、道岔、信号机、站台、应答器、坡度、静态限速、进路、防护区段、接近区段、触发区段、防淹门、隧道、车档、SPKS 开关、车库门、碰撞区域等对象。

从项目角度看，这份表可以作为以下模块的核心输入：

1. 线路与轨道仿真模块
2. 信号与列控仿真模块
3. 车站与站台仿真模块
4. 调度与运行图仿真模块的一部分基础设施数据
5. 环境与扰动仿真模块的设备/区段故障注入基础
6. 数据记录、评估与可视化模块的线路底图和对象索引

它目前不直接提供车辆动力学参数、牵引供电参数、运行图时刻表和客流数据，但能为这些模块提供空间位置、限速、坡度、站点、信号和进路约束。

## 2. 表格结构规律

每个工作表基本遵循同一结构：

| 行号 | 含义 |
|---|---|
| 第 1 行 | 元信息字段：数据类型、表单编号、行数、列数 |
| 第 2 行 | 表单编号、声明行数、声明列数 |
| 第 3 行 | 字段数据类型，例如 UDE2、UDE4、STR16 |
| 第 4 行 | 字段中文名 |
| 第 5 行起 | 真实数据 |

特殊值：

```text
65535 通常表示无效/空引用/不存在
0x55、0xaa 等常用于方向或属性编码
厘米 cm 是主要长度单位
厘米/秒 cm/s 是部分限速单位
```

需要注意：很多工作表的 Excel UsedRange 被格式撑大到 256 列或数百行，但真正有效的数据范围应以第 2 行声明的“行数、列数”为准。

## 11. 建议的数据处理方式

建议先写一个“线路数据解析器”，不要直接在业务代码里读 Excel。

推荐转换流程：

```text
Excel 原表
  ↓
解析第 1-4 行元信息和字段名
  ↓
按声明行数/列数读取真实数据
  ↓
将 65535 转为空引用
  ↓
统一单位：cm → m，cm/s → m/s 或 km/h
  ↓
生成中间 JSON/SQLite
  ↓
供仿真模块读取
```

建议中间数据结构：

```text
Point
Segment
AxleSection
PhysicalSection
LogicalSection
Switch
Signal
Station
Platform
Balise
Gradient
SpeedLimit
Route
ProtectionSection
ApproachSection
TriggerSection
DeviceMapping
```

## 13. 结论

这份 `线路数据(1).xls` 是项目中非常关键的基础数据。它不仅能支持“线路与轨道仿真”，还能支撑“信号与列控”“车站站台”“环境扰动”“可视化线路图”和“调度/进路仿真”的一大部分功能。

从数据质量看，核心对象之间的引用关系基本完整，未发现明显断链问题。最值得立即做的下一步，是把它解析为程序可用的结构化数据，并建立一个最小电子地图：

```text
Seg 拓扑 + 道岔 + 站台 + 信号机 + 限速 + 坡度
```

这样我们的仿真系统就可以从“手写玩具线路”升级为“基于老师真实线路数据的仿真平台”。
