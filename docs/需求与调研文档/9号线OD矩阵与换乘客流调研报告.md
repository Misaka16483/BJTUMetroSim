# 匿名场景 A 的 OD 矩阵与换乘客流调研报告

> 调研日期：2026-07-10
> 调研范围：AFC站间OD、换乘客流、列车匹配、视频客流检测、多源数据融合及仿真标定
> 项目边界：以经过变换的匿名线路场景为仿真主线，其他线路先作为边界客流源/汇，不要求立即仿真完整城市线网。

## 1. 结论先行

当前客流模块已经能根据“车站、方向、时段、到达率”累积候车人数，并根据固定下车比例更新列车载客量，但它还不是OD客流模型。最关键的缺失是：乘客没有目的地、没有乘车路径、没有换乘来源和去向，也没有按目的地组织的乘客批次。因此，系统无法守恒地计算某列车在北京西站、军事博物馆等换乘站应有多少人下车、换乘和继续乘坐。

本项目应将客流模块升级为以下三层：

1. **需求层**：分时段OD矩阵，回答“何时有多少人从哪里去哪里”。
2. **分配层**：路径、列车和换乘分配，回答“乘客走哪条线路、乘哪一班车、在哪里换乘、是否滞留”。
3. **观测与标定层**：AFC、列车运行日志、视频区域计数和人工抽样共同约束仿真结果。

推荐实施顺序是：

1. 先实现可解释的OD基线、乘客批次、流量守恒和换乘边界节点。
2. 再接入历史AFC和列车运行日志，重建历史OD与乘客行程。
3. 最后处理实时AFC“乘客尚未出站导致目的地未知”的问题，并用视频计数校正站内状态。
4. AI方法用于实时目的地补全、短时OD预测、路径选择学习和视频计数，不应取代第一阶段的守恒基线。

## 2. 现有系统缺口核查

### 2.1 当前代码实际能力

| 位置 | 当前能力 | 对OD/换乘仿真的限制 |
|---|---|---|
| `app/domain/station/services.py` | `PassengerDemandProfile`只有车站、方向、时段、到达率和固定下车比例 | 没有目的地和换乘去向 |
| `PassengerFlowGenerator.arrivals()` | 按到达率和小数余量确定性生成人数 | 不是文档中描述的非齐次泊松过程，也没有OD采样 |
| `PassengerFlowGenerator.alighting_ratio()` | 每站按固定比例下车 | 下车人数不由乘客真实目的地决定，无法保证OD守恒 |
| `StationService.process_train_stop()` | 根据容量、车门能力和固定下车比例计算上下车 | 无法区分出站、换乘、继续乘车和滞留乘客 |
| `app/core/engine.py::_build_station_service()` | 目前主要配置单方向、单站到达率 | 不能形成双方向完整OD需求 |
| `POST /api/passenger/profiles`设计 | 文档示例已有`odDistribution` | 当前实现尚未形成可执行的OD导入、校验和仿真闭环 |
| `station_passenger_records` | 记录到达、上下车、候车、滞留 | 没有OD快照、换乘流、乘客批次和估计置信度 |

### 2.2 必须纠正的概念

**AFC站间OD不等于换乘路径。** 北京地铁封闭式AFC通常能够从进站、出站交易得到线网起点和终点，但付费区内换乘一般不再次刷卡。因此，AFC可以直接形成“线网进站站点到线网出站站点”的OD，却不能单独证明乘客经过哪座换乘站、选择了哪条路径或哪班列车。

**实时OD不等于历史OD。** 历史交易在乘客出站后可以配对；实时系统中，大量已经进站但尚未出站的乘客没有目的地记录，当前时间片OD天然不完整。实时OD必须估计，而不是简单聚合。

**视频不能直接恢复完整OD。** 视频适合提供区域人数、通道方向流量、站台候车、上下车和滞留人数等观测量。除非跨摄像头持续识别个人，而这会显著增加技术难度和隐私风险，否则视频应作为OD状态估计的约束，不应声称它能独立生成全程OD。

## 3. 匿名场景 A 建模边界

### 3.1 不能只建立13×13封闭矩阵

9号线有13个本线车站，但存在8个换乘站。根据项目已有北京地铁路网数据，各换乘关系如下：

| 9号线车站 | 外部线路 | 建议边界节点 |
|---|---|---|
| 郭公庄 | 房山线 | `EXT_FS_GGZ` |
| 丰台南路 | 16号线 | `EXT_L16_FTNL` |
| 七里庄 | 14号线 | `EXT_L14_QLZ` |
| 六里桥 | 10号线 | `EXT_L10_LLQ` |
| 北京西站 | 7号线、国铁及地面接驳 | `EXT_L7_BJXZ`、`EXT_RAIL_BJXZ` |
| 军事博物馆 | 1号线 | `EXT_L1_JSBWG` |
| 白石桥南 | 6号线 | `EXT_L6_BSQN` |
| 国家图书馆 | 4号线、16号线 | `EXT_L4_GJTSG`、`EXT_L16_GJTSG` |

因此建议同时维护三种流量：

1. `line9_internal_od`：9号线13个站之间、全程仅需9号线完成的站间OD。
2. `gateway_od`：外部线路/国铁边界节点与9号线站点之间的需求。
3. `physical_station_flow`：进出闸、换乘通道、站厅、站台和车门处的实际人数观测。

如果后续建模房山线与9号线跨线运行，还必须把“无需下车的跨线直通客流”和“郭公庄下车换乘客流”分开，不能都记作换乘。

### 3.2 两个重点站的特殊性

**北京西站**至少包含四类流：9号线本线上下车、7号线与9号线互换、国铁到达旅客进入地铁、地铁乘客前往国铁。北京市应急管理局公开材料显示，北京西站站区所辖6站日均客运量约40万人次，占9号线总客运量70%以上，北京西站最高单日进出站量达31.8万人次。这说明该站不能只使用一条平滑到达率曲线，还要引入国铁列车到达造成的脉冲客流。[来源](https://yjglj.beijing.gov.cn/art/2022/12/20/art_9408_638032.html)

**军事博物馆**包含1号线与9号线双向换乘，且换乘客流与进出站客流在通道和站厅汇聚。它适合用于验证“方向性换乘流、走行时间、站台注入和列车载客变化”是否守恒，而不只是验证总人数。[站内流线背景资料](https://www.chinanews.com.cn/auto/2013/12-22/5648055.shtml)

## 4. 需要完成的研究问题

### R1：AFC数据语义与质量

需要查清老师/实验室能提供的是：原始闸机交易、已经配对的行程表、5分钟OD聚合表，还是只有站级进出量。不同数据能够支持的模型层级完全不同。

重点研究：

- 进站、出站、更新、退款、无效交易等交易类型。
- 单程票、交通卡、二维码等票种是否能够统一配对。
- 同一物理换乘站的线路站码、付费区站码和闸机站码如何映射。
- 闸机时钟与ATS/列车日志时钟是否同步。
- 重复记录、漏刷、超时行程、跨日行程和异常长旅行如何清洗。
- 数据上传延迟分布，而不只是平均延迟。

### R2：历史动态OD构建

研究如何把配对后的AFC行程按5分钟或15分钟时间片聚合为：

\[
Q_{ij}^{t}=\text{在时间片 }t\text{ 从站 }i\text{ 进站、最终从站 }j\text{ 出站的人数}
\]

必须满足：

\[
O_i^t=\sum_j Q_{ij}^t, \qquad D_j^t=\sum_i Q_{ij}^t
\]

其中`O`和`D`分别是进站与出站边际流量。应研究时间片按“进站时刻”还是“出站时刻”归属；本项目需求生成应以进站时刻为主，同时保留出站时间片用于实时估计和校验。

### R3：路径、列车和换乘分配

同一OD可能有多条可行路径。需要结合线网拓扑、时刻表、列车到发、换乘走行时间、拥挤和滞留概率生成候选行程，并估计路径选择比例。

建议研究顺序：

1. 最短广义费用路径基线。
2. 多项Logit路径选择，费用包含车内时间、候车时间、换乘走行、换乘次数和拥挤惩罚。
3. EM、贝叶斯或最大似然标定路径选择参数。
4. 结合列车运行日志推断乘客可能搭乘的具体列车。
5. 将“未能登上第一班车”的滞留概率纳入列车匹配。

### R4：实时OD补全与短时预测

实时系统中，已进站未出站乘客的目的地未知。应明确区分：

- **当前OD补全**：估计当前尚未完成行程的目的地。
- **未来OD预测**：预测未来5、15、30分钟将出现的OD需求。

建议建立三档算法：

1. 基线：历史条件概率`P(destination | origin, time, dayType)`。
2. 可解释在线估计：状态空间模型/Kalman滤波，使用进站量、已完成OD和出站量更新状态。
3. AI深化：HIAM、CAS-CNN或图时空网络，显式输入不完整OD、未完成行程向量、日型、天气和事件。

### R5：视频客流检测

视频侧应拆成三个任务，不使用一个模型包打天下：

1. **通道方向流量**：检测+多目标跟踪+虚拟线穿越计数，输出每个方向的通过人数。
2. **站台/站厅区域人数**：密度图回归，处理高密度遮挡和透视变化。
3. **列车到发事件下的上下车/滞留**：以开门、关门和列车离站为事件窗口，计算站台人数变化与滞留。

北京西站优先布设：国铁到地铁通道、7↔9换乘通道、9号线上下行站台和关键闸机区域。军事博物馆优先布设：1→9、9→1两个方向的换乘通道以及9号线上下行站台。

### R6：多源融合与可信度

融合的核心不是把所有数据拼接进神经网络，而是明确每个观测量约束什么状态：

| 数据源 | 直接观测 | 不能直接证明 |
|---|---|---|
| AFC进站交易 | 进站站点、进站时刻、进站人数 | 付费区内换乘路径、当前所在位置 |
| AFC出站交易 | 出站站点、出站时刻、已完成OD | 尚未完成行程的目的地 |
| ATS/列车日志 | 列车到发、运行时刻、可行乘车链 | 实际上车人数 |
| 视频通道计数 | 指定通道的方向流量 | 乘客完整OD和个人身份 |
| 视频区域计数 | 区域人数、密度、滞留变化 | 精确列车行程 |
| 人工调查 | 局部换乘比例、计数真值、行为参数 | 连续全时段状态 |

推荐以状态空间/贝叶斯数据同化为第一版融合方法。设OD、站内区域人数和列车载客量为隐状态`x_t`，AFC/视频/列车日志为观测`y_t`：

\[
x_t=f(x_{t-1},u_t)+w_t, \qquad y_t=H_t x_t+v_t
\]

这样能够同时给出估计值和不确定性，并检查流量守恒。AI预测结果也应作为带误差的先验或输入，而不是绕开守恒约束。

## 5. 需要索取和采集的数据

### 5.1 AFC行程数据：首选实验室直接输出脱敏行程表

为减少隐私风险，优先请数据提供方在受控环境内完成进出站配对，只向项目提供随机`trip_id`，不提供原始卡号、手机号或支付账户。

建议字段：

| 字段 | 必需 | 说明 |
|---|---:|---|
| `trip_id` | 是 | 不可逆随机行程ID，不跨目的复用 |
| `service_date` | 是 | 运营日，处理跨午夜问题 |
| `entry_station_code` | 是 | 进站站码 |
| `entry_gate_id` | 推荐 | 可用于入口分布与站内走行标定 |
| `entry_time` | 是 | 至少秒级，保留原始时区 |
| `exit_station_code` | 是 | 出站站码 |
| `exit_gate_id` | 推荐 | 可用于出口与站内走行标定 |
| `exit_time` | 是 | 至少秒级 |
| `media_type` | 推荐 | 票种差异与数据质量分析 |
| `validity_flag` | 是 | 是否为正常完整行程 |
| `fare_product` | 可选 | 检查票制与特殊票影响 |
| `upload_time` | 实时阶段必需 | 测量数据上传延迟 |

如果只能获得原始交易，另需`transaction_type`和可配对的脱敏介质标识；脱敏标识应限定保存期限和访问范围。

### 5.2 列车运行数据

| 数据 | 最低粒度 | 用途 |
|---|---:|---|
| 列车ID、运行方向、交路 | 每列车 | 构造可行行程 |
| 各站实际到达/开门/关门/出发时刻 | 1秒 | AFC到列车匹配、停站与上下车窗口 |
| 计划时刻表 | 每列车 | 计算延误和候选列车 |
| 编组、额定容量、可用容量 | 每列车 | 载客与滞留约束 |
| 跳停、清客、折返、取消等事件 | 事件级 | 排除不可能行程 |
| 站台与方向映射 | 静态 | 连接列车与视频区域 |

### 5.3 换乘设施与步行数据

至少采集8个换乘站的线路连接关系；北京西站和军事博物馆需进一步采集：

- 站厅、站台、闸机、楼梯、扶梯和换乘通道的区域拓扑。
- 通道长度、有效宽度、扶梯方向和设计通过能力。
- 典型步行时间分布，而不是单一最短时间。
- 摄像头位置、朝向、覆盖ROI和盲区。
- 客流组织措施：单向通行、隔离栏、限流点和关闭策略。
- 北京西站国铁列车计划/实际到达时刻，至少保留到达方向、到达时刻和估计下车人数的聚合值。

### 5.4 视频输出和人工标定

生产数据管道只需要输出匿名聚合结果：

| 字段 | 说明 |
|---|---|
| `camera_id`, `zone_id` | 摄像头与物理区域 |
| `window_start`, `window_end` | 建议1到5秒窗口，平台融合可再聚合到30秒/1分钟 |
| `in_count`, `out_count` | 虚拟线双向通过人数 |
| `occupancy_count` | 区域人数 |
| `density_pax_per_m2` | 已标定有效面积后的密度 |
| `confidence` | 模型置信度或估计方差 |
| `occlusion_level` | 遮挡/不可用标记 |
| `model_version` | 保证结果可追溯 |

人工标定至少包括：

- 每个重点摄像头覆盖早高峰、平峰、晚高峰各一组视频片段。
- 每个摄像头建议标注500到1000帧人头点或边界框，用于本地场景微调和透视检查。
- 每个重点通道至少人工复核5个30分钟窗口的双向穿越人数。
- 每个重点站至少选择2个工作日进行列车级上下车与滞留人工计数。
- 标注雨伞、反光、列车进站遮挡、超密度、摄像头抖动等困难样本。

### 5.5 时间跨度

最低可用数据集：连续4周，覆盖不少于10个工作日、4个周末日，并包含AFC和列车运行日志的同日数据。

推荐数据集：连续8到12周，覆盖节假日、国铁客流波动、异常延误或大客流事件。若只能获得短周期数据，应明确模型只能用于普通工作日标定，不能宣称具备节假日泛化能力。

## 6. 数据管道设计建议

```text
AFC交易/行程 ──> 校验与脱敏 ──> 进出站配对 ──> 5分钟历史OD ─┐
                                                               │
ATS/时刻表 ─────> 时钟对齐 ──> 候选列车/路径/滞留模型 ────────┼─> OD与乘客状态估计
                                                               │
视频流 ─────────> 边缘检测 ──> 区域人数/方向流/置信度 ────────┤
                                                               │
人工计数/调查 ──> 真值与误差模型 ──────────────────────────────┘
                                  │
                                  v
                    场景OD -> 乘客批次 -> 上下车/换乘 -> 调度闭环
```

数据处理必须保留以下审计信息：原始来源、采集时间、事件时间、处理版本、质量标记、延迟、缺失原因和估计方法。任何估计值都应区分`MEASURED`、`INFERRED`、`PREDICTED`和`SYNTHETIC`。

## 7. 拿不到真实AFC时的可执行替代方案

真实AFC数据可能因运营安全、隐私和商业限制无法获得。项目仍可先完成算法和仿真闭环，但必须明确使用合成数据。

### 7.1 构造种子OD

根据站点类别、距离衰减、换乘枢纽权重和时段方向性构造种子矩阵：

\[
Q_{ij}^{0,t}=A_i^t B_j^t \exp(-\beta c_{ij}) W_{ij}^t
\]

其中`A`为起点发生强度，`B`为终点吸引强度，`c`为广义出行成本，`W`表示通勤潮汐、枢纽和事件权重。对角线和不可行方向必须为0。

### 7.2 用IPF满足边际约束

如果能获得站级进出量或视频计数，用迭代比例拟合IPF调整种子矩阵，使行和、列和匹配观测。IPF只能在既有非零结构内缩放，因此种子矩阵不能随意把可能存在的OD设为0。

### 7.3 建立换乘边界

对8个换乘站分别给出`external_in`和`external_out`分时曲线；北京西站额外叠加与国铁到达相关的脉冲需求。即使没有完整线网，也能在9号线内部守恒地模拟外线乘客注入、换出和列车负载变化。

### 7.4 合成数据的使用限制

- 可以验证软件结构、守恒、调度响应和算法对比。
- 可以做敏感性分析和扰动实验。
- 不能把合成OD的绝对客流值宣传为真实北京9号线客流。
- 所有图表必须显示`SYNTHETIC`或`CALIBRATED_SYNTHETIC`质量标签。

## 8. 仿真内核需要的数据结构

### 8.1 OD配置

```json
{
  "timeSliceStart": "07:30:00",
  "timeSliceSec": 300,
  "originNodeId": "S-BJXZ",
  "destinationNodeId": "S-JSBWG",
  "passengerCount": 126,
  "source": "AFC_HISTORICAL",
  "quality": "MEASURED"
}
```

### 8.2 乘客批次

不需要为每个乘客创建对象。建议按共同属性建立`PassengerCohort`：

```text
cohort_id
origin_node_id
destination_node_id
entry_time_window
count
assigned_path_id
current_leg_index
next_alight_station
waiting_since
source
confidence
```

列车到站时，下车人数应由`next_alight_station == current_station`的车内批次决定；换乘批次进入下一线路/边界队列，出站批次离开系统。这样才能替换固定`alighting_ratio`并保持人数守恒。

### 8.3 建议新增的数据表

| 表 | 作用 |
|---|---|
| `passenger_od_profiles` | 场景级分时OD输入 |
| `passenger_od_cells` | 每个时间片的OD单元格 |
| `transfer_gateways` | 9号线与外线/国铁边界定义 |
| `passenger_cohort_records` | 批次的等待、乘车、换乘和完成状态 |
| `transfer_flow_records` | 换乘站按来源线路、去向线路、方向记录流量 |
| `zone_flow_observations` | 视频/人工区域人数和方向流观测 |
| `od_estimation_runs` | 算法版本、参数、输入范围和评价指标 |

## 9. 分阶段实施方案

### Phase OD-0：可运行OD基线

目标：不依赖真实数据，替换固定下车比例，形成OD守恒的客流闭环。

工作项：

- 定义13个本线站和换乘边界节点。
- 支持CSV/JSON分时OD导入。
- 实现种子矩阵+IPF合成OD。
- 实现乘客批次、按目的地下车、换乘注入/流出和滞留。
- 前端显示OD热力图、断面客流、换乘方向流和数据质量标签。
- 保留当前站级到达率模式作为兼容输入，并转换为合成OD，而不是继续使用固定下车比例。

### Phase OD-1：历史AFC与行程重建

目标：从脱敏历史AFC得到可回放、可标定的5分钟OD和换乘路径比例。

工作项：

- AFC导入、清洗、配对、异常行程识别。
- 生成历史OD快照和日型模板。
- 结合时刻表生成候选路径与候选列车。
- 标定换乘走行时间、路径选择和滞留概率。
- 用北京西站、军事博物馆视频/人工计数校验换乘流。

### Phase OD-2：实时融合

目标：在数据到达后60秒内更新当前客流状态，并给出置信度。

工作项：

- 接入AFC增量、ATS/列车事件和视频匿名计数流。
- 实现历史条件概率和状态空间/Kalman基线。
- 对未完成行程进行目的地概率分配。
- 输出当前OD、站台人数、列车载客量、换乘流和滞留概率。
- 处理数据延迟、断流、摄像头遮挡和时钟漂移。

### Phase OD-3：AI深化

目标：在严格的时间切分测试中显著优于OD-2基线，并服务调度优化。

候选方法：

- HIAM：融合不完整OD、未完成行程和反向DO信息。
- CAS-CNN或图时空网络：预测未来5到30分钟OD。
- 贝叶斯/EM路径选择：学习换乘时间、拥挤和换乘惩罚。
- 视频密度估计和多目标跟踪：提高高密度站台与通道计数质量。
- 守恒约束损失或投影层：确保预测OD与进出站边际、区间载客和视频观测相容。

## 10. 标定与验收指标

### 10.1 数据质量

- 正常交易配对成功率。
- 重复、缺失、超时和不可能行程比例。
- AFC、ATS和视频时钟偏差的P50/P95。
- 数据上传延迟的P50/P95/P99。
- 视频可用率、遮挡率和低置信度时长。

### 10.2 OD与换乘估计

建议同时报告MAE、RMSE、WMAPE、CPC和Jensen-Shannon散度，不只报告单一准确率。

建议项目验收目标：

| 指标 | OD-0 | OD-1/OD-2目标 |
|---|---:|---:|
| 人数守恒误差 | 小于0.1% | 小于0.1% |
| OD行/列边际误差 | 小于1% | 小于2%或不超过观测误差 |
| 重点站15分钟换乘流WMAPE | 场景自洽 | 不高于15% |
| 重点站全日换乘流误差 | 场景自洽 | 不高于10% |
| 视频普通密度计数MAPE | 不要求 | 不高于10% |
| 视频高密度计数MAPE | 不要求 | 不高于15% |
| 实时状态更新延迟 | 不要求 | 数据到达后60秒以内 |
| 未来15分钟OD预测WMAPE | 建立历史概率基线 | AI需相对基线改善至少10% |

阈值是项目工程目标，不是对现有数据质量的事实判断。获得真实数据后应根据人工计数误差和运营方要求重新定稿。

### 10.3 验证方式

- 按日期划分训练、验证、测试，不能随机打散同一天的时间片。
- 单独报告工作日、周末、节假日和扰动日。
- 北京西站、军事博物馆作为重点站留出测试。
- 路径与列车分配使用列车载客、视频换乘流和人工滞留计数交叉验证。
- 每个AI模型必须与历史条件概率、IPF、状态空间模型等简单基线比较。

## 11. 建议的验证场景

1. **北京西站国铁到达脉冲**：一班大客流国铁到达后，`EXT_RAIL_BJXZ -> 9号线`需求在10到20分钟内上升，检查站厅、站台、列车负载和调度响应。
2. **北京西站7↔9换乘潮汐**：分别模拟早高峰7→9和晚高峰9→7，检查方向性换乘流和站台注入守恒。
3. **军事博物馆1→9换乘**：检查换乘走行延迟、9号线站台累积、首班未登车概率和后续列车负载。
4. **军事博物馆9→1换出**：检查9号线列车下车批次是否正确离开9号线仿真边界。
5. **视频断流**：遮蔽一个摄像头10分钟，系统应降低置信度并依靠AFC/历史先验继续估计，而不是输出0人。
6. **AFC上传延迟**：人为增加5分钟上传延迟，比较历史概率、状态空间和AI补全方法。
