## 1. 文档目的

本文档在《轨道交通综合仿真系统需求说明书》1.0、现有软件实现、2026年7月10日答辩点评以及老师补充线路与车辆资料的基础上，重新确定系统第二阶段的产品定位、功能边界、模型要求、跨模块闭环、验收场景和实施次序。

本版本重点解决以下问题：

1. 将项目从“状态监控与页面展示”重新聚焦为“场景驱动的仿真与实验系统”。
2. 将单车演示升级为所有页面和后端服务一致支持的多列车仿真。
3. 将供电、驾驶与动力学、联锁与调度、客流与热力图四个模块串成可验证的完整闭环。
4. 放弃缺乏可靠数据支撑的乘客OD和个体级人群仿真，改用站台方向分时段的宏观客流模型。
5. 使用老师补充的上下行Link公里标、车辆参数和线路数据说明，替换散落在代码中的临时常量。
6. 明确定义停车、速度跟踪、能量平衡、人数守恒等指标的参照对象和计算方法。

本文档是后续软件设计、数据库设计、API设计、测试设计和五人开发分工的上位依据。若后续设计与本文档冲突，应先修改并重新审核本需求基线。

## 2. 项目定位

### 2.1 产品定义

本项目拟开发一个基于教学线路数据的多列车轨道交通闭环仿真系统。系统通过线路参数、列车参数、运行计划、客流强度、驾驶操作、通信时延和设备故障等输入驱动模型演算，研究列车运行、联锁调度、牵引供电和客流负载之间的耦合关系，并输出可回放、可比较、可评价的实验结果。

系统的核心不是“看见列车在移动”，而是回答以下类型的问题：

1. 不同牵引或制动级位如何影响速度、停车、能耗和供电负荷？
2. 多列车同时牵引或制动时，供电电压、峰值功率和再生能量如何变化？
3. 进路冲突、区段占用和前车未出清时，联锁与调度如何保障安全并调整运行？
4. 站台客流增加后，候车人数、列车质量、停站时间、运行间隔和供电能耗如何变化？
5. 牵引所退出、临时限速、通信时延等扰动如何沿“供电-车辆-联锁-调度-客流”链路传播？

### 2.2 数据真实性声明

老师提供的线路数据以北京地铁9号线为原型，但设备位置、信号位置、区段长度和部分参数经过修改。本项目应将其称为“9号线衍生教学线路”或“教学线路9”，不得宣称为北京地铁9号线真实工程数字孪生。

系统输出用于课程研究、算法验证和软件工程实践，不得用于真实行车指挥、设备控制、安全认证或工程设计。

### 2.3 仿真系统与监控系统的边界

系统可以提供线路图、联锁图、司机台和供电面板，但可视化必须服务于以下至少一种目的：

1. 设置仿真输入或操作条件。
2. 观察模型内部状态和跨模块影响。
3. 对比基线与实验场景。
4. 解释告警、干预和失败原因。
5. 查看可量化指标或回放仿真过程。

仅显示静态状态、装饰性动画或不能影响仿真结果的控制，不作为核心功能验收成果。

### 3.2 Link、Seg与公里标规则

1. 公里标是线路上的唯一位置坐标。
2. Link由计轴器、道岔等线路设备位置划分，Link长度等于相邻设备公里标之差。
3. 本教学项目的信号显示Seg编号按`SegId = LinkId + 300`映射；若原始表中存在例外，应以显式映射表为准，禁止在业务代码中到处直接加300。
4. 线路长度数据以厘米记录，导入后统一转换为米作为仿真内部单位，同时保留原始厘米值和数据来源。
5. 常规运营仿真只使用主线路上行和下行连接次序；车辆段和支线数据仅在明确启用车辆段详细场景时加载。

### 3.3 主线Link基线

| 方向 | Link数量 | 起点公里标 | 终点公里标 | 连续长度 | 连续性 |
|---|---:|---:|---:|---:|---|
| 上行 | 83 | 3000 cm | 1660041 cm | 1657041 cm，即16570.41 m | 相邻Link首尾连续 |
| 下行 | 73 | 3000 cm | 1663762 cm | 1660762 cm，即16607.62 m | 相邻Link首尾连续 |

系统不得假设上下行拥有相同Link编号、相同Link数量或完全相同长度。列车位置、区段占用、进路和供电定位必须携带方向信息。

### 3.4 坡度数据规则

坡度数据按以下优先级获取：

1. 从老师之前提供的《线路数据》中提取可确认的坡度区段。
2. 根据公里标构造教学用坡度配置。
3. 若使用工程假设，任一区段坡度绝对值不得超过15‰。

每条坡度记录必须包含起点、终点、沿公里标递增方向的坡度千分数、来源和质量标记。下行列车反向通过同一区段时，动力学计算使用相反符号。系统不得只记录坡度绝对值而丢失方向。

### 3.6 数据来源与质量标记

所有关键参数应使用以下来源标记：

| 标记 | 含义 |
|---|---|
| `TEACHER_PROVIDED` | 老师文件直接提供 |
| `DERIVED` | 可由老师数据确定性计算得到 |
| `ASSUMED` | 因缺失数据采用的教学工程假设 |
| `CALIBRATED` | 通过参考系统、试验结果或优化标定得到 |
| `RUNTIME_INPUT` | 用户在场景中输入 |

关键结果应继承输入质量信息。含有`ASSUMED`参数的结果必须显示“工程估算”，不得显示为“实测”或“真实值”。

## 5. 系统范围

### 5.1 本版本范围内

1. 9号线衍生教学线路主线上下行分别建模。
2. 统一多列车仿真内核，首个完整验收规模为3至5列车。
3. DC750V准静态牵引供电、多车负荷与再生能量模型。
4. 多级位驾驶、一维纵向动力学、ATO和ATP监督。
5. 固定闭塞/计轴区段占用、计算机联锁和基础ATS调度。
6. 抽象车库发车、上线、运营、折返、收车生命周期。
7. 分站台、分方向、分时段宏观客流和热力图。
8. 客流载荷对车辆质量、动力学、停站和供电的闭环影响。
9. 场景、故障、通信时延、记录、指标、回放和对比。
10. 为实体司机台和老师实验室平台保留适配接口。

### 5.2 本版本明确不做

1. 真实乘客个体、行走路径和OD矩阵仿真。
2. 全北京地铁网络或多线路换乘网络仿真。
3. 完整车辆段轨道、调车进路和车辆段联锁。
4. 多质点、轮轨黏着、蛇行等高阶车辆动力学。
5. 工程认证级CBTC、ATP或安全完整性认证。
6. 与真实北京地铁9号线结果的一比一精度承诺。
7. 云数据库作为运行前提；本阶段允许使用本地SQLite和文件配置。
8. 与核心仿真无关的纯装饰性监控功能。

## 6. 总体架构与完整闭环

### 6.1 四个业务模块

| 编号 | 模块 | 核心职责 |
|---|---|---|
| M1 | 供电仿真 | 多车牵引/再生负荷、DC潮流、故障重构、限牵和能量统计 |
| M2 | 驾驶与一维动力学仿真 | 多级位控制、ATO/ATP、纵向运动、停车和载荷响应 |
| M3 | 联锁与调度仿真 | 多车任务、进路安全、运行图、车库发收车、折返和扰动处置 |
| M4 | 客流与热力图仿真 | 分时段站台客流、上下车、拥挤、停站和列车载荷 |

### 6.2 公共仿真底座

以下能力是四模块共享基础，不作为第五个业务模块：

1. 统一仿真时钟、倍率、暂停、单步、复位和随机种子。
2. 场景加载、参数校验、事件调度和故障注入。
3. 统一对象ID、单位、时间戳和状态契约。
4. 消息总线、API、WebSocket或等价实时推送。
5. 数据记录、指标计算、回放、导出和对比。
6. 数据来源、质量标记和假设登记。

### 6.3 闭环链路

```text
分时段站台客流
  -> 候车、上下车、滞留和列车载客量
  -> 列车总质量与停站时间
  -> ATS发车计划和调度命令
  -> CI进路检查、道岔锁闭和信号开放
  -> 手动级位或ATO控制命令
  -> 多车牵引/再生功率请求
  -> 供电潮流、列车电压和限牵系数
  -> 一维车辆动力学和区段占用
  -> 实际到发时间、间隔和后续客流积聚
```

任何模块不得绕过安全或物理边界直接修改另一模块的最终状态。例如，调度模块只能提出发车或进路请求，不能直接开放信号；客流模块只能修改载荷和停站条件，不能直接修改列车速度。

### 6.4 单仿真步推荐顺序

1. 时钟推进并激活本步场景事件。
2. 更新站台客流到达和既有停站作业。
3. ATS生成任务、调度命令或进路申请。
4. CI检查区段、道岔和冲突进路，输出信号与运行许可。
5. 驾驶/ATO根据可见状态和通信时延生成控制请求。
6. 汇总全部列车的牵引、辅助和再生功率请求。
7. 供电模块求解潮流并输出每车电压和限牵系数。
8. 动力学模块应用限牵后的实际力，更新全部列车车头和车尾位置。
9. 更新区段占用、进路释放、到站、折返和车库状态。
10. 计算调度、客流、车辆和供电指标并生成原子快照。

若实现采用上一步供电限制或预测-校正方式，必须在设计中明确，不得形成未说明的代数环或一拍延迟。

## 8. 模块一：供电仿真需求

### 8.1 模块目标

供电模块应回答“列车在哪里、以何种级位运行、同时有多少列车牵引或制动，会对供电网络产生什么影响”。模块采用可解释的DC750V准静态模型，不以黑盒常量直接生成监控数据。

### 8.2 功能需求

| ID | 优先级 | 需求 |
|---|---|---|
| REQ-PWR2-001 | Must | 系统应以可配置文件维护牵引所、馈电臂、接触轨分段、回流轨、联络开关和供电分区。 |
| REQ-PWR2-002 | Must | 每个供电设备应具有公里标、方向、状态、额定参数、来源和质量标记。 |
| REQ-PWR2-003 | Must | 系统应根据列车车头、车尾和方向确定其受电所在供电分区。 |
| REQ-PWR2-004 | Must | 系统应同时接收全部列车的牵引、辅助和再生功率请求，禁止逐车独立求解后简单相加电压。 |
| REQ-PWR2-005 | Must | 牵引机械功率应由实际牵引力和速度计算，并考虑传动与逆变效率。 |
| REQ-PWR2-006 | Must | 再生功率应由实际电制动力和速度计算，并与机械制动功率区分。 |
| REQ-PWR2-007 | Must | 系统应求解每列车电压、电流、功率和限牵系数。 |
| REQ-PWR2-008 | Must | 系统应求解每座牵引所和每条馈电臂的有符号电流与功率；正值表示供电，负值表示吸收或反馈。 |
| REQ-PWR2-009 | Must | 系统应计算线路损耗、牵引能量、再生生成、邻车吸收、可逆变电所反馈和制动电阻浪费。 |
| REQ-PWR2-010 | Must | 当一车再生、另一车牵引时，应优先计算网络内部的再生能量利用，再计算反馈或浪费。 |
| REQ-PWR2-011 | Must | 可逆变电所吸收的再生功率必须分配到具体设备和馈电路径，不能只保留全线汇总值。 |
| REQ-PWR2-012 | Must | 系统应支持牵引所退出、馈电臂断开、联络开关分合和正常恢复。 |
| REQ-PWR2-013 | Must | N-1场景应重新计算供电路径、设备负载、列车电压和限牵，而不是只修改状态颜色。 |
| REQ-PWR2-014 | Must | 欠压、过流或容量不足时，供电模块应输出`tractionLimitRatio`，动力学模块必须实际应用该限制。 |
| REQ-PWR2-015 | Must | 供电复位不得清空仿真时间或产生并发状态竞争。 |
| REQ-PWR2-016 | Should | 系统应支持牵引所整流单元数量、内阻、空载电压和反馈容量配置。 |
| REQ-PWR2-017 | Should | 系统应支持供电容量、牵引所位置和线路电阻的敏感性实验。 |
| REQ-PWR2-018 | Should | 系统应支持削峰、限牵和再生利用策略的批量对比。 |

### 8.3 符号与能量平衡

1. 列车功率`P_train > 0`表示从直流网取电，`P_train < 0`表示向直流网再生。
2. 牵引所功率`P_sub > 0`表示向直流网供电，`P_sub < 0`表示从直流网吸收并反馈交流侧。
3. 馈电臂电流和功率应采用与公里标方向一致的有符号约定。
4. 每步应计算能量平衡残差：

```text
residualKw = sum(P_substationKw)
             - sum(P_trainKw)
             - lineLossKw
             - wastedRegenKw
```

稳态正常场景的相对平衡残差目标不超过1%，扰动切换步目标不超过3%；超过阈值必须记录告警和求解上下文。

### 8.4 供电可视化需求

1. 显示实际供电拓扑、列车位置和有方向的潮流。
2. 正功率、负功率、再生吸收和浪费应明确区分。
3. 曲线必须显示单位、时间轴和数值量程，禁止各序列独立归一化后造成误导。
4. 故障、恢复和开关操作应在曲线上形成唯一事件标记。
5. 表格应显示实际计算值，不能在存在再生反馈时将所有牵引所和馈电臂均显示为0。

## 9. 模块二：驾驶与一维动力学仿真需求

### 9.1 模块目标

本模块将司机台视为仿真的输入外设，将牵引/制动级位转换为可解释的力和运动结果。手动驾驶、ATO和ATP必须共用同一车辆动力学模型。

### 9.2 驾驶级位需求

默认级位为：


| 级位 | 曲线能力系数 | 来源 |
|---|---:|---|
| P1/P2/P3/P4 | 0.25/0.50/0.75/1.00 | `ASSUMED`，可配置 |
| COAST | 0 | 控制定义 |
| B1至B7 | 分别为1/7至7/7 | `ASSUMED`，可配置 |
| EB | 独立紧急制动力参数 | 不等同于B7 |

级位系数用于缩放当前速度对应的牵引或常用制动能力曲线，不得用固定加速度直接替代车辆特性。接入实体司机台或获得标定数据后，应以`CALIBRATED`映射覆盖默认值并保留版本。

### 13.3 基线与实验对比

每个扰动场景应关联一个无扰动基线，并输出绝对值和相对变化。仅展示扰动后的曲线而没有基线，不视为完整实验。

## 16. 模型验证、误差与可信度要求

### 16.1 总体原则

任何“误差”都必须明确谁与谁比较、单位、采样范围和场景。没有参照对象的“误差很小”不得写入成果结论。

### 16.2 车辆与控制指标

| 指标 | 定义 | 初始验收目标 |
|---|---|---|
| 停车误差 | 最终停车参考点位置减目标停车点位置 | 正常基线绝对值不超过1.0 m |
| 速度跟踪RMSE | 实际速度与目标速度在运行区间的均方根误差 | 记录并与基线策略比较 |
| 运行时间误差 | 实际区间运行时间减计划运行时间 | 场景配置容差内 |
| 最大加速度/减速度 | 仿真全程绝对峰值 | 不超过车辆配置范围 |
| 最大加加速度 | 相邻加速度变化率峰值 | 用于比较舒适性，不虚构真实合格值 |
| 步长收敛误差 | `dt`与`dt/2`运行结果差异 | 停车位置差目标不超过0.2 m |

停车误差是相对于仿真目标点的控制误差，不代表相对于真实列车试验的模型误差。若未来获得老师平台或实测曲线，应新增“参考系统误差”指标并单独报告。

### 16.3 供电指标

| 指标 | 定义 | 初始验收目标 |
|---|---|---|
| 能量平衡残差 | 按8.3节有符号功率平衡计算 | 稳态不超过1%，切换步不超过3% |
| 电压边界 | 所有列车和设备电压在配置求解范围内 | 不得出现非有限值或无解释负电压 |
| 功率来源一致性 | 机械功率、效率和电功率可追溯 | 每个结果可回溯到列车请求 |
| 再生去向闭合 | 生成=邻车吸收+反馈+浪费+相关损耗 | 每步可核对 |

### 16.4 联锁安全不变量

1. 同一区段不得被两列不相容列车同时占用。
2. 冲突进路不得同时锁闭。
3. 锁闭道岔不得转换。
4. 信号不得在前方进路未建立时开放。
5. 列车车尾未出清时区段不得释放。
6. 调度命令不得绕过CI或ATP。

上述任一不变量违反均判定该运行失败，不以“事故次数很少”作为通过条件。

### 16.5 客流人数守恒

每次停站应满足：

```text
platformBefore + arrivals - boarded = platformAfter
onboardBefore - alighted + boarded = onboardAfter
```

人数不得为负，不得超过配置容量且不得凭空消失。因清客、限流等事件移出的乘客必须记录去向类型。

### 16.6 与参考系统对比

若实验室平台能够返回同输入条件下的车辆或其他模块结果，应保存输入快照并比较速度、位置、到站时间和关键事件。供电模块当前没有老师协议中的独立仿真接口，因此不得声称完成供电黑盒对标；可先使用解析算例、能量守恒和敏感性趋势验证。

## 17. 非功能需求

### 17.1 性能

1. 5列车、250 ms仿真tick的基线场景应支持1倍实时运行。
2. 在项目组统一测试电脑上，单tick计算时间P95目标不超过125 ms。
3. 前端状态刷新频率目标为4至5 Hz，绘图不得阻塞仿真线程。
4. 无界面批量运行应能够高于1倍速度执行。

### 17.2 可靠性

1. 任一模块输入非法时应拒绝场景启动并指出字段。
2. 单个前端断开不得停止后端仿真。
3. 故障注入、恢复和开关操作应具备线程安全和幂等保护。
4. 求解失败时应保留最后有效状态、失败输入和错误原因，不得继续发布伪正常数据。

### 17.3 可测试性

1. 每条Must需求至少对应单元测试、集成测试或场景验收中的一种。
2. 四模块应支持独立测试桩，也应有完整闭环测试。
3. 所有随机客流测试固定随机种子。
4. 手工联调脚本不得被自动测试发现器误当成单元测试。

### 17.4 可维护性

1. 参数、场景和拓扑不得硬编码在UI组件或主循环中。
2. 模块通过稳定契约交换数据，不直接访问其他模块私有字段。
3. 线路、车辆、客流和供电参数应支持版本化配置。
4. 文档、API、数据库和实现字段发生变化时应同步更新映射表。

### 17.5 安全与合规声明

1. 系统界面应明确显示“教学仿真，非真实行车控制”。
2. 实体司机台仅作为输入外设；安全制动仍由仿真ATP优先处理。
3. 系统不存储真实乘客个体信息，不进行人脸或个人轨迹采集。

## 18. 2.0分阶段实施计划

### 18.1 Stage 0：数据与公共契约重构

目标：建立可支撑四模块的统一数据底座。

必须完成：

1. 将Strict OOXML的Link文件规范化导入JSON/数据库。
2. 分别建立上行83个、下行73个主线Link顺序。
3. 导入车辆编组、质量、长度、轮径和52点曲线。
4. 建立坡度配置和来源标记。
5. 统一多车`TrainState`、车头/车尾位置和生命周期。
6. 定稿核心跨模块契约、单位和tick顺序。

退出条件：数据校验通过，能够加载5列静止列车并生成一致快照。

### 18.2 Stage 1：多级位动力学与宏观客流

目标：先建立“客流-质量-驾驶-动力学”闭环。

必须完成：

1. P1-P4、B1-B7、COAST和EB。
2. 老师车辆参数驱动的一维动力学。
3. 分站台分时段客流配置和四类预设。
4. 上下客、滞留、停站时间和质量更新。
5. 3列车基础任务与多车统一展示。

退出条件：高客流相对正常客流能稳定导致载荷、停站和动力学结果变化，且人数守恒。

### 18.3 Stage 2：联锁、调度与运营生命周期

目标：完成多列车安全运行和车库发收车闭环。

必须完成：

1. 区段占用、进路、道岔、信号和逐段释放。
2. ATS运行计划、发车间隔和扣车。
3. 抽象车库发车、终点折返和收车。
4. 固定闭塞运行许可和ATP监督。
5. 3至5列车连续运营场景。

退出条件：`SCN-DEPOT-CYCLE`和`SCN-ROUTE-CONFLICT`通过，联锁安全不变量零违反。

### 18.4 Stage 3：多车供电与完整闭环

目标：让车辆控制和多车运行真实驱动供电，而供电限制反向影响车辆与调度。

必须完成：

1. 多车统一DC潮流求解。
2. 有符号牵引所/馈电臂功率。
3. 邻车再生吸收、可逆变电所反馈和浪费分配。
4. N-1、欠压限牵和调度反馈。
5. 供电专用页面量程、单位和能量闭合展示。

退出条件：`SCN-REGEN-MATCH`和`SCN-PWR-N1`通过，能量平衡残差达标。

### 18.5 Stage 4：时延、对比与研究实验

目标：形成可答辩、可复现实验和后续优化研究能力。

建议完成：

1. 通信时延、抖动和丢包场景。
2. 基线/实验同步对比和回放。
3. 自动生成指标报告和失败归因。
4. ATO参数、发车间隔、供电削峰和再生利用优化。

退出条件：八个必备场景可一键运行并生成一致报告。

## 19. 关键验收场景与通过标准

### 19.1 正常多车运营

1. 至少3列车从抽象车库按计划依次发车。
2. 列车通过CI建立进路后进入正线。
3. 运行中无冲突进路、重叠不相容占用或闯入未授权区段。
4. 至少一列完成终点折返，至少一列完成收车。
5. 所有页面列车数量、位置和状态一致。

### 19.2 多级位驾驶

1. 依次操作P1-P4，牵引力和加速度总体单调增加，受速度特性和限牵约束。
2. 依次操作B1-B7，制动力和减速度总体单调增加。
3. 满制动至速度0后列车不反向。
4. EB能够覆盖牵引命令，ATP能够覆盖人工牵引。

### 19.3 高客流闭环

相对正常客流基线，高客流场景应至少产生以下可解释变化：

1. 部分站台候车和滞留增加。
2. 部分列车载客量和质量增加。
3. 上下客时间或停站时间增加。
4. 实际间隔、延误或调度决策发生变化。
5. 相同驾驶策略下牵引响应或能耗发生变化。

### 19.4 再生匹配

1. 设置一列牵引、一列再生制动且处于可电气耦合区域。
2. 再生能量应优先降低牵引所正向输出。
3. 剩余再生能量应明确进入反馈或浪费。
4. 列车、馈电臂和牵引所的正负功率方向一致。
5. 能量平衡残差满足要求。

### 19.5 供电N-1

1. 注入一座牵引所退出。
2. 该牵引所和相关馈电臂状态真实改变。
3. 网络重新求解，邻近设备负荷、电压或供电路径发生变化。
4. 必要时产生限牵并改变列车运行。
5. 调度读取限牵或延误后产生可追溯决策。
6. 恢复后拓扑、告警和运行状态一致恢复，历史曲线保留事件。

### 19.6 通信时延

1. 对同一ATO场景分别设置0 ms和500 ms状态时延。
2. 系统记录控制器实际使用的状态时间戳。
3. 输出停车、速度跟踪、ATP干预和运行时间差异。
4. 不得以当前真实状态替代延迟状态后仍宣称完成时延实验。
