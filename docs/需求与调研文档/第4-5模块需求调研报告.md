调研对象：

4. 信号与列控仿真模块

- 《前八个业务模块互联网需求调研指导文档》

调研日期：2026-07-06

## 1. 调研结论摘要

本次调研认为，第 4 模块和第 5 模块应作为轨道交通仿真系统中的“运行约束与操控核心”。信号与列控仿真模块负责把信号灯、闭塞分区、移动授权、道岔和限速区段转化为对列车的速度-距离运行授权与触发点序列；司机台/驾驶行为仿真模块负责模拟司机（或 ATO/ATP 辅助下司机）在驾驶台上的手柄、按钮、模式选择、注意力与反应时间等行为，并把这些行为转化为对车辆动力学模块的牵引/制动指令。

在实训项目周期较短、真实 CBTC/ATC 联锁接口不开放的情况下，建议采用“接口仿真 + 行为建模”双轨方案：

1. 最小可行版本：固定闭塞、离散信号灯位、简化移动授权、离散牵引/制动级位、单司机手动驾驶、零反应延迟、ATO 透传。
2. 可演示版本：准移动闭塞、临时限速、模式切换（手动/ATO/ATP 监督下的人工）、司机制动响应延迟、ATO 速度曲线跟踪、紧急制动触发。
3. 增强版本：移动闭塞仿真（虚拟块）、完整 MA（移动授权）计算、司机注意力衰减、个性化驾驶风格、打盹/分心故障注入。

两模块之间的核心关系是：

```text
信号模块输出：信号灯位、限速、MA 终点、临时限速区段、授权变化事件
        ↓
司机台模块使用：决定可用速度、限速值、目标速度曲线、是否需要制动
        ↓
司机台模块输出：牵引指令、制动指令、模式切换、紧急制动、DriverInput
        ↓
信号模块记录：授权使用情况、闯红灯、保护区段占用、模式异常
```

## 4. 第4模块：信号与列控仿真模块

## 4.1 模块定位

信号与列控仿真模块用于模拟铁路信号系统对列车运行的约束与保护，是车辆运行安全、自动驾驶、调度进路和司机决策的基础模块。

本模块不要求在实训阶段复现真实 CBTC/ATC 联锁机或轨旁电子单元（LEU/CTC）的完整协议。建议范围限定为：

```text
固定闭塞（最小版本）
可选准移动闭塞 / 虚拟块（增强版本）
信号灯位变化
闭塞分区占用
移动授权（MA）终点
临时限速区段
道岔与进路
闯红灯触发紧急制动
模式状态：FS / OS / LS / SH / SR
```

## 4.2 业务目标

信号与列控模块应回答以下问题：

1. 当前列车前方的信号灯是什么灯位？
2. 当前可用的移动授权（MA）终点在哪里？
3. 列车当前位置属于哪个闭塞分区？
4. 当前限速是多少？是否还有临时限速区段？
5. 列车是否越过了信号授权终点？
6. 信号是否对列车施加紧急制动？
7. 列车是否进入了不正确的进路？

## 4.3 来源分析

ETCS/ERTMS baseline（ERA）定义了 ATP 必须保证的安全功能，包括模式监督、限速监督、MA 监督、退行监督和冒进防护。本模块必须实现其中“安全相关”的最小集：MA 监督、限速监督、模式状态监督，并在实训版中以简化的“信号-分区-MA”三件套形式落地。

railML interlocking 与 signalling 子模式提供信号元素、联锁逻辑、移动授权、信号、限速区段、保护区段、触发点与信号箱的标准化数据模型。因此本模块必须维护这些对象并发布标准化消息，不能仅以字符串和布尔表示信号。

IEEE 1474（CBTC 体系）给出连续式 ATP/ATO 系统的术语和功能边界，包括位置报告、MA 计算、保护区段、速度-距离曲线监督，可作为本模块在“准移动闭塞 / 增强版本”中的参考。

OpenTrack 仿真工具在闭塞、信号和冲突检测方面提供参考实现，SUMO 在铁路场景下提供信号灯与冲突解决。这两个工具共同支持“信号-列车”联动的最小可演示模式。

## 4.4 需求范围

### 4.4.1 Must 范围

必须实现：

- 信号灯位模型：GREEN / YELLOW / RED / DOUBLE_YELLOW。
- 闭塞分区占用与释放。
- 固定闭塞下的移动授权（MA）终点计算。
- 限速区段：固定限速 + 临时限速。
- 信号事件触发：信号变化、MA 收缩、临时限速生效。
- 闯红灯触发紧急制动。
- 模式状态：FS（完全监督）/ OS（目视行车）/ LS（限制模式）/ SH（调车）/ SR（冒进后救援）。
- SignalState 标准消息发布（按后四模块设计文档）。
- 触发点 / 警告区抽象。

### 4.4.2 Should 范围

建议实现：

- 准移动闭塞 / 虚拟块（MA 终点随前车尾部滚动）。
- 道岔与进路（简化进路表）。
- 保护区段占用检查。
- 退行监督（防止列车越过后退限制点）。
- ATP 紧急制动干预事件。
- 信号日志（每列车通过的信号记录）。

### 4.4.3 Could 范围

时间允许时实现：

- 移动闭塞仿真（位置报告周期 200ms）。
- 完整联锁逻辑（route locking、approach locking、flank protection）。
- 临时限速 TSR 的多段叠加与解除时间。
- 信号灯故障注入（错误显示、灯丝断）。
- 车载 ATP 完整状态机。

## 4.6 信号计算模型建议

### 4.6.1 固定闭塞下的 MA 计算

```text
ma_end_position = next_red_signal_position
ma_permitted_speed = min(
    section_permanent_speed,
    yellow_signal_speed_limit,
    end_of_ma_deceleration_limit
)
```

### 4.6.2 准移动闭塞下的 MA 计算

```text
ma_end_position = preceding_train_rear_position - safe_distance
ma_permitted_speed = min(
    section_permanent_speed,
    end_of_ma_deceleration_limit
)
```

### 4.6.3 限速叠加

```text
effective_speed_limit = min(
    permanent_speed_limit,
    temporary_speed_limit_overlapping,
    ma_permitted_speed
)
```

### 4.6.4 ATP 紧急制动触发

```text
if trainPositionM > ma_end_position:
    emergencyBrakeRequested = true
    signalEvent = SIGNAL_VIOLATION
if speedMps > effective_speed_limit + tolerance:
    emergencyBrakeRequested = true
    signalEvent = OVERSPEED
```

### 4.6.5 模式状态转移

```text
default: FS
if driver_selects_shunting: SH
if driver_selects_on_sight: OS
if driver_selects_limited_supervision: LS
if train_passes_red_signal: SR
```

## 4.9 典型场景

### SIG-SC-001 正常绿灯运行

目标：验证固定闭塞下 MA 与限速输出。

流程：

```text

MA 终点取下下一架信号
允许速度取区段限速

```

- MA 终点为下下架信号里程。

### SIG-SC-002 黄灯减速

流程：

```text

```

验收：

- SignalState.currentAspect=YELLOW。
- permittedSpeedMps 小于区段限速。
- targetDistanceM 与 targetSpeedMps 描述一条可用制动曲线。

### SIG-SC-003 闯红灯紧急制动

目标：验证 ATP 闯红灯防护。

流程：

```text
列车未授权冲过红灯
系统检测列车越过 MA 终点
ATP 触发紧急制动

```

验收：

- emergencyBrakeRequested=true。
- SignalEvent.eventType=SIGNAL_VIOLATION。
- 车辆进入紧急制动阶段。

### SIG-SC-004 临时限速生效

目标：验证 TSR 对允许速度的影响。

流程：

```text
列车接近临时限速区段

允许速度临时降低
TSR 解除后允许速度恢复
```

验收：

- 进入 TSR 区段后 permittedSpeedMps 取 TSR 值。
- 离开后 permittedSpeedMps 恢复区段值。
- 事件包含 TSR_APPLY 与 TSR_RELEASE。

### SIG-SC-005 模式切换

目标：验证 FS/SH/LS 切换与 ATP 行为。

流程：

```text

司机选择目视行车 OS
回到 FS
```

验收：

- SignalState.mode 同步变化。
- 在 SH/OS 模式下，ATP 仅做退行监督。
- 模式切换产生事件日志。

## 5. 第5模块：司机台/驾驶行为仿真模块

## 5.1 模块定位

司机台/驾驶行为仿真模块用于模拟司机在驾驶台上的操作行为，作为车辆动力学、信号列控、调度和自动控制之间的人工操控层。对于本实训项目，它的重点不是高保真的人因模型，而是：

```text
手柄与按钮（牵引级位、制动级位、方向、警惕按钮、模式切换）
驾驶模式（手动 / ATO 辅助 / ATP 监督下的人工）
目标速度曲线生成
反应时间建模
紧急制动与警惕按钮
信号变化下的响应行为
驾驶风格（标准 / 平稳 / 激进）
```

如果后期有足够时间，可以扩展为更接近真实司机行为仿真（注意力衰减、个性化驾驶风格、疲劳/分心故障），但第一版应优先保证能在 ATO 透传 + 手动驾驶两种模式下正确向车辆模块发出指令。

## 5.2 业务目标

司机台模块应回答以下问题：

1. 当前司机处在哪种驾驶模式？
2. 司机当前发出的牵引/制动指令是什么？
3. 司机是否按信号变化及时响应？
4. ATO 是否在跟踪目标速度曲线？
5. 在 ATP 紧急制动被请求时，司机的操作如何被覆盖？
6. 司机的驾驶风格对运行时间和能耗有多大影响？

## 5.3 来源分析

railML rollingstock 包含司机参数、列车操作手柄与执行器、ATO/手动驾驶相关字段。这支持本模块必须维护“司机手柄状态”和“操作执行器”作为核心对象。

OpenTrack 提供司机对时刻表的跟踪、加速/制动调整与速度曲线逼近模型。这支持本模块采用“基于速度误差调整牵引/制动级位”的简化控制策略。

ATO 速度曲线跟踪研究表明，ATO 跟踪目标速度曲线存在稳态误差、滞后与平稳性指标。这支持本模块在 ATO 模式下产生速度误差并记录平稳性指标。

人因研究表明，司机对信号变化和警告的响应时间呈对数正态分布，可作为司机反应时间建模的统计依据。实训项目可使用“配置化固定延迟”代替分布抽样，但保留扩展空间。

IEC 62278 / EN 50126（铁路 RAMS）为司机台 HMI 在可靠性/可用性/可维护性/安全性方面提供框架性要求。ETCS 司机台 DMI（Driver Machine Interface）参考提供速度表盘、模式显示、目标距离/目标速度、警告区等元素，可作为本模块界面抽象依据。

## 5.4 需求范围

### 5.4.1 Must 范围

必须实现：

- 司机台手柄：牵引级位、制动级位、方向（前进/后退）。
- 司机台按钮：警惕按钮、模式切换、ATO 启动/退出、紧急制动。
- 驾驶模式：MANUAL（手动）/ ATO（自动驾驶）/ ATP_SUPERVISED（ATP 监督下人工）。
- 牵引/制动指令从司机手柄到 ControlCommand 的映射。
- 目标速度曲线生成（ATO 模式下）。
- 信号变化下的司机反应时间（可配置固定延迟）。
- 司机台对 ATP 紧急制动请求的覆盖。
- DriverInput / ControlCommand 标准消息发布（按后四模块设计文档）。
- HMI 抽象（速度表盘、模式显示、目标距离/目标速度、警告区）。

### 5.4.2 Should 范围

建议实现：

- 司机反应时间分布（对数正态）。
- 驾驶风格（标准/平稳/激进）。
- 警惕按钮未按时按下的紧急制动。
- 司机误操作（按下紧急制动但未松开警惕按钮的告警）。
- ATO 速度曲线跟踪误差统计。
- 司机疲劳/分心故障注入。

### 5.4.3 Could 范围

时间允许时实现：

- 司机个体差异（不同司机不同反应参数）。
- 注意力衰减模型。
- 多次 ATP 干预后的司机行为修正。
- 与调度中心语音通信抽象。
- 司机台日志回放与教学。

## 5.6 司机操控计算模型建议

### 5.6.1 最小指令映射

```text
if mode == MANUAL:
    throttle = driver_throttle_level
    brake = driver_brake_level
if mode == ATO:
    throttle = ato_throttle_level
    brake = ato_brake_level
if mode == ATP_SUPERVISED:
    throttle = min(driver_throttle_level, ato_throttle_level)
    brake = max(driver_brake_level, ato_brake_level)
```

### 5.6.2 ATO 速度曲线生成

```text
target_speed = min(
    permitted_speed_from_signalling,
    permanent_speed_limit,
    next_speed_restriction
)
target_distance = ma_end_position - train_position
```

### 5.6.3 ATO 牵引/制动决策

```text
speed_error = target_speed - current_speed
if speed_error > 0.5:
    throttle_step_up
elif speed_error < -0.2:
    brake_step_up
else:
    coast
```

### 5.6.4 司机反应延迟

```text
effective_command = apply(driver_command, delay=driver_reaction_time_ms)
```

### 5.6.5 ATP 紧急制动覆盖

```text
if atp_emergency_brake_requested == true:
    final_command = EMERGENCY_BRAKE
    source = ATP_OVERRIDE
```

### 5.6.6 警惕按钮检查

```text
if simTimeMs - lastVigilanceTime > vigilanceThreshold:
    emergencyBrake = true
    reason = VIGILANCE_LOST
```

## 5.9 典型场景

### DRV-SC-001 手动驾驶基本流程

目标：验证司机手柄与 ControlCommand 的映射。

流程：

```text
司机拉牵引级位 2
司机按警惕按钮
速度上升
司机推制动级位 1
速度下降
```

验收：

- DriverInput.throttleLevel 与 ControlCommand.throttleLevel 一致。
- 警惕按钮周期内被按下。
- 速度变化方向与指令方向一致。

### DRV-SC-002 ATO 自动驾驶

目标：验证 ATO 目标速度曲线与跟踪误差。

流程：

```text
司机按下 ATO 启动
ATO 生成目标速度曲线
ATO 控制车辆跟踪目标速度
```

验收：

- ATOState.active=true。
- 速度曲线接近目标速度。
- speedErrorMps 在 ±0.5 m/s 内。

### DRV-SC-003 ATP 监督下人工驾驶

目标：验证 ATP_SUPERVISED 模式下 ATP 限制。

流程：

```text
司机手柄牵引级位 4
ATP 限制 permittedSpeedMps
车辆被 ATP 限制在更低速度
```

验收：

- ControlCommand.throttleLevel 被 ATP 钳制。
- speedErrorMps 反映 ATP 干预。
- 事件日志记录 ATP 覆盖。

### DRV-SC-004 闯红灯紧急制动覆盖

目标：验证 ATP 紧急制动对司机指令的覆盖。

流程：

```text
司机保持牵引级位 3
信号模块检测到闯红灯
ATP 触发紧急制动
ControlCommand.emergencyBrake=true
```

验收：

- ControlCommand.source=ATP_OVERRIDE。
- 紧急制动生效。
- 事件日志记录覆盖原因。

### DRV-SC-005 警惕按钮超时

目标：验证警惕按钮超时触发的紧急制动。

流程：

```text
司机长时间未按警惕按钮
超过 vigilanceThreshold
系统触发紧急制动
```

验收：

- 紧急制动生效。
- 事件 reason=VIGILANCE_LOST。
- 司机按警惕按钮后系统恢复。

## 5.10 评估指标

| 指标 | 公式/判定 | 用途 |
|---|---|---|
| ATO 跟踪误差均值 | mean(abs(speedErrorMps)) | ATO 控制质量 |
| ATO 跟踪误差最大值 | max(abs(speedErrorMps)) | 平稳性 |
| ATO 牵引/制动切换次数 | Σ lastAction 变化 | 平稳性 |
| 反应延迟 | driver_command_emit_time - signal_change_time | 司机行为评价 |
| ATP 覆盖次数 | Σ ATP_OVERRIDE 事件 | 安全监督评价 |
| 警惕超时次数 | Σ VIGILANCE_LOST 事件 | 安全性 |
| 模式切换次数 | Σ mode 变化 | 场景覆盖 |
| 驾驶风格偏差 | mean(|throttle|) / profile.throttleStep | 风格评价 |
| DriverInput 更新频率 | 1 / 平均更新间隔 | 实时性 |

## 6. 两模块联动需求

## 6.1 联动数据流

```text
仿真时钟 Tick
  ↓
信号模块计算 SignalState（MA 终点、允许速度、模式、紧急制动）
  ↓
司机台模块读取 SignalState 和 TrainState
  ↓
司机台模块计算 ControlCommand / DriverInput
  ↓
车辆模块接收 ControlCommand
  ↓
下一周期 SignalState 和 TrainState 更新
```

## 6.2 联动需求表

| Req ID | 类型 | 需求描述 | 优先级 | 验收方式 |
|---|---|---|---|---|
| SD-IF-001 | 接口 | 信号模块应向司机台模块提供 MA 终点、允许速度、模式和紧急制动请求。 | Must | 联调时司机台能识别限制条件。 |
| SD-IF-002 | 接口 | 司机台模块应向信号模块反馈 ATP 紧急制动覆盖原因。 | Should | 事件日志包含覆盖原因。 |
| SD-FR-001 | 功能 | 当信号模块输出 permittedSpeedMps 降低时，司机台 ATO 应相应调整目标速度。 | Must | TSR 生效后 ATO 跟踪到新目标速度。 |

## 9. 对本项目的落地建议

## 9.1 第一版实现建议

第一版目标是“能跑、能控、能保、能展示”。

信号模块：

```text
固定闭塞
信号灯位 GREEN / YELLOW / RED
限速段（固定限速）
移动授权 MA
闯红灯触发紧急制动
模式状态机 FS / OS / SH
SignalState 标准化消息
```

司机台模块：

```text
司机台手柄与按钮
驾驶模式 MANUAL / ATO / SH
ATO 速度曲线跟踪
司机反应延迟（可配置固定值）
ATP 紧急制动覆盖
DriverInput / ControlCommand 标准化消息
HMI 抽象（速度表盘、模式显示、目标距离/目标速度、警告区）
```

第一版验收标准：

- 信号变化时 SignalState 正确发布。
- 闯红灯时 ATP 紧急制动被司机台覆盖并触发。
- ATO 模式下速度跟踪误差在合理范围内。
- 司机手柄与 ControlCommand 映射正确。
- HMI 字段与实际状态一致。

## 9.2 第二版实现建议

第二版目标是“更像真实系统”。

信号模块：

```text
准移动闭塞 / 虚拟块
道岔与简化进路
保护区段占用检查
退行监督
信号日志
```

司机台模块：

```text
司机反应时间分布
驾驶风格 STANDARD / COMFORT / AGGRESSIVE
警惕按钮超时紧急制动
司机疲劳/分心故障注入
ATO 跟踪误差统计
```

第二版验收标准：

- 多车运行时 MA 随前车尾部滚动。
- 不同驾驶风格对运行时间和能耗有可量化差异。
- 警惕超时被系统记录并触发紧急制动。
- 能展示 ATP 干预对比与 ATO 跟踪误差曲线。

## 9.3 不建议第一阶段做的内容

以下内容技术价值高，但不适合第一阶段投入太多时间：

- 真实 CBTC 通信协议（IEEE 1474 完整实现）。
- 真实联锁机 IXL 逻辑（route locking、approach locking、flank protection 完整复现）。
- 完整 ETCS 模式机（FS/OS/LS/SH/SR/P/T 等全部模式）。
- 车载 ATP 完整状态机。
- 司机个体差异与人因全要素模型。
- DMI 高保真视觉与语音告警。
- 真实调度中心通信协议。

这些内容可作为答辩中的“后续扩展方向”。
