# 轨道交通仿真系统需求收集指导文档

## 1. 文档目的

本文档用于指导小组在“轨道交通仿真系统”实训项目中完成需求收集、需求分析、需求确认和需求管理。目标不是一次性写出完美需求，而是在 7 月 6 日至 7 月 15 日的短周期内，快速形成可实现、可验证、可答辩的需求基线。

本项目建议聚焦为：

> 开发一个连接轨道交通仿真设备/实体司机台的软件系统，完成列车状态采集、控制指令下发、手动驾驶、自动停站、运行数据记录和效果评估。

## 2. 初步网上调研结论

### 2.1 需求工程方法

ISO/IEC/IEEE 29148 强调需求工程覆盖系统和软件全生命周期，要求定义“好需求”的结构、属性和特征，并支持需求过程的迭代应用。因此本项目的需求不应只写“实现自动停车”，而应写清楚触发条件、输入、输出、约束、验收方法和优先级。

来源：

- IEEE/ISO/IEC 29148-2018: https://standards.ieee.org/ieee/29148/6937/

### 2.2 轨道交通控制系统的关键关注点

IEEE 1474.1-2025 对 CBTC 的描述显示，通信列控系统关注高精度列车定位、连续双向车地通信、车载和轨旁处理单元、ATP 自动防护、可选 ATO 自动运行和 ATS 自动监控。因此本项目即使不实现完整 CBTC，也应把“定位/速度、通信、保护约束、自动运行、监控展示”作为核心需求维度。

来源：

- IEEE 1474.1-2025: https://standards.ieee.org/ieee/1474.1/6959/

### 2.3 信号系统接口数据分类

《城市轨道交通信号系统运营技术规范（试行）》提出，信号系统应向智能管理系统提供线路场站基础信息、列车运行信息、设备状态信息、故障及报警信息。对本项目而言，这可以直接转化为数据需求分类。

来源：

- 交通运输部《城市轨道交通信号系统运营技术规范（试行）》: https://xxgk.mot.gov.cn/jigou/ysfws/202201/P020220121613617393399.pdf

### 2.4 仿真平台常见模块

列车群运行仿真平台研究中，系统可划分为数据结构化存储与维护、仿真基础建模、列车运行仿真等模块；OpenTrack/railML 等资料也显示，轨道仿真通常围绕基础设施、车辆、运行图、信号/联锁、运行结果评估组织数据。

来源：

- 面向列车群的运行仿真平台关键技术研究: https://pdf.hanspub.org/ojtt20210400000_52911852.pdf
- railML 3.3 概览: https://www.railml.org/en/news/railml-3-3-overview-of-key-developments

### 2.5 自动停站验收思路

全自动运行系统验收资料中包含“列车站台自动对位功能检测”，其思路是设置欠停/过停距离，观察并记录列车到站后的状态。我们可借鉴为自动停站需求的验收方式：多次运行、记录停车误差、是否满足开门条件、是否触发异常。

来源：

- 城市轨道交通全自动运行系统验收标准: https://zjw.sh.gov.cn/cmsres/2c/2c6800adc8f3498ab389d9b224f62bec/6f53f566cfe3953c7cbc06ffd2330e2e.pdf

## 4. 需求收集方法

### 4.1 PPT 和课程材料分析

从 PPT 中已确认的硬性背景：

- 项目主题是轨道交通仿真系统。
- 系统包含供电仿真、车辆仿真、轨道仿真、信号系统等。
- 需要根据接口协议设计接口通信程序。
- 需要驱动实体司机台。
- 需要完成交通场景仿真，例如手动停站、自动停站、智能运行。
- 评分关注立项、中期、最终答辩、代码、文档、周报、项目管理和停车杯表现。

### 4.2 访谈

建议在立项前完成 20 至 30 分钟访谈，访谈对象优先级：

1. 指导老师。
2. 熟悉 704 仿真台的同学或助教。
3. 小组内负责通信、算法、界面、测试的成员。

访谈问题清单：

| 类别 | 问题 |
|---|---|
| 项目范围 | 本次实训最低必须实现哪些功能？哪些功能属于加分项？ |
| 设备接口 | 仿真台通过 TCP、UDP、串口还是其他方式通信？是否有接口文档和示例代码？ |
| 数据输入 | 能读取哪些状态量：速度、位置、目标站、信号、牵引级位、制动级位、车门状态？ |
| 控制输出 | 能下发哪些指令：牵引、制动、惰行、紧急制动、开关门、模式切换？ |
| 运行场景 | 必须演示手动停站、自动停站还是智能运行？是否需要参与停车杯？ |
| 评价指标 | 停车误差多少算成功？是否关注运行时间、舒适度、能耗、超速、越站？ |
| 设备限制 | 每组可使用仿真台多久？是否允许高频发送指令？是否有安全边界？ |
| 答辩要求 | 最终演示必须现场连接设备，还是可以用录屏/离线模拟补充？ |

### 4.3 现场观察

使用仿真台前，安排 1 名同学记录现场观察结果：

- 设备启动流程。
- 司机台有哪些手柄、按钮、仪表。
- 仿真软件界面显示哪些状态。
- 操作一次手动停车需要哪些步骤。
- 哪些状态变化对自动控制最关键。
- 设备异常或通信中断时的表现。

现场观察产物：

| 产物 | 内容 |
|---|---|
| 操作流程图 | 从启动仿真到完成停车的步骤 |
| 状态字段清单 | 能观察到或能读取的数据 |
| 控制指令清单 | 能操作或能发送的控制量 |
| 风险清单 | 可能导致设备异常或演示失败的问题 |

## 5. 需求分类

### 5.1 功能需求

建议功能需求分为 8 类：

| 类别 | 示例 |
|---|---|
| 连接管理 | 连接、断开、重连、心跳检测 |
| 状态采集 | 读取速度、位置、信号、目标站、模式、车门 |
| 指令控制 | 下发牵引、制动、惰行、紧急制动、开关门 |
| 手动模式 | 支持人工/司机台输入，显示操作反馈 |
| 自动停站 | 根据距离和速度计算控制指令，完成停车 |
| 安全约束 | 超速保护、越站检测、通信异常保护 |
| 数据记录 | 记录运行曲线、指令、事件、停车结果 |
| 可视化 | 显示线路、列车状态、控制模式、曲线、日志 |

### 5.2 非功能需求

| 类别 | 示例需求 |
|---|---|
| 实时性 | 状态刷新周期不高于 100 ms 或按设备协议确定 |
| 可靠性 | 通信断开后能提示并尝试重连 |
| 可用性 | 主界面能一眼看到速度、位置、目标距离和当前指令 |
| 可维护性 | 通信协议字段、控制参数可配置 |
| 可测试性 | 支持离线模拟模式，不依赖仿真台也能测试算法 |
| 安全性 | 不发送超出协议范围的控制指令，异常时进入惰行或制动策略 |
| 可追溯性 | 每条需求能关联设计、代码、测试和演示证据 |

### 5.3 数据需求

按轨交信号系统接口数据分类，可整理为：

| 数据类 | 字段示例 |
|---|---|
| 线路场站基础信息 | 线路编号、车站、停车点、区间长度、坡度、限速 |
| 列车运行信息 | 车次号、方向、位置、速度、驾驶模式、到发点、计划运行图 |
| 设备状态信息 | 司机台、信号机、道岔、站台门、轨道区段、车门状态 |
| 故障及报警信息 | 通信中断、超速、越过停车点、设备故障、紧急制动 |
| 控制指令信息 | 牵引级位、制动级位、惰行、紧急制动、开关门、模式切换 |
| 评估结果信息 | 停车误差、运行时间、最大速度、制动次数、成功/失败 |

示例：

```text
需求编号：FR-ATO-001
需求名称：自动进站停车
需求类型：功能
优先级：Must
需求描述：系统应根据列车当前位置、速度、目标停车点和限速信息，自动生成牵引、惰行或制动指令，使列车在目标停车点附近停车。

输入：当前位置、当前速度、目标停车点、线路限速、控制周期。
处理：计算目标距离和目标速度，决定牵引/制动级位。
输出：牵引/制动/惰行控制指令，当前控制阶段。
约束：不得发送超过协议允许范围的指令；通信异常时停止自动控制。
验收标准：连续 3 次测试中至少 2 次停车误差小于小组设定阈值，例如 1 m；全过程无通信崩溃。
来源：PPT 场景要求、停车杯任务、指导老师访谈。
负责人：算法负责人。
状态：草稿。
```

## 8. 优先级建议

### 8.1 Must：必须完成

- 能连接仿真系统或离线模拟器。
- 能读取并显示列车速度、位置、目标距离。
- 能下发基本牵引/制动指令。
- 能完成一次手动或自动停站演示。
- 能记录运行数据并计算停车误差。
- 有需求文档、设计文档、测试记录和项目管理材料。

### 8.2 Should：应该完成

- 自动停站算法参数可配置。
- 支持通信断开重连。
- 支持速度/位置曲线可视化。
- 支持多次运行结果统计。
- 支持离线仿真模式，减少对 704 仿真台的依赖。

### 8.3 Could：时间允许再做

- 能耗估计。
- 舒适度指标。
- 简单 AI/机器学习参数调优。
- 多车运行或调度场景。
- 故障注入与回放。

会议输出：

| 输出物 | 说明 |
|---|---|
| 需求范围表 | 明确本次做什么、不做什么 |
| 场景清单 | 至少 3 个演示场景 |
| 接口风险表 | 记录未知接口、设备限制、联调风险 |
| MVP 功能表 | 明确最小可演示版本 |
| 分工表 | 每个需求有负责人 |

会议结论模板：

```text
本组项目聚焦于：单列车自动停站与司机台控制。
本组必须完成：通信、状态显示、控制指令、自动停站、数据记录、结果评估。
本组暂不完成：完整供电潮流计算、多车调度、真实 CBTC 安全认证。
主要风险：设备使用时间有限、接口协议理解不足、自动控制参数需要现场调试。
缓解措施：先做离线模拟器；接口层和算法层解耦；每次上台测试前准备测试脚本。
```
